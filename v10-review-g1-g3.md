## 1. TOGA四环模型架构评估

### 1.1 逻辑完整性（需要肯定）

TOGA四环(T→O→G→A)从运营流程角度看是自洽的，八步闭环覆盖了"知识生产→专家决策→AI执行→自进化反馈"全链路。20遍审计点每一条都有明确的检查项。

### 1.2 但是——这是一个"二层架构"方案

**核心缺陷**: TOGA模型完全运行在「AI对话层+专家人工层」之上，完全没有触及「系统架构层」。

- T(知识库): 只有三层知识库的「读取和摘要」，没有知识库本身的高可用、版本一致性、冷热分离
- O(专家): 只定义了专家的「会议和签发」，没有专家在技术架构决策中的参与机制（如ADR评审）
- G(AI开发): 忽略了G的运行时底座——边缘服务器/离线/安全/双通道这些基础设施
- A(自进化): 进化局限于"prompt模板更新"，不涉及架构演进、组件替换、数据库迁移

**陈架构(E1)的视角**: TOGA需要补充第五环——**S(系统/System)**，将架构层面纳入闭环。

**周CTO(E44)的视角**: 团队<20人不拆微服务是正确决策，但V10没有给出模块化单体的内部治理规则——没有`ILYTAdapter`接口契约、没有反模式检测与代码架构偏离的关联。

### 1.3 20遍审计的盲区

| 审计编号 | 审计内容 | 架构盲区 |
|:--------:|----------|----------|
| #1-#4 T→O | 知识简报读与推 | 未审计知识库数据一致性、版本冲突 |
| #5-#8 O→G | 需求卡ID强制引用 | 未审计需求卡是否包含安全/合规/非功能性约束 |

## 3. 时间线预算法则 —— maxConcurrentRuns=3是重大架构风险

### 3.1 量化压力分析

| 时段 | 并发事件 | 说明 |
|:----:|:--------:|------|
| 07:50-08:00 | AI简报(1) | 单任务，无冲突 |
| 08:00-08:30 | 晨学(1) → 派单(1) → 晨会(1) | 3个systemEvent连续，但非同时 |
| **08:50** | **晨会20min中+推进检查** | **最高危时段** |
| 09:30-09:45 | 开发对齐(1) | 1个 |
| **09:45-10:45** | **开发对齐结束→AI自进化检查** | **对齐cron如果延迟15min，将和自进化检查重叠** |
| 11:00 | 知识库日采(1, isolated) | 不阻塞 |
| **11:00-12:00** | **日采60min(1)+推进检查触发** | **推进入口可能和日采同时跑** |
| 23:00 | 日终汇总(1, isolated)+偏差分析(1)+prompt更新(1)+明日建议(1) | **23:00~23:10之间可能有4个任务需要并发** |
| 全天 | Every组: 10min开发+验收脉冲+监控+keepalive | **持续占用并发槽位** |

### 3.2 关键结论

**周CTO(E44)判定**: `maxConcurrentRuns=3`是**不合理的硬编码**，原因:

1. **23:00高峰**: 日终汇总(isolated)+偏差分析+prompt更新+明日建议+验收脉冲→至少4个任务需要3以上并发
2. **Every组持续占用**: 10min自动开发、30min验收脉冲、每小时监控+keepalive这4个任务是**永久驻留**的，一旦占用3槽，所有systemEvent排队卡住
3. **非均匀分布**: 白天systemEvent密集(13个在12h内)，实际上每个事件都有cron窗口边界→超时的连锁反应
4. **没有优先级队列**: 当前只有"谁先到谁占"，验收脉冲(P0)和监控(P1)如果被低优先级任务卡住→风险

**建议方案**:
- `maxConcurrentRuns=5`是底线，建议`=8`以应对高峰
- 增加任务优先级: P0(验收/日终/监控)排队插队，P3(审计/日志)等资源
- Isolated任务不占用并发槽位
- 每15min检测队列深度，深度>3触发压缩策略

### 3.3 预算法则硬编码问题

每个锚点之间的"预算→剩余"计算是**确定性调度**，但AI开发任务耗时是**非确定性**的:
- 树哥(XiaoGe)一个任务可能5分钟也可能2小时
- 验收脉冲如果触发大量回写，可能远超30min预算
- 监控每小时一次，但如果被排队阻塞监控饥饿→系统无监控状态

**结论**: 预算法则的理论基础是「所有任务时间已知且可控」，这在AI开发场景不成立。需要引入:
1. 动态时间窗口(不是硬编码的15/30/60min)
2. 任务超时熔断(>预算的1.5倍→kill后重新排队)
3. 手动干预通道(店长/CTO可手动加急或取消任务)

## 5. V10架构层面的大漏洞

### 漏洞1: 安全基线完全缺失 🔴 P0

**V10全篇没有一句关于安全的描述**。从陈架构(E1)、李安全(E2)、周CTO(E44)的角度看，这是不可接受的。

现存安全P0问题(全部未在V10中提及):
- 盲盒API端点无AuthGuard → 任何人可调用
- 盲盒服务无速率限制 → 可被批量刷API(cost++攻击)
- RLS行级安全未实现 → SQL注入可跨租户越权
- Gateway层tenant_id校验缺失 → 恶意请求可传入任意tenant_id
- Push模块无deviceToken绑定校验 → 可通过任意deviceToken推送
- 盲盒概率引擎用内存Math.random() → 可被调试修改

**V10需要**: 每个cron任务附加安全基线检查清单，每个验收脉冲强制过安全门。

### 漏洞2: 离线/边缘部署方案为零 🔴 P0

**王硬件(E3)+钱店长(E11)+李收银(E13)一致判定**:

V10的整个TOGA模型假设**始终在线**，但现实是:
- 游乐场网络不可靠(人多WiFi拥塞、4G信号弱)
- 门店断网时收银台必须能工作
- 边缘服务器(NUC/Mac Mini)本地存储才能支撑离线

**当前状态**:
- ESP32离线排队→未实现
- 盲盒/通知模块全内存Map→边缘部署需SQLite
- CRDT冲突合并→未实现
- 双机冷备→无方案

**V10需要**: 每个验收脉冲增加"边缘兼容性"门，至少标记哪些功能离线可用/不可用。

### 漏洞3: WebSocket+HTTP双通道未落地 🔴 P0

从V5.1就提出的AD-0005决议，至今仅实现了WebSocket reconnect，**HTTP轮询降级缺失**:
- 对G3(收银/导购): 收银台/导玩员终端在WS断开超过10s后无HTTP降级→用户感知断连=系统崩溃
- 对C端: 排队通知、到号通知在WS断开时丢失
- 与`maxConcurrentRuns=3`联动: 如果验收脉冲占用3槽，双通道回写无法运行

### 漏洞4: 盲盒合规尚未达到监管红线 🟡 P1

**刘合规(E6)+沈监管(E38)**:
V10完全没有涉及盲盒合规的三档监管实现:
- 普通盲盒概率展示——收银端强制展示
- 中级盲盒限购(单日≤20)——会员系统限购逻辑

- 未成年保护——年龄判断+家长监护
- 7天冷静期退款——退款引擎

**店A倒计时22天，盲盒合规是P0红线**，V10计划中完全空白。

### 漏洞5: 知识库/配置版本管理未定义 🟡 P1

## 6. 对V11的3条核心建议

### 建议1: 将TOGA四环升级为TOGAS五环 — 增加S(系统/安全)环

**问题**: 当前的TOGA模型是纯流程方案，不触及系统架构。

**方案**:
```
T(知识库) → O(专家) → G(AI开发) → A(自进化)
                                         ↓
                              S(系统/安全架构)
```

S环职责:
1. 架构偏离检测: AD-DEV-02/06/08的修复进度作为cron健康指标
2. 安全基线每年: 每次验收强制过AuthGuard/RateLimit/RLS/Encryption四道门
3. 架构决策自动记录: 每个ADR(架构决策记录)自动写入知识库
4. 边缘兼容性标记: 新增功能标注"offline OK/online only"
5. 合规审计自动触发: 每次数据生命周期变更自动检查个保法合规

**责任人**: E1陈架构(主)+E2李安全(副)

### 建议2: V11前必须建立安全基线+合规基线，作为所有cron/验收的前置条件

**问题**: V10完全没有安全基线，店A倒计时22天的安全底线不明确。

**方案**:

| 基线 | 检查项 | 责任人 | 期限 |
|------|--------|:------:|:----:|
| 盲盒API AuthGuard | 所有端点添加@AuthGuard+@RateLimit | E2李安全 | 7/13 |
| RLS行级安全 | Prisma启用RLS策略，tenant_id全覆盖索引 | E1陈架构 | 7/15 |
| Gateway tenant_id校验 | traffic-governance.guard强制校验 | E1陈架构 | 7/15 |
| 推送deviceToken绑定 | pushToiOS前验证持有者 | E2李安全 | 7/17 |
| 盲盒概率Lua原子操作 | Redis Lua替代Math.random() | E1陈架构 | 7/17 |
| 盲盒三档合规 | 概率展示+限购+实名+退款 | E38沈监管 | 7/20 |
| 未成年保护 | 年龄判断+消费限额+监护平台 | E38沈监管 | 7/20 |
| 离线运营分类清单 | 离线可用/不可用功能清单 | E11钱店长 | 7/18 |

每条安全基线通过后，在知识库中产生一个安全检查点，作为后续验收的前置必要条件。

### 建议3: 解决maxConcurrentRuns瓶颈 + Every任务抢占问题

**问题**: 当前`maxConcurrentRuns=3`在V10的schedule下一定会触发队列死锁。

**方案**:

1. **提高并发上限**: 将`maxConcurrentRuns`从3提升到**8**（底线5）
2. **引入优先级调度**:

| 优先级 | 任务类型 | 说明 |
|:------:|----------|------|
| P0 | 验收脉冲·日终汇总·监控 | always run，永不排队 |
| P1 | systemEvent(会议·对齐·检查) | queue最多15min |
| P2 | Every(自动开发·keepalive) | 可被P0/P1抢占 |
| P3 | 审计·日志·报告 | 仅空闲时运行 |

3. **Isolated任务不占用并发槽位**: `agentTurn` + `sessionTarget=isolated` 的任务(日采/日终)不计入并发计数
4. **超时熔断**: 任何任务超过预算的1.5倍→自动熔断(kill)→重新排队
5. **队列监测**: cron监测每15min检查队列深度，深度>5触发告警→推送周CTO(E44)
6. **手动干预**: 店长/CTO可通过API手动提升或取消队列中的任务

**操作建议**: 在7/13(V11发布)之前完成上述调整，否则按V10的cron配置，**店A上线第一天就会发生并发死锁**。

## 总结

| 维度 | 结论 |
|:----|------|
| TOGA模型 | 运营流程逻辑完整，但缺少架构/安全/边缘基座，需扩展为TOGAS五环 |
