# Production DB Module Sequencing

## Purpose

This document extends the foundation initialization plan and proposes the next execution order for the remaining `22` non-foundation tables in `apps/api/prisma/schema.prisma`.

## Remaining Tables

- `Tenant`
- `Brand`
- `Store`
- `User`
- `MemberProfile`
- `MemberProfileExtension`
- `LytMemberSnapshot`
- `LytOrderSnapshot`

- `MemberOperationsTask`

- `AuditLog`

## Recommended Sequence

### Phase A

Initialize market and tenant master data first:

- `MarketProfile`
- `Tenant`
- `Brand`
- `Store`
- `User`

Why first:

- These are the primary ownership and scope anchors
- Several later modules imply tenant, brand, store, or market dimensions

### Phase B

Initialize regional and portal configuration next:

- `RegionalConfig`
- `RegionalConfigOverride`
- `PortalSite`
- `EmailChannelConfig`
- `SocialChannelConfig`
- `TaxPolicyConfig`

Why second:

- These are configuration-heavy tables tied to market and portal behavior
- They are lower risk than member or operations data ingestion

### Phase C

Initialize member domain tables:

- `MemberProfile`
- `MemberProfileExtension`
- `LytMemberSnapshot`
- `LytOrderSnapshot`
- `LytPaymentSnapshot`

Why third:

- This phase starts bringing in business state and external snapshot payloads
- It should wait until foundation identity, config, and audit tables are already available

### Phase D

Initialize operations, audit, and integration tables:

- `MemberOperationsTask`
- `MemberOperationsExecutionReceipt`
- `AuditLog`
- `LytConnection`
- `marketing_push_decision_log`
- `inspection_task`

Why fourth:

- These are write-heavy or event-like tables
- They are best introduced after tenant, market, config, and member scaffolding is stable

## Risk Notes

- `marketing_push_decision_log` and `inspection_task` use explicit `@@map` names and should be reviewed by physical table name, not only model name
- Member snapshot tables may become large quickly and should be introduced with retention and index review in the same window
- `AuditLog` can become hot once APIs begin writing, so its index plan should be verified before production traffic fully depends on it
- `LytConnection` and snapshot tables may imply external system onboarding steps outside pure schema creation

## Practical Order

If the goal is minimum-risk production bootstrap, use this order:

1. `foundation-wave0.sql`
2. `foundation-wave1.sql`
3. `foundation-wave2-wave3.sql`
4. `remaining-wave0.sql`
5. `phase-a-master-data.sql`
6. `phase-b-regional-portal.sql`
7. `phase-c-member-domain.sql`
8. `phase-d-ops-audit.sql`

## Suggested Pause Points

- Pause after `remaining-wave0.sql` to ensure enum creation did not conflict with existing objects
- Pause after `phase-a-master-data.sql` and verify `MarketProfile`, `Tenant`, `Brand`, `Store`, and `User`
- Pause after `phase-b-regional-portal.sql` and verify `PortalSite` plus market-scoped config tables
- Pause after `phase-c-member-domain.sql` before any external member or order ingestion is enabled
- Pause after `phase-d-ops-audit.sql` before allowing audit-heavy or operations-heavy workloads to write

## Stop Conditions

Pause the rollout immediately if any of the following occurs:

- `foundation-verify.sql` no longer returns the expected counts after a wave
- any `CREATE TYPE` or `CREATE TABLE` conflicts with unexpected pre-existing objects
- application startup begins failing with relation mismatch errors after a supposedly successful wave
- external ingestion jobs attempt to write business data before the corresponding phase is fully initialized
