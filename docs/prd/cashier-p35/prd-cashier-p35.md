## 1. 目标

### 1.1 业务目标
构建店A前台完整收银闭环系统，实现客户从到台至支付完成的标准化体验(≤30秒/单)，支撑至少3种支付方式，错误率<1%。

### 1.2 产品目标
- 收银员可在收银台上完成商品/套餐选择、会员识别、折扣应用、支付收款全流程
- 支付成功后自动触发小票打印、积分累计、库存扣减
- 异常场景(超时、余额不足、库存不足)有明确拦截提示

### 1.3 成功指标
| 指标 | 目标值 | 测量方式 |
|:----|:------|:---------|
| 单笔收银时长 | ≤30秒 | 从客户到台→支付完成 |
| 支付方式覆盖 | 3种(微信/余额/现金) | 功能检查 |
| 支付错误率 | <1% | 退款/异常订单 ÷ 总订单 |
| 商品搜索响应 | <300ms | 压力测试 |

## 5. 数据约束

### 5.1 数据库约束

| 约束 | 说明 | 理由 |
|:-----|:-----|:-----|
| 订单号唯一 | `ORD-{yyyyMMdd}-{4位流水}` 格式，每日重置 | 便于人工识别与检索 |
| 手机号唯一 | `member.phone` 字段 UNIQUE 索引 | 会员识别核心字段 |
| 金额精度 | 所有金额以`分`为单位存储(整数) | 避免浮点精度问题 |
| 状态机约束 | 订单状态流转: `pending → paid / cancelled`，不允许逆向 | 事务完整性 |
| 事务一致性 | 支付回调处理: 订单写入 + 库存扣减 + 积分增加 必须在同一DB事务内 | 防资损 |
| 无级联删除 | 不做物理删除，订单标记 cancelled 保留记录 | 审计需求 |
| 操作审计 | 每一步关键操作记录 audit_log 表 | 追溯与对账 |

### 5.2 关键数据实体

```typescript
interface CashierOrder {
  id: string;              // ORD-yyyyMMdd-xxxx
  storeId: string;
  items: OrderItem[];
  memberId?: string;
  subtotal: number;        // 分
  discount: number;        // 分
  couponId?: string;
  total: number;           // 分
  payment: 'wechat' | 'balance' | 'cash';
  status: 'pending' | 'paid' | 'cancelled';
  cashReceived?: number;   // 现金实收(分)
  cashChange?: number;     // 找零(分)
  createdAt: Date;
  paidAt?: Date;
  operatorId: string;
}

interface OrderItem {
  productId: string;
  name: string;
  unitPrice: number;       // 分
  quantity: number;
  subtotal: number;        // 分
}
```
