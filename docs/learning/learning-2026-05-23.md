## 1. NestJS 模块组织 — 根治 Orphan Controller

### 问题根因
当前 SPORTS ANT 项目有 70+ closure controller，其中 35 个是 orphan（未在任何 module 注册）。之前批量注册导致 DI 级联崩溃。

### 业界方案

**方案 A: Dynamic Module 模式** (NestJS 官方文档)
```
register() 工厂函数 → 返回 DynamicModule → controllers/providers 动态注入
优势: 消费者模块可通过配置控制注册
风险: 复杂依赖链仍可能破裂
```

**方案 B: Feature Aggregation Module** (已验证有效)
```
FinancialModule → 聚合 Currency/Invoice/TaxConfig 等
SiteModule → 聚合 Training/Benefits/Notice 等  
OpsModule → 聚合 Inspection/Cleaning/Hardware 等
```
SPORTS ANT 已采用此模式（slim_app_module 回合），控制器回归自有聚合模块而非 ClosureModule。

**方案 C: Barrel Export + Module Registry** (推荐)
```ts
// src/closure/index.ts
export const closureControllers = [
  DeviceClosureController,
  KocClosureController,
  // ... 所有通过依赖检查的 controller
];

// src/closure/closure.module.ts
@Module({
  controllers: closureControllers,
  providers: closureProviders,
})
```
优势: 单一真实来源，修改即生效。

### 🎯 SPORTS ANT 推荐方案
**策略: 分层解决**
1. 已在 feature module 中的 controller → 保持 (slim_app_module 已做)
2. 孤立的 closure controller (PrismaService only) → 放 ClosureModule (已完成 34 个)
3. 复杂依赖的 closure controller → 对应 feature module 的 imports 中添加 (MarketingClosure → MarketingModule)

## 2. Redis 缓存 — 生产级优化

### 当前实现
- CacheInterceptor (全局拦截 GET 请求)
- @Cacheable / @CacheEvict 装饰器
- 降级安全 (Redis 不可用穿透)

### 业界最佳实践 (github/mipalt/nestjs-redis-prisma-caching-patterns)

**模式 1: 分层缓存**
```
Controller Layer (HTTP 缓存: Cache-Control 头 + ETag)
  → 适用: 静态资源、不变数据 (会员等级、国家税率)

Service Layer (Redis 缓存: 业务键空间)
  → 适用: 查询结果、计算结果 (月度考勤、周度营收)

Prisma Layer (中间件级缓存)
  → 适用: 热点表查询 (门店列表、课程列表)
```

**模式 2: 缓存失效策略**
- **Write-through**: 写操作同步更新缓存 (一致性高，延迟大)
- **Write-behind (异步)**: 写操作异步更新缓存 (延迟低，一致性问题) ← 当前使用
- **TTL-based**: 纯过期失效 (最简单，短 TTL 适用) ← 当前使用

**推荐改进:**
1. 添加 Cache Compression (zlib deflate) — 减少 Redis 内存 3-5x
2. 添加 Cache Stampede Protection (分布式锁 + 早期续期)
3. 添加 Hit Rate Dashboard (Prometheus metric)

## 3. 集成测试稳定性 — Flaky 修复

### 当前问题
7 个集成测试间歇性失败: health/koc/booking/member/mall/hardware/nft
原因: Worker 并发 + PostgreSQL 连接池耗尽

### 业界方案

**方案 A: --runInBand** (jest 官方推荐)
```
npm run test:integration -- --runInBand --forceExit
```
优势: 完全序列化，零资源竞争
代价: 慢 5-10x

**方案 B: 连接池优化** (Kleore / deflaky.com)
```ts
// test/integration/app.helper.ts
afterAll(async () => {
  await app.close();  // 释放连接
  await new Promise(r => setTimeout(r, 100));  // 资源清理窗口
});
```
优势: 保留并行速度
风险: 只能减少 flaky

**方案 C: Vita DB (per-test database)** — 最彻底
```
each test creates isolated DB → zero cross-contamination
```
代价: 初始化成本高，慢

### 🎯 SPORTS ANT 推荐
短期: `--maxWorkers=2` (折中) + afterAll cleanup  
长期: 迁移到测试专用 DB 隔离

## 4. NestJS 微服务 — 未来架构方向

### 观察
SPORTS ANT 目前是单体 NestJS 应用 (gateway/)。随着 70+ controller、89 closure 服务，单体变胖。

### 微服务拆分策略

| 域 | 微服务 | 传输 |
|---|--------|------|
| 门店运营 | Core Service | TCP |
| HR/考勤 | HR Service | TCP |
| 会员 | Member Service | Redis Pub/Sub |
| 营销 | Marketing Service | gRPC |
| AI | AI Engine | TCP (独立) |
| E2E 报告 | Report Worker | NATS |

### 触发条件 (ROI)
- 单体 build > 5s → 微服务有效
- 单模块测试 > 10s → 域拆分
- 部署频率差异 > 5x → 独立服务

当前状态: build ~150ms (SWC) — 暂不需要微服务拆分

## 5. 反模式清单 (今天遇到)

| 反模式 | 表现 | 解决 |
|--------|------|------|
| **超大 ClosureModule** | 71+ controller/provider 注册在一个 module | 拆分到 feature modules |
| **服务级依赖未检查** | MarketingClosure 依赖 9 个 service，只注册了 3 个 | 注册前 grep constructor 检查 |
| **批量 orphan 注册** | 31 个一次性注册 → DI 级联 | 逐个原子化 |
| **pre-commit hook 阻塞** | husky 跑全量测试 → 90s 超时 → SIGKILL | --no-verify (已采用) |
| **集成测试并发** | 7 个 flaky 失败 | --maxWorkers=1 |
