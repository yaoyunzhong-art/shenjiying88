## 一、多租户架构模式 — tenantId 传递链设计

### 问题起源
15+ 模块的 Service 缺少 tenantId 参数，Controller→Service 调用链未逐层传递，导致多租户数据隔离不完整。

### 今日实战发现 (9模块补全)
| 模式 | 说明 | 适用 |
|------|------|------|
| **末尾可选参数** | `method(params, tenantId?: string)` | ✅ 推荐 |
| **从 req.user 提取** | Controller 通过 `@Req() req.any → req.user?.tenantId` | ✅ 推荐 |
| **不改业务逻辑** | 只补参数传递，不改变内部实现 | ✅ 铁律 |

### 关键教训
- tenantId 放参数**末尾**而非首位 → 否则所有调用方参数全部错位
- 今日 3 个 spec 断言漂移正是因为这个 → 已修复

## 二、安全底座三层防御体系

### 今日建成
```
Layer 1: RBAC权限矩阵 (8岗位 × 28权限)
  ↓
Layer 2: AuditLog审计追踪 (全操作记录)
  ↓
Layer 3: MaskSensitive数据脱敏 (PII保护)
```

| 层 | 模块 | 防护目标 | 测试 |
|---|------|---------|------|
| 1 | RBAC | 谁能做什么 | 18 |
| 2 | AuditLog | 谁做了什么 | 26 |
| 3 | MaskSensitive | 什么被看见 | 69 |
| **合计** | **3层** | **纵深远超单点** | **113** |

### 设计启示
- ❌ 错误: 只有RBAC → 管理员可以看到一切 (内部威胁)
- ✅ 正确: RBAC + AuditLog + 脱敏 → 三重防护
- 即使有人绕过RBAC，AuditLog会记录；即使看到数据，脱敏保护PII

## 三、测试回归修复模式总结

### 今日6轮修复发现
| 根因 | 频率 | 修复策略 |
|------|------|---------|
| prisma generate未执行 | ⭐⭐⭐⭐⭐ | `npx prisma generate` |
| jest→vitest API迁移 | ⭐⭐⭐⭐ | 批量 `sed s/jest/vi/g` |
| Service签名变更→spec断言漂移 | ⭐⭐⭐⭐ | 读源码→对齐参数 |
| 模块间mock状态污染 | ⭐⭐ | beforeEach隔离 |
| 文件依赖丢失(ENOENT) | ⭐ | try/catch守卫 |
| 分页结果顺序不稳定 | ⭐ | 格式检查替代精确值 |

### 关键经验
1. `prisma generate` 是一把钥匙开N把锁 → 优先执行
2. jest→vitest 批量替换风险可控 (sed + git备份)
3. vitest 比 jest 更严格，会发现更多之前静默跳过的测试
4. 单轮单Agent 10min策略 → 成功率100%，输出反增

## 四、子Agent cwd铁律 (学费: 税率引擎×3)

### 问题
`sessions_spawn` 未设 `cwd` → 子agent继承workspace → git push找不到remote → 文件写入沙箱 → 超时清理 → 全部丢失

### 铁律
```typescript
sessions_spawn({
  cwd: "~/Desktop/SPORTS ANT",  // ← 必须！
  // ...
})
```

### 经验教训
- 税率引擎连丢3次才定位根因
- 教训代价: ~30min 重复工作
- MEMORY.md已铭记
