## 一、薪酬引擎架构 (Payroll Engine Patterns)

### 1.1 行业标准: 三层调节架构
来源: PayrollEngine.org (MIT开源薪酬引擎)

```
国家调节层 (Country Regulations)
  → 行业调节层 (Industry Regulations)  
    → 公司调节层 (Company Regulations)
      → 员工薪资计算
```

**核心思路**: 不硬编码规则，而是可组合的、有版本的调节层。上层覆盖下层。

**SPORTS ANT 适用性**: 已有 tax-config 模块 (8国税率)，可扩展为 Regulation Layer 模式。

### 1.2 薪酬处理管道 (Payrun Pipeline)
```
Case → Payrun → Results & Reports
 输入    处理      输出
```

- **Case**: 工资结构、缺勤、福利作为时间戳事件
- **Payrun**: Wage types + Collectors 在调节层中计算
- **Output**: 工资单、法定申报、数据导出

### 1.3 跨国薪酬的核心挑战 (dev.to 实战)
**"同一薪资幻觉"**:
- €60k在德国≠€60k在瑞士≠€60k在美国
- 德国: 社保嵌入工资扣款
- 瑞士: 医保外挂在工资系统之外  
- 美国: 联邦+州+FICA+雇主赞助 碎片化

**两层架构**:
1. **国别执行层**: 税级、养老金、社保规则、申报类别
2. **标准化输出层**: 有效税率、可支配收入、购买力信号

**最难的不是税**: 是定义"净收入"在各国意味着什么

## 二、数据中台架构

### 2.1 OpenTelemetry 可观测性三支柱
- **Traces**: 请求全链路追踪
- **Metrics**: 指标聚合 (Prometheus兼容)
- **Logs**: 结构化日志

### 2.2 数据血缘 (Data Lineage)
```
源表 → ETL转换 → 聚合指标 → 看板展示
  ↓        ↓         ↓         ↓
timestamp + transform + source + consumer
```

**关键字段**: sourceTable, transformChain, freshness, lastUpdated

### 2.3 数据质量监控维度
- **Freshness**: 数据最后更新时间 (超过阈值→告警)
- **Completeness**: 缺失值比例 (NULL检测)
- **Accuracy**: 与基准值的偏差
- **Consistency**: 跨表数据一致性校验

  - ❌ 数据血缘追踪
  - ❌ 质量监控
  - ❌ 异常检测
  - ❌ 预测分析

## 三、P6 设计决策

### 3.1 薪酬闭环 (Payroll Closure)
**闭环流程**: 月结触发 → AI核验(考勤+绩效) → 薪资计算 → AI异常检测 → 审批 → 生成工资单 → 发放

**新服务**:
- `payroll_closure.service.ts`: 月结管道 + AI核验 + 异常检测
- `payslip.service.ts`: 工资单生成 (PDF/HTML)
- `payroll_anomaly.service.ts`: AI异常检测引擎

### 3.2 数据中台闭环 (Data Platform Closure)
**闭环流程**: 数据采集 → 血缘记录 → 质量检测 → 异常告警 → 预测

**新服务**:
- `data_lineage.service.ts`: 数据血缘追踪
- `data_quality.service.ts`: 质量监控 (Freshness/Completeness/Accuracy)
- `data_anomaly.service.ts`: 指标异常检测

### 3.3 优先级排序
1. **P0**: 薪酬月结管道 + AI异常检测 (直接业务价值)
2. **P1**: 数据质量监控 (系统健康)
3. **P2**: 数据血缘追踪 (治理)
4. **P3**: 工资单生成 (体验)
