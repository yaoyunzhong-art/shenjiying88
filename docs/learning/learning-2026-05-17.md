## 二、五大技术域最新方案研究 (2025-2026)

### 域 1: 工作流引擎 — Temporal vs Camunda vs 自研

#### 2026 年格局
- **Camunda 8**: BPMN 可视化建模 + Zeebe引擎，企业标准。Apache 2.0 自托管 / SaaS $200/mo。
  - ✅ 适合: 审批流、人工任务、组织架构审批链
  - ❌ 不适合: 代码驱动、异步补偿、微服务Saga
- **Temporal**: 代码优先，Go/Java/TS/Python SDK。MIT 许可。
  - ✅ 适合: 长时间运行事务、重试/补偿、分布式Saga
  - ❌ 不适合: 需要非技术人员编辑流程
- **自研 WorkflowEngine** (当前方案):
  - ✅ 适合: 业务逻辑简单、流程数量 < 20
  - ❌ 不适合: 复杂分支、Saga补偿、可视化编排需求

#### 核心教训: "动态审批矩阵" 模式 (Dynamic Approval Matrix)
```
硬编码 if-else → 数据库驱动的规则矩阵 → 业务人员可配置
```
- **矩阵表**: `approval_rules(condition_type, condition_value, approver_role, stage, escalate_to, timeout_hours)`
- **引擎**: 运行时不读代码，查矩阵表动态计算审批路径
- **关键**: 规则外部化 → 改规则不改代码 → 业务自服务

#### 对Sports Ant的建议
当前 WorkflowEngine 模块（5 endpoints · 26 tests）是自研引擎。P0-8 的 70% 完成度大概率是基本状态机。建议：
- 短期 (Round 59-62): 完善自研引擎 — 支持 Multi-stage、Escalation、Parallel Branch、Timeout
- 中期 (Round 63+): 评估引入 Camunda 做审批流可视化编排
- 长期: Saga 模式用 Temporal 做分布式事务补偿

### 域 2: AI底座 — LangChain + NestJS 2026 实践

#### 2026 年 NestJS+AI 核心模式

1. **AI Gateway 模块** (最重要!)
   ```
   AIGatewayModule
   ├── summarization.service.ts  → 摘要/分析
   ├── extraction.service.ts     → 实体提取
   ├── generation.service.ts     → 内容生成
   ├── embedding.service.ts      → 向量化
   └── rag-pipeline.service.ts   → RAG检索增强
   ```
   - 统一 AI 调用入口，所有 AI 请求经此路由
   - 支持多 Provider (OpenAI / Claude / DeepSeek / 本地模型)
   - 内置 fallback, retry, budget, rate-limit, audit log

2. **RAG Pipeline** (检索增强生成)
   - Chunk → Embed → Store (pgvector) → Retrieve → Rerank → Generate
   - Multi-tenant: 每个 tenant 独立向量命名空间
   - 评价体系: recall@k, precision, answer relevance

3. **LangGraph.js** (Agent 工作流)
   - 替代 LangChain chains → 用 Graph 定义 Agent 决策流
   - `StateGraph` 节点 + 条件边 → 可控、可观测、可测试
   - 适合: KOC裂变推荐、智能客服、经营诊断AI

4. **生产级 AI 管道规则** (devflares.com playbook)
   ```
   Sanitize Input → Assemble Context → Call Model → Validate Output → Store & Audit → Feedback Loop
   ```
   每个步骤必须有: schema 约束、timeout、retry with jitter、per-tenant budget、output validation

#### 对Sports Ant的建议
当前 `ai-engine/` 目录仅4个文件 (controller/service/module/types)。P0-9 的 70% 大概率是基础 LLM 调用。
- 短期: 建立 AIGateway 门面模块，统一多模型路由
- 中期: 构建 RAG Pipeline (pgvector + tenant隔离)
- 长期: LangGraph Agent 做自主诊断/推荐

### 域 3: 店长经营诊断 — 零售KPI仪表盘最佳实践

#### 2025 零售 KPI 仪表盘核心指标分层

| 层级 | 指标 | 频率 |
|------|------|------|
| 📊 宏观 | 营收增长率、净利润率 | 月/季 |
| 👥 客户 | 客单价(AOV)、客户终身价值(CLV)、复购率 | 月 |
| 💰 销售 | 完成订单数、分渠道销售额、售罄率 | 日/周 |
| 📦 库存 | 库存周转率、断货率、滞销品占比 | 日 |
| 😊 体验 | 满意度评分、响应时间、转化率 | 周 |

#### 诊断闭环的核心设计原则

```
数据采集 → 指标计算 → 异常检测 → 原因分析 → 建议生成 → 执行追踪 → 回测验证
   │                                                              │
   └──────────────── 闭环反馈 ────────────────────────────────────┘
```

关键设计点:
1. **指标可配置**: 不同门店/业态不同关键指标，不能硬编码
2. **阈值热更新**: 异常判定规则应可调，无需重新部署
3. **诊断建议分级**: 紧急→警告→建议，带置信度分数
4. **执行追踪闭环**: 店长看建议→执行→系统追踪效果→反馈到模型

#### 对Sports Ant的建议
`diagnosis/` 模块 (3 endpoints · 26 tests) 是最全测的模块。P0-1 已完成 80%。
- 剩余 20%: 指标可配置化、告警通知推送、执行追踪闭环
- 数据源: 联调 Sales/Member/Inventory/Booking 模块做交叉分析

### 域 4: 店长审批闭环 — 多级审批 + 拒绝链

#### 审批流核心场景 (潮玩馆SaaS)
```
门店申请(采购/调价/活动/退款) → 店长审批 → 区域经理 → 总部 → 财务
                                 ↓ 拒绝
                              原路退回 + 原因
```

#### 关键技术设计

1. **审批矩阵表设计**
   ```sql
   -- 动态规则驱动审批路径
   approval_rules (
     rule_type ENUM('amount','category','urgency'),
     condition JSONB,          -- {"min": 5000, "max": 50000}
     tenant_id UUID,
     stages JSONB,             -- [{order:1, role:"store_manager"}, {order:2, role:"area_manager"}]
     escalation_hours INT,
     is_active BOOLEAN
   )
   ```

2. **拒绝链级联**: 任一环节拒绝 → 自动退回申请人 + 通知所有已审批人
   - 树形结构，拒绝即剪枝，不回退已通过环节的决策

3. **审批超时升级**: `escalation_hours` 超时 → 自动升级到上级角色
   - 防止一人在请假时流程卡死

4. **并行分支**: 多部门并行审批(如法务+财务同时审) → 全部通过才继续

#### 对Sports Ant的建议
`approval-workflow/` 已有 5 endpoints · 22 tests。P0-2 完成度 80%。
- 剩余 20%: 动态审批规则(摆脱硬编码)、拒绝链/超时升级、审批历史审计

### 域 5: 供应链采购 — SaaS化采购系统设计

#### 采购管理核心模块

```
供应商管理 → 采购申请 → 询价/比价 → 采购订单 → 收货入库 → 对账结算
   │              │                    │            │
   └── 评价体系 ──┘                    └── 库存联动 ──┘
```

#### 模块边界设计
- **供应商管理**: 档案、资质、评价(质量/交期/价格/服务 四维评分)
- **采购申请**: 多级审批流(复用 ApprovalWorkflow)
- **采购订单**: PO 生命周期(待确认→已确认→部分收货→完成→已取消)
- **入库对接**: 与 Inventory 模块联动,自动更新库存
- **对账**: 采购金额 vs 实际收货 vs 付款,三单匹配

#### 对Sports Ant的建议
`procurement/` 有 5 endpoints · 26 tests。P0-7 完成度 70%。
- 剩余 30%: 供应商评价模型、库存联动(与Inventory模块集成)、采购审批钩子

## 三、跨域架构问题 (超越 P0 清单)

### 3.1 模块间耦合度分析

当前 25 Controller 仍在一个 NestJS 进程内 (Modular Monolith)。关键问题:

| 模块对 | 应有关联 | 当前状态 |
|--------|---------|---------|
| Procurement → ApprovalWorkflow | 采购需审批 | ⚠️ 可能各自实现,未复用 |
| Procurement → Inventory | 收货即入库 | ⚠️ 可能手动同步,未联动 |
| Diagnosis → Sales/Member/Inventory | 诊断需交叉数据 | ⚠️ 可能直接读表,未走 Service facade |
| AIEngine → Diagnosis | AI分析诊断 | ⚠️ 可能未集成 |
| WorkflowEngine → ApprovalWorkflow | 引擎应驱动审批 | 🟢 Engine 是底层,Approval是其应用 |

### 3.2 Modular Monolith 何时拆?

**当前是正确选择**。Modular Monolith 适合:
- < 10 人团队 ✅ (树哥 + 龙虾哥)
- 业务边界仍在演化 ✅ (25 Controller 仍在增加)
- 模块间数据共享频繁 ✅

**拆分信号** (触发即拆):
- 某模块独立部署需求 (> 部署频率差异大)
- 某模块独立扩缩容 (> 负载差异大)
- 某团队说"不想等另一个模块的 CI"

**推荐迁移路径**: Strangler Fig
1. Module facade 加接口层
2. 目标模块抽离为独立 NestJS 微服务 (通过 `@nestjs/microservices`)
3. 旧调用点逐步迁移到消息/HTTP
4. 最终下线旧代码
