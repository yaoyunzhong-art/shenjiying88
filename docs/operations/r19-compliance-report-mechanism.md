## 二、偏离识别与整改流程

### 2.1 偏离发现机制

```
每周自动扫描流程:
1. 读取开发任务映射表（所有任务卡片）
2. 提取所有 6-8_refs 字段中的 BS 编号
3. 对比 BS 编号总表（308条）
4. 发现未被任何任务覆盖的 BS 编号 → 登记为偏离
5. 发送偏离报告到 maintainers@m5

手动触发:
- PR 提交时自动触发合规检查
- 合规检查不通过 → 生成偏离记录
- 每个偏离自动生成唯一编号 DEV-nnnn
```

### 2.2 偏离分级

| 级别 | 定义 | 后果 | 解除条件 |
|:----:|------|------|----------|
| **P0 致命偏离** | 安全/合规/核心架构缺失 | 不可合并（Block Merge） | 修复+Champion复审 |
| **P1 严重偏离** | 重要功能/用户体验/数据规范 | 需Champion审批+工期承诺 | 审批通过+规划修复 |
| **P2 轻微偏离** | 优化项/非核心/长期迭代 | 可后续补+记录 | 纳入下期迭代计划 |

### 2.3 整改流程

```
识别 → 登记 → 派发 → 修复 → 验收 → 关闭

Step 1: 偏离识别
  - 自动扫描发现未被覆盖的BS编号
  - 手动评审发现合规不通过项

Step 2: 登记偏离
  - 生成 DEV-nnnn 编号
  - 登记到偏离清单(严重度/责任人/期限)

Step 3: 派发修复
  - P0: 直接生成开发任务+指定责任人
  - P1: Champion审查+派发
  - P2: 排入迭代计划

Step 4: 执行修复
  - 开发人员按合规阀门流程提交PR
  - PR必须包含 6-8_refs + 合规声明

Step 5: 验收关闭
  - 合规验证人复审
  - 更新BS覆盖矩阵: ❌→✅
  - 关闭偏离单

超时升级规则:
  - 超过2次未整改 → 自动升级为P0
  - P0超过原定期限1倍 → 升级至委员会决议
```

## 三、合规Dashboard指标设计

### 3.1 核心指标

| 指标 | 公式 | 目标值 | 监控频率 | 告警阈值 |
|------|------|:------:|:--------:|:--------:|
| **覆盖率** | 已覆盖BS数/总BS数 × 100% | ≥90% | 每周 | <60% 红色告警 |
| **通过率** | 已验证通过BS数/已覆盖BS数 × 100% | ≥95% | 每周 | <80% 黄色告警 |
| **偏离率** | 偏离项数/总BS数 × 100% | ≤5% | 每周 | >15% 红色告警 |
| **闭环时效** | 偏离发现→闭环平均天数 | ≤14天(P0)/≤30天(P1)/≤90天(P2) | 月度 | 超过目标50%告警 |

## 五、Compliance工具体系

### 5.1 脚本工具设计

```
# tools/compliance-check.sh
# 用法: ./compliance-check.sh [pr-number]

# 功能:
# 1. 解析 PR 内容 → 提取 6-8_refs
# 2. 查询 BS 映射表 → 提取所有涉及的 BS 编号
# 3. 验证合规声明完整度
# 4. 检查偏离清单是否有待处理项
# 5. 输出合规报告

# tools/compliance-report.sh
# 用法: ./compliance-report.sh [--full|--summary]

# 功能:
# 1. 全量扫描任务卡片
# 2. 重新计算覆盖率/通过率/偏离率
# 3. 生成偏离清单
# 4. 输出 markdown 报告
```

### 5.2 存储结构

```
.trae/compliance/
├── bs-catalog.json           # BS编号总表(308条)
├── deviation-registry.json   # 偏离登记表
└── coverage-matrix.json      # 覆盖矩阵(自动生成)
```
