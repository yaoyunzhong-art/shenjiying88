## 四、合规阀门设计

### 4.1 合规阀门动作流程

```
开发任务创建
    ↓
开发人员在任务卡中填写 6-8_refs + 合规声明
    ↓  [阀门1]
Code Review检查合规声明是否完整 → 不完整→退回修改
    ↓  [阀门2]
CI/CD合规检查步骤执行:
  1. 解析 PR 中修改文件
  2. 匹配 6-8_refs 对应的 BS 编号
  3. 检查合规声明是否覆盖所有涉及的 BS
    ↓  [阀门3]
不合规 → PR自动标记 [COMPLIANCE_FAIL] + BLOCK MERGE
通过 → 允许合并
    ↓  [阀门4]
合并后 → 更新 BS 覆盖矩阵 → 触发 Compliance 报告更新
```

### 4.2 CI/CD合规检查伪代码

```
# pipeline/.compliance-check.sh
# 在 PR merge 前执行

1. 提取 PR 修改的文件列表: git diff --name-only origin/main...HEAD
2. 解析 PR 描述中的 6-8_refs 字段
3. 查询 BS 映射表，获取涉及的 BS 编号
4. 验证合规声明完整覆盖：
   - 每个 6-8_refs 条目都在合规声明中有明确状态
   - 状态为 "待验证" 的需审批
   - 状态为 "未通过 无" 的直接 BLOCK
5. 输出结果：
   - PASS: 合规检查通过
   - FAIL(需审批): 有 "待验证" 条目，需 Champion 签名
   - BLOCK: 不完整或未通过，自动添加 compliance-fail 标签
```

### 4.3 不合规处理

| 违规类型 | 标签 | 后果 | 解除条件 |
|----------|:----:|------|----------|
| 合规声明缺失 | COMPLIANCE_MISSING | PR无法合并 | 补充合规声明 |
| 6-8_refs不完整 | REFS_INCOMPLETE | PR标记待处理 | 补充全部引用 |
| 合规未通过 | COMPLIANCE_FAIL | BLOCK MERGE | 修改代码至合规 |
| 待验证条目需审批 | NEEDS_APPROVAL | 需Champion签名 | Champion审批通过 |
