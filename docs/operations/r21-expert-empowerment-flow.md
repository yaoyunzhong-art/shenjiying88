## 一、44位专家的五重身份

每一位 expert/E*.md 卡片中的专家同时拥有这5种身份：

### 1.1 使用者 👤
- 系统是给专家用的
- E11钱店长要用收银台 → 他提收银台需求
- E4张营销要用推送系统 → 他提推送需求
- E8周运营要用看板系统 → 他提看板需求
- **"不用这个功能我干不了活" = 最高优先级**

### 1.2 策划者 📋
- 开发计划由专家制定
- 技术组制定技术路线（E1+E2+E3+E44）
- 业务组制定功能路线（E4+E5+E6+E8+E10+E11）
- 用户组制定体验路线（E7+E9+E12+E40）
- 监督组制定合规路线（E36+E38+E6+E2）
- **TASKS_STATUS.md 的内容由专家决定**

### 1.3 需求提出者 💡
- 每个功能的原始需求由专家提出
- 格式: `[需求] E4张营销: 我需要推送分级P0-P3功能，因为每日发太多推送干扰正常运营`
- 需求必须附带"使用场景"说明
- **没有使用场景的需求不开发**

### 1.4 测试者 🧪
- 专家从使用者角度测试功能
- 不是跑单元测试，是**真实场景验证**
- E11钱店长测试收银台: "我点3下能完成收银吗?"
- E4张营销测试推送: "我能不能一键选所有铂金会员?"
- **测试标准: 30秒内完成核心操作**

### 1.5 验收者 ✅
- 专家签收功能完成
- 验收标准:
  - 功能满足使用场景
  - 30秒内能完成核心操作
  - 没有多余步骤
  - 不会用错
- **专家说"好用了" = 功能闭环**

## 二、开发流程 (以专家为核心)

### 2.1 需求产生阶段
```
🏛️ E11钱店长(使用者): "我需要一个排队看板"
    ↓ (写入 expert/E11-qian-store.md 需求章节)
    ↓
🦞 整理需求: "排队看板P0需求"
    ↓
🏛️ 技术组(E1+E44): 评估技术可行性
🏛️ 业务组(E8+E11): 确认业务合理性
    ↓
🦞 创建开发任务 → TASKS_STATUS.md
```

### 2.2 计划和设计阶段
```
🏛️ E11钱店长(策划者): "看板上要显示实时排队人数+预估等待时间"
🏛️ E1陈架构: "用SSE推送, 不能轮询"
🏛️ E2李安全: "排队的用户ID要脱敏"
    ↓
🦞 整合为设计文档
    ↓
🏛️ 涉及专家全部签署 → 进入开发
```

### 2.3 开发阶段
```
🐜 树哥按照专家设计文档开发
    ↓
🏛️ E1陈架构(监督): "SSE推送实现符合规范吗?"
🏛️ E2李安全(监督): "数据脱敏做了吗?"
    ↓
🦞 进展跟踪
```

### 2.4 测试和验收阶段
```
🏛️ E11钱店长(测试者): 打开看板 → 操作1次
    - "排队人数实时更新了吗?" ✅
    - "预估时间准吗?" ✅
    - "我能看所有门店吗?" ❌ 缺这个功能!
    ↓
🦞 记录: 缺跨门店查看
🦞 派树哥加功能
    ↓
🏛️ E11钱店长(测试者): 再次测试
    - "现在可以了，**好用了**"
    ↓
✅ 功能闭环
```
