## 二、每日流水线

### 选择当日 Phase 的规则

每轮开发周期（每1-2天）聚焦 **1-2个主 Phase + 周边 Phase**：

```
今日开发 = P-主(2人) + P-辅(1人)
  主Phase产生80%开发量，辅Phase产生20%
  每个Phase必须有明确的专家需求来源
```

## 三、专家五重身份在每日落地

```
                    🏛️ 专家
                     ┌──────┐
         使用者 ───→ │ 需求 │ ←─── 验收者
                     │ 驱动 │
         策划者 ───→ │      │ ←─── 测试者
                     └──────┘
                        │
                        ↓ 需求提出者
```

### 使用者（专家说"我需要这个"）
- 每天轮到 Phase 的专家，其 `experts/E*.md#使用场景` 章节就是需求来源
- 例：E13李收银说"我要扫码枪超时自动取消收银" → 这就是P-35的P0需求

### 策划者（专家说"应该这样设计"）
- 在15:00午会中，业务专家对开发中的功能提出改进意见
- 专家不是等开发完再看，而是开发中参与

### 需求提出者（专家说"要A/B/C"）
- 10:00晨会后，技术组签发每个Phase的详细需求卡
- 需求卡必须有使用场景说明
- 没有使用场景的需求不开发

### 测试者（专家说"试试看能不能用"）
- 每项功能开发完成后，对口专家做使用者测试
- 测试标准：核心操作≤30秒，不会用错
- 不是跑代码测试，是真人士操作模拟

### 验收者（专家说"好用了"）
- 20:00晚会中，本Phase的Champion签署
- 6道门全部通过 → "好用了" → 闭环
- 任何一门不过 → 退回
