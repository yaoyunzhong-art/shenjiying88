## 三、审计对齐方案

### 原则：旧代码不重写，但做"需求追溯"

每个已有的 Phase 做三件事：

```
1. 写PRD (描述"这个Phase应该做什么")
   ↓
2. 现有代码→映射到PRD的RQ需求卡
   ↓
3. 标记缺口 (PRD写了但代码没有的 → 生成新任务)
```

### P-35 审计对齐示例（已做）

```
PRD-001 写了 10个RQ需求卡
树哥产出: 收银台页面+45个测试
对齐结果:
  ✅ RQ-35-01~08 已有对应代码 (商品选择/套餐/金额/会员/折扣/支付/成功)

  ⬜ RQ-35-10 积分累计 (P1) → 依赖P-36会员API
PRD覆盖率: 8/10 = 80%
```
