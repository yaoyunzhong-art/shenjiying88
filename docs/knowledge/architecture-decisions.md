#### ADR-041: AI D2推荐引擎算法决策

| 字段 | 内容 |
|:----|------|
| **标题** | AI D2推荐引擎算法 |
| **日期** | 2026-07-15 |
| **决策** | 采用趋势热度权重+三维用户画像+协同过滤的混合推荐引擎 |
| **原因** | V18试跑发现单一协同过滤命中率仅32%，无法感知用户隐式行为模式（如常兑换食品类但未兑换过的饮料品牌）；规则引擎130条if-else维护成本高 |
| **影响** | 推荐命中率从32%提升至68%+；新增用户行为Embedding化(pgvector)+实时重排(ONNX轻量模型)双阶段；替换原有纯规则引擎，保留规则作为审计兜底 |
| **状态** | ✅ 裁定通过 |

#### ADR-042: P-38对账升级为财务健康仪表盘决策

| 字段 | 内容 |
|:----|------|
| **标题** | P-38对账升级为财务健康仪表盘 |
| **日期** | 2026-07-15 |
| **决策** | P-38支付对账模块升级为全维度财务健康仪表盘，覆盖支付渠道级对账 |
| **原因** | 原P-38仅做交易级对账，缺乏支付渠道粒度（微信/支付宝/银联）独立核算能力；V18暴露商户资金偏差无法快速定位到具体渠道 |
| **影响** | 新增渠道级对账面板：微信→支付宝→银联独立核算+差异预警；支持按渠道/时段/门店三维筛选；升级后财务月结时间从3天缩至4小时 |
| **状态** | ✅ 裁定通过 |

#### ADR-043: SLA看板对标凤梨30秒响应决策

| 字段 | 内容 |
|:----|------|
| **标题** | SLA看板对标凤梨30秒响应 |
| **日期** | 2026-07-15 |
| **决策** | SLA监控看板以凤梨系统30秒响应为标准，建立全链路SLA指标体系 |
| **原因** | V18验收脉冲暴露出44分钟实际延迟vs6秒表显的假阳问题，缺乏真实端到端SLA监控；对标凤梨行业标准的30秒响应能力 |
| **影响** | 新增全链路SLA仪表盘：API响应时间(30s目标)→数据库查询(5s)→缓存命中(1ms)→三方支付回调(10s)；建立SLA达标率日/周/月报表；超出阈值自动升级告警至P0通道 |
| **状态** | ✅ 裁定通过 |

| ADR-044 | 全天候V20 cron调度架构 |
|:----|------|
| **标题** | 全自动无间断开发调度体系 |
| **日期** | 2026-07-19 01:55 |
| **决策** | 部署12段cron全天自动调度：02:00→04:00→06:00→08:00→10:00→12:00→14:00→16:00→18:00→20:00→22:00→00:00。每2小时自动开发→验收→修复→闭环。所有cron使用systemEvent模式唤醒主session驱动。| 
| **原因** | V20要求全天全自动无间断开发；人工值守模式无法覆盖24h；旧cron体系有多个error（网络断开/超时/模型idle）| 
| **影响** | 1) 主session会被频繁唤醒(每2h)，需注意session膨胀/炸裂风险 2) 树哥AM-011不持久化问题影响自动修复链可靠性 3) 需要增加熔断机制——TSC失败自动暂停新派单 4) P-31/P-37截止Phase必须在前4段（02:00-08:00）优先完成 |
| **状态** | ✅ 裁定通过 |
