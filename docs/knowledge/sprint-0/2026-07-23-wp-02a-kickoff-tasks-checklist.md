> 工作包: `WP-02A`
> 对应 BS: `BS-0030 ~ BS-0047`, `BS-0060`
> 优先级: `P0`
> 关联总表: `docs/knowledge/2026-07-23-6-8-development-master-backlog-v2.md`

---

## Kickoff

### 目标

- 把“多租户、品牌、门店、数据归属、共享控制”从抽象要求落成可核验清单。
- 明确当前系统哪些地方已经具备 `tenant/brand/store` 维度, 哪些仍是单租户或弱隔离。

### 本包范围

- 三级组织架构和租户实体边界
- tenant/brand/store 上下文透传
- 数据归属、共享、冷热分层、生命周期入口
- 只读/投资人视角与跨品牌互通边界

### 不在本包范围

- 不实现所有业务模块的完整租户化改造
- 不在 Sprint-0 内完成所有历史数据迁移
- 不涉及 TLS/AES/2FA 等安全侧内容

### 入场条件

- `WP-00` 已提供底座现状
- 当前 API / storefront 已存在 `tenant/brand/store` 头透传经验
- `WP-COMPLIANCE` 已定义 `6-8_refs` 和证据规则

### 退出标准

- 形成多租户当前态矩阵
- 明确当前已具备的上下文透传链
- 明确弱隔离模块清单
- 给出 `P0` 范围内必须先补的隔离项

### 风险

- 如果只看头部传参, 容易误判数据层也已隔离
- 如果只看代码结构, 容易忽略历史内存态/单例态污染风险

### 证据要求

- tenant 相关模块/中间件/guard 的代码路径
- storefront scope 透传路径
- 实际接口或测试证据
- 弱隔离点清单

### 回滚口径

- Sprint-0 以基线核验和差距分级为主
- 若产生配置或默认租户策略调整, 必须单列 `RB-WP-02A-xxx`
