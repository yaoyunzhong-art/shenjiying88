# 🧬 44专家自学习·自进化系统 V1.0

> 版本: V1.0 · 生效: 2026-07-10
> 原理: 每专家每天3学习 × 3会议 × 3产出 = 9产出/人/天
> 约束: LLM并行配额 ≤3路 · 每个sub-agent处理1组(3-5人) · 科学分批

---

## 一、科学依据

### 1.1 为什么是3×3×3

| 维度 | 次数/天 | 理由 |
|:-----|:-------:|:-----|
| 学习 | 3 | 早(行业) + 午(代码) + 晚(跨界) |
| 会议 | 3 | 早(签阅Phase) + 午(审查代码) + 晚(签署决策) |
| 产出 | 3 | 早(洞察简报) + 午(审查意见) + 晚(开发建议) |

**总量**: 44人 × 9项/天 = 396项/天
**处理量**: 12组 × 3批/天 = 36 sub-agent次/天
**配额**: 3路并行 → 36÷3 = 12轮 → 每轮~5min → 全天~1h主session投入

### 4.2 sub-agent派单模板 (12组通用)

每组派1个sub-agent，处理该组全部3-5位专家：

```
任务: 专家组{组名} · 批号{1/2/3} · {日期}
━━━━━━━━━━━━━━━━━━━━━━━━━━
处理: {专家列表}
━━━━━━━━━━━━━━━━━━━━━━━━━━
每步输出写到该专家的 experts/E*.md 文件：
  ## 学习笔记 → 追加日期标题+内容
  ## 反馈日志 → 追加新行
  ## 投票记录 → 追加新行
  ## 专业洞察 → 追加新洞察

[批1: 晨间]
1. 对每人: web_search 该领域最新趋势, 3条insight
2. 检查git log --since=yesterday 该Phase提交, 签阅
3. 写K1洞察简报

[批2: 午间]
1. 对每人: git log --since=00:00 该Phase commit
2. 抽查1个测试文件质量
3. 写K2审查意见卡

[批3: 晚间]
1. 读另一组今日K1简报(跨界)
2. 签署今日Phase进展
3. 写K3明日开发建议
━━━━━━━━━━━━━━━━━━━━━━━━━━
约束: 只追加不删除 · 只改该组文件
```

## 五、质量保证

### 5.1 避免"假学习"规则

- ❌ 不允许AI编造搜索内容 → 必须实际调用web_search/wen_fetch
- ❌ 不允许日报模板化 → 每次L1学习必须有至少1个具体项目关联
- ❌ 不允许重复insight → 同组专家不能写一样的insight
- ✅ 每次产出必须包含: 数(人/行/commit) → 判断(好/差/需修) → 建议(具体怎么做)

### 5.2 质量监控cron

```yaml
name: "🧬 15:00 专家质量检查 [shenjiying88]"
schedule: "0 15 * * *"
sessionTarget: "main"  
payload.text: "🦞 质量检查: 抽查3个专家档案
  1. 检查今天是否写入了新内容
  2. 检查内容是否有具体项目关联
  3. 不合格的标记并重写"
```

### 5.3 自修复机制

- 审计发现某专家>3天未更新 → 补过期学习笔记
- 审计发现某专家insight与项目实际不符 → 修正
- 连续2次质量检查不合格 → 标记该组prompt需优化
