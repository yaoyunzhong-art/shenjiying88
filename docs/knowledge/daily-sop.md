## 3️⃣ 科学5Phase流程 (每任务必经)

```
Phase0: 需求卡 (🦞 2min)
  ├─ 大飞哥/脉冲 → 问题描述
  ├─ 🦞 派遣前三问:
  │   ① 哪个文件出问题了? (精确路径)
  │   ② 预期行为是什么? (当前vs期望)
  │   ③ 修正方向是什么? (修改什么)
  ├─ 如果三问答不出→ 🧠 先走根因分析再派
  └─ 输出: 任务卡(含P0/P1/P2分级)

Phase1: 设计 (🦞 3min)
  ├─ 涉及哪些文件? (scope)
  ├─ 有没有类似修复历史? (evolution-log)
  ├─ 需要改动多少行? (<10行/10-50行/50+行)
  └─ 输出: 设计要点+预计完成时间

Phase2: 开发 (🐜 10min超时)
  ├─ 单模块小粒度: entity→service→controller
  ├─ 10min未完成→自动标记🆘+回退
  ├─ as any/ts-ignore 必须注明理由
  └─ 输出: git commit

Phase3: 测试 (🐜 5min)
  ├─ 改前测: vitest run affected-module (基线)
  ├─ 改后测: vitest run affected-module (验证)
  ├─ 冒烟: smoke test (如果涉及页面)
  └─ 输出: 测试通过确认

Phase4: 验收 (🦞 3min)
  ├─ TSC --noEmit ✅?
  ├─ 测试比改前相等或增加?
  ├─ 模块整体全绿?
  └─ 如果fail→自动派树哥修(自我修复)
      如果连续2次fail→升级P0

Phase5: 部署 (🦞 1min) — 店A前只在本地
  ├─ git commit + 工作区清理
  └─ 更新phase-progress.md

⚠️ 跳过任一Phase→该任务标记"跳过PhaseN"→次日晨会追问
```

### 核心逻辑

> 不靠系统提醒，靠🦞直接点名

```

## 8️⃣ 发现→修复→验收→闭环 流水线

### 发现问题
```
来源: 大飞哥指正 / 脉冲扫描 / 审计 / cron
  ↓
🦞 ① 确认是否已知? (evolution-log查重)
   ↓(已知) 查当前修复状态
   ↓(未知) 继续↓
🦞 ② 派遣前三问:
   ├─ 哪个文件出问题了?
   ├─ 期望行为vs当前行为?
   └─ 修正方向?
  ↓(答不出) → 🧠 根因分析优先
  ↓(答得出) → 继续↓
🦞 ③ 分级: P0(立即) P1(24h) P2(本周) P3(店A后)
```

### 修复派遣
```
🦞 写出任务卡: 文件路径+预期+修正方向+超时时间
  ↓ (≤3路并行)
🐜▶️ 树哥开始修 (10min超时)
  ↓ 每步提交git (小粒度)
🦞 等待完成→验收
```

### 验收检查
```
✅ TSC --noEmit = 0?
✅ 模块测试 = 全绿?
✅ 测试数比改前增加or相等?
✅ 工作区 ≤5变更?
✅ 没有as any新增?
  ┌─ 全部通过 → 闭环 ✅
  └─ 任意失败 → 自动派树哥修
                   ↓
                连续2次失败 → 升级P0 🚨
```

### 闭环记录
```
✅ 记录到 evolution-log.md
  ├─ 问题: P0-A1 await兼容性
  ├─ 根因: getOrder变异步后16调用点未更新
  ├─ 修复: 6个文件await补充
  └─ 预防: 迁移后全链依赖检查
```

   └─ → V(N+1)改进草案

② 反模式检查
   ├─ 今天遇到什么新坑?
   ├─ 这是之前犯过的吗? (evolution-log查重)
   └─ 如果重复 → 提高治理级别 🚨

③ 正向模式检查
   ├─ 今天什么做得特别顺?
   ├─ 能否固化到 SOP?
   └─ → 写入正向模式库

④ cron健康检查
   ├─ 哪些cron挂了/撞了?
   ├─ 并发配额够不够?
   └─ → 修复/调整

⑤ V(N+1)产出
   ├─ 明天的任务优先级
   ├─ 明天的轮换专家
   └─ 明天的P0核心
```
