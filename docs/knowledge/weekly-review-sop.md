# 🏛️ 每周完整审校 SOP · V1.0

> 大飞哥指令：每周一次完整代码+页面审核
> 审校员 = 54专家(真人) + AI模拟资深技术专家
> 每位审校官在各自领域具备"独立否决权"
> 每项提交必须经过"人工级别核验"方可通过
>
> 编制: 2026-07-13 01:13 · 🦞龙虾哥
> 集成: `daily-sop.md V1.0 §5(专家对口) + §6(Gate签署) + §3(5Phase)`

---

## 0️⃣ 核心原则

1. **没人读的代码等于没写**：每周必须有人实际看过每条 `git diff`
2. **没人点的页面等于没做**：每周必须有人实际操作每个页面流程
3. **退回要有根因，不能只说"不行"**：退回必须带定位(文件+行号)+原因+修改建议
4. **AI是专家助手，不是替代品**：AI模拟专家生成初评，54真人专家最终签核
5. **通过≠完美，退回≠失败**：通过=功能正确+安全合规+可维护，退回=有明确问题需修复后重新提审
6. **一周不Review = 一周白干**：所有未Review的提交不进入正式版本

## 2️⃣ 审校流程：5步核验法

### Step 1: 自动化扫查（预先 · 耗时~30min）

> 自动化扫查由龙虾哥调度AI-E1~AI-E6执行，产出初评报告

```
┌─ 1.1 代码逻辑检查 ─────────────────────────────┐
│  AI-E1:  git diff HEAD~7d 扫查                   │
│  检查项: unreachable code / dead branch /         │
│          inconsistent naming / stubbed logic      │
│  产出: ✍️ 逻辑意见卡                             │
└──────────────────────────────────────────────────┘
┌─ 1.2 规范性检查 ───────────────────────────────┐
│  AI-E1 + Lint: ESLint / Prettier / TSC           │
│  检查项: 命名规范 / 文件组织 / import顺序        │
│  产出: ✍️ 规范性意见卡                           │
└──────────────────────────────────────────────────┘
┌─ 1.3 安全检查 ─────────────────────────────────┐
│  AI-E2:  grep -Rn 'dangerouslySetInnerHTML'      │
│           grep -Rn '.exec\|eval\|innerHTML'      │
│           grep -Rn 'any\|@ts-ignore'             │
│  检查项: XSS / 注入 / 硬编码密钥 / 类型逃逸      │
│  产出: ✍️ 安全意见卡                              │
└──────────────────────────────────────────────────┘
┌─ 1.4 可维护性检查 ─────────────────────────────┐
│  AI-E3:  圈复杂度 / 单文件行数 / 重复代码        │
│  检查项: 重复逻辑 / 超大函数 / 缺少注释          │
│  产出: ✍️ 可维护性意见卡                          │
└──────────────────────────────────────────────────┘
```

### Step 2: AI模拟专家代码审查（HR+ · 耗时~30min）

> 6位AI专家各自独立审阅全量代码，每人产出审查意见

| 专家 | 审查范围 | 审查时长 | 产出格式 |
|:-----|:---------|:--------:|:---------|
| 👴 AI-E1 架构陈老 | 全部代码文件(周diff) | 1h | 架构评审报告 |
| 🔒 AI-E2 安全李工 | 安全敏感文件筛查 | 30min | 安全评估报告 |
| ⚡ AI-E3 性能张工 | 前端页面+API热点 | 30min | 性能评估报告 |
| 🎨 AI-E4 UX王工 | 前端页面(tob+storefront) | 30min | UX评估报告 |
| 🧪 AI-E5 测试赵工 | 全部测试文件 | 30min | 测试评估报告 |
| 📦 AI-E6 DevOps刘工 | 构建/配置/CI | 15min | DevOps评估报告 |

**各AI专家独立运作，不互通结论，避免偏见叠加。**

### Step 3: 真人专家抽查验证（HR+ · 耗时~30min）

> 对口真人专家在AI初评基础上，实地核查 + 补充判断

从本周54人轮换清单中选取当日专家（每日8人）：

| 时间 | 任务 | 负责人 | 耗时 |
|:-----|:-----|:-------|:----:|
| 周六08:00 | 📋 收到AI初评报告 | 🦞龙虾哥分派 | 5min |
| 周六08:30 | 🔍 E1陈架构抽查3个TS文件 | E1对口 | 15min |
| 周六08:45 | 🔍 E4市场抽查PRD+业务代码 | E4对口 | 10min |
| 周六09:00 | 🔍 E7体验抽查前端页面代码 | E7对口 | 10min |
| 周六09:15 | 🔍 E40客户抽查业务关键流 | E40对口 | 10min |
| 周六09:30 | 🔍 E2安全抽查安全关键点 | E2对口 | 10min |

**独立否决权**：任何一位对口专家在发现严重问题时（P0安全漏洞/数据泄露/核心功能缺失），可直接退回整周提交。

### Step 4: 页面实操遍历（HR+ · 耗时~60min）

> 每周由🎨 AI-E4 UX王工 + 1位真人专家实地操作页面

**操作范围（店A关键页面）**：

```

│  □ 收银台(/cashier) — 选品+结算+支付+打印          │
│  □ 会员中心(/member-center) — 查看+续费            │
│  □ 自助充值(/self-recharge) — 金额选择+支付        │
│  □ 设备预约(/device-reservation) — 选设备+预约     │
│  □ 团体预约(/group-booking) — 信息填写+提交        │
└──────────────────────────────────────────────────┘
┌─ frontend B: admin-web ───────────────────────────┐
│  □ 仪表板($/) — 数据加载+图表渲染                 │
│  □ 门店管理(/stores) — 列表+详情+编辑              │
```

**检查清单（每个页面）**：

| 检查项 | 说明 | 发现问题的处理 |
|:-------|:-----|:-------------|
| 🏃 加载速度 | 首屏<1s, 交互响应<200ms | ⚠️ 退回→性能优化 |
| 🖱️ 交互流畅 | 点击/滑动/输入无卡顿 | ⚠️ 退回→交互修复 |
| 📱 响应式 | 桌面+平板+手机正常渲染 | ⚠️ 退回→响应式修复 |
| 🔄 异常处理 | 网络断开/空数据/错误提示 | ⚠️ 退回→异常处理 |
| 🎨 视觉一致 | 字体/颜色/间距符规范 | ⚠️ 退回→视觉调整 |
| 🌐 多语言 | 中英文切换完整 | ⚠️ 退回→国际化补全 |
| ♿ 无障碍 | 键盘导航+屏幕阅读器 | 📝 记入下期改进 |

### Step 5: 审校裁决（5min）

> 审校主席(E1) + 客户代表(E40) + 监管代表(E38) 联合裁决

```
                    ┌─────────────┐
                    │  AI初评报告  │
                    │  +真人抽查   │
                    │  +页面实操   │
                    └──────┬──────┘
                           │
                    ┌──────▼──────┐
              ┌─────┤ 审校委员会  ├─────┐
              │     └──────┬──────┘     │
        ┌─────▼─────┐  ┌──▼──┐  ┌──────▼──────┐
        │  全部通过  │  │有条件│  │   退回      │
        │  直接签署  │  │ 通过 │  │ 确定退回项  │
        └─────┬─────┘  └──┬──┘  └──────┬──────┘
              │           │             │
              ▼           ▼             ▼
        ✅ 归档    ⏸️ 标记修复期    ❌ 退回复审
                    (24h内修完)    (修复后重新提审)
```

**裁决结果类型**：

| 结果 | 条件 | 操作 |
|:-----|:-----|:-----|
| ✅ 直接通过 | 无退回事由·无安全漏洞 | 主席签署→归档 |
| ⏸️ 有条件通过 | 有轻微问题(≤3个)·非安全级别 | 标记问题+分派→48h修复验证 |
| ❌ 退回 | 有≥1个P0/P1问题·或≥4个轻微问题 | 退回+定位+原因+要求→修复后重审 |

```

  │    指定修复人     │          │           │
  │    设定DDL(24h)   │          │           │
  └──────────────────────────────────────────────┘
```

### 退回率阈值

| 周次 | 退回率上限 | 目标 |
|:-----|:---------:|:-----|
| 第1-2周(试运行) | ≤30% | 团队适应审校流程 |
| 第3-4周(成长期) | ≤20% | 质量意识提升 |

## 附录：AI模拟专家审查细则

### AI-E1 架构陈老 — 代码逻辑检查清单

```
□ 模块间依赖是否存在循环引用
□ 函数/方法是否职责单一 (单一职责原则)
□ 是否有未使用的变量/函数/import (dead code)
□ 异步链是否使用async/await (避免回调地狱)
□ 异常处理是否完整 (try-catch或Result类型)
□ 配置是否分离 (硬编码值→环境变量/配置中心)
□ 数据库查询是否有N+1问题
□ 缓存策略是否合理 (Cache-aside/Refresh-ahead)
□ 接口设计是否符合RESTful规范
□ 错误码/异常是否统一处理
```

### AI-E2 安全李工 — 安全检查清单

```
□ 所有用户输入是否经过验证/消毒
□ HTML渲染是否使用安全方式 (dangerouslySetInnerHTML→DOMPurify)
□ API端点是否经过鉴权/授权 (auth guard + store ownership)
□ SQL/NoSQL查询是否使用参数化 (避免注入)
□ 敏感数据(密码/密钥)是否加密存储
□ JWT/令牌是否设置过期时间
□ CORS配置是否限制来源
□ 文件上传是否限制类型/大小
□ 日志是否泄露敏感信息 (密码/令牌)
□ TypeScript类型是否严格 (避免as any逃逸)
```

### AI-E3 性能张工 — 性能检查清单

```
□ 是否存在不必要的重复渲染 (React.memo/useMemo)
□ 大数据列表是否使用虚拟化 (virtual scroll)
□ API请求是否合并 (避免 waterfall)
□ 图片是否懒加载
□ 是否存在内存泄漏 (setInterval未清理)
□ 首屏JS是否过大 (code splitting)
□ 缓存策略 (HTTP缓存/Service Worker)
□ 数据库查询是否有必要索引
□ 是否存在同步阻塞操作 (主线程)
```

### AI-E4 UX王工 — UX检查清单

```
□ 页面加载是否有骨架屏/loading
□ 空数据是否有"暂无数据"提示
□ 错误是否有用户友好提示 (非技术栈信息)
□ 表单校验是否有实时反馈
□ 按钮/链接是否有悬停/点击状态
□ 可点击元素是否足够大 (≥44px触摸目标)
□ 键盘导航是否完整 (Tab顺序合理)
□ 颜色对比度是否满足WCAG AA标准
□ 字体大小是否可读 (≥14px正文)
□ 国际化是否完整 (所有文本走i18n)
```

### AI-E5 测试赵工 — 测试检查清单

```
□ 新增功能是否有对应的测试
□ 测试是否包含正例+反例+边界
□ 测试是否使用mock而非真实依赖
□ 测试命名是否清晰 (should...when...)
□ 是否存在flaky test (依赖时序/未清理)
□ 测试是否存在冗余 (重复断言)
□ 覆盖率是否合理 (不追求100%, 追求关键路径)
□ E2E测试是否覆盖核心用户流程
□ 角色测试是否覆盖8种角色视角
```

### AI-E6 DevOps刘工 — DevOps检查清单

```
□ 构建配置是否正确 (tsconfig/package.json)
□ 依赖版本是否锁定 (lockfile)
□ 环境变量是否分离 (dev/staging/prod)
□ CI流程是否包含构建+测试+lint
□ Dockerfile是否优化 (多阶段构建)
□ 部署脚本是否幂等
□ 日志级别是否合理 (debug不输出到生产)
□ 监控告警是否有对应指标
```
