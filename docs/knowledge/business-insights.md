## 1. 门店SaaS多租户架构最佳实践

### 1.1 架构选型对比

| 模式 | 隔离粒度 | 适用场景 | shenjiying88 选择 |
|------|---------|---------|:----------------:|
| 数据库独立 (Database-per-Tenant) | ⭐⭐⭐⭐⭐ | 大型客户/合规敏感 | ❌ 成本高 |
| Schema独立 (Schema-per-Tenant) | ⭐⭐⭐⭐ | 中型客户 | ❌ 迁移复杂 |
| 逻辑隔离 (Shared Table + tenant_id) | ⭐⭐⭐ | 海量小微客户 | ✅ 当前选择 |
| 混合模式 (Hybrid) | 灵活 | 大小客户混合 | ✅ P0演进方向 |

### 1.2 shenjiying88 当前架构（已验证）

**核心模型**: Shared Table + tenant_id 策略 (shenjiying88 已验证可行)

```
┌─────────────┐     ┌──────────────┐     ┌──────────────┐
│ API Gateway  │────→│ 多租户中间件   │────→│   共享数据库    │
│ (JWT解析)    │     │ (tenant_id    │     │ (tenant_id     │
│  tenant_id   │     │  自动注入)    │     │  索引覆盖)     │
└─────────────┘     └──────────────┘     └──────────────┘
```

**关键实现**:
- **租户上下文**: 所有请求经网关解析 JWT → 提取 tenant_id → 注入请求上下文
- **数据隔离**: 核心表以 tenant_id 为第一分区键,所有查询强制带 tenant_id 过滤
- **索引策略**: (tenant_id, created_at) 复合索引覆盖绝大多数查询

### 1.3 2026年行业趋势

| 趋势 | 说明 | 优先级 |
|------|------|:------:|
| **分库分表+租户路由** | 当单库超500租户或单表超2亿行,按租户id范围/哈希分片 | ⭐⭐⭐⭐⭐ |
| **租户级连接池** | 每个租户独立连接池上限,防止"烂租户"拖垮全站 | ⭐⭐⭐⭐ |
| **跨租户管理** | 集团客户统一管理多个门店(加盟品牌),需 tenant group 能力 | ⭐⭐⭐⭐⭐ |
| **智能降级** | 非核心租户自动降级到只读缓存,保障核心租户 | ⭐⭐⭐ |
| **AI辅助迁移** | 按租户活跃度/数据量 AI 自动建议隔离升级 | ⭐⭐⭐ |

### 1.4 关键约束

```
⚠️ tenant_id 必须：
  1. 所有 API 请求自动注入(不可靠客户端传入)
  2. 所有 SQL 查询显式带上 WHERE tenant_id = ?
```

### 3.2 盲盒合规设计Checklist

```
✅ 概率公开：每个SKU的获得概率明确展示(精确到小数点后2位)
✅ 保底机制：N抽必出(例:30抽必出SSR)
✅ 公示位置：购买页主流程可见,不得藏于二级页面
✅ 未成年人保护：单日消费限额(建议¥200上限)·家长监控
✅ 防沉迷提示：弹窗提醒·连续抽取间隔提示
✅ 退款机制：未拆封盲盒支持7天无理由退货(仅限线上购买)
✅ 记录留存：购买记录+概率抽取记录保留≥3年(合规审计)
✅ 概率变更通知：调整概率需提前30天公示
```

### 4.2 核心设计模式

| 模式 | 说明 | shenjiying88 场景 |
|------|------|:----------------:|
| **CQRS** | 读写分离,查询走物化视图 | 📊 报表/库存看板 |
| **Event Sourcing** | 库存变更事件溯源(入库/出库/盘点/调拨) | 📦 库存历史追溯 |
