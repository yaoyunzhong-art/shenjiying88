## 一、四大核心原则

### 1.1 便捷原则

| 约束项 | 规格值 |
|--------|--------|
| 核心操作步数 | ≤ 3 步（预约、下单、兑换、分享等） |
| 关键信息展示 | 一屏展示，无需滚动即见核心内容 |

**落地约束：**
1. 所有 C 端页面（预约页、商品详情、权益中心、活动页面）必须通过 UI 设计评审确保主路径 ≤ 3 步。
2. 首页 / 核心页面在 375 px 宽度下一屏可视区内展示：核心操作入口、关键数据（余额 / 积分 / 等级）、当前权益。
3. 每一步操作必须有明确的进度指示器（Step Indicator / Breadcrumb）。
4. 懒加载 / 骨架屏仅在非核心区域使用，关键信息区禁止占位。

### 1.2 科学原则

| 约束项 | 规格值 |
|--------|--------|
| 营销触达 AI 依据 | 100% 营销推送必有 AI 决策依据记录 |
| 推荐匹配度 | > 80% |

**落地约束：**
1. 每次营销推送前必须经过 AI 决策引擎，决策记录写入 `marketing_push_decision_log`。
2. 推荐匹配度算法：客户画像特征与推送内容标签的余弦相似度 ≥ 0.80。
3. 匹配度低于 80% 的推送被引擎自动拦截，并记录拦截原因。
4. 所有 AI 决策必须可追溯（决策 ID、时间、画像快照、匹配分数、算法版本）。

### 1.3 少打扰原则

| 约束项 | 规格值 |
|--------|--------|
| 每日营销推送上限 | 每客户 ≤ 1 条（P2 + P3 合计） |
| 免打扰时段 | 23:00 - 08:00 静默 P2 / P3 |
| 一键关闭营销 | 客户可一键关闭所有 P3 级推送 |

**落地约束：**
1. 频率计数器基于 `(tenant_id, member_id, date)` 聚合，Redis 原子递增。
2. 免打扰时段在推送调度层做硬拦截，P2 / P3 推送队列在 23:00 - 08:00 不发送。
3. 一键关闭：在客户偏好设置页提供"关闭全部营销推送"开关，写入 `member_push_preference.marketing_optout = true`。
4. 营销推送发送前，中间件检查 optout 标记——已关闭的直接跳过。

### 1.4 AI 精准营销原则

| 约束项 | 规格值 |
|--------|--------|
| 营销转化率提升 | ≥ 30%（相对基准） |
| 客户投诉率下降 | ≥ 50%（相对基准） |

**落地约束：**
1. 建立转化率基线（基于推送后 7 天内转化事件），AI 持续优化。
2. 投诉 / 退订作为负反馈进入模型训练。
3. 月度看板自动对比基线 vs AI 决策推送的转化率、投诉率差异。
4. 连续 2 个月未达标的自动触发告警 + 模型回滚机制。

## 二、推送分级体系（P0 - P3 四级）

### 2.1 分级定义与渠道策略

| 级别 | 定义 | 渠道策略 | 客户可否关闭 |
|------|------|----------|-------------|
| **P0 交易类** | 消费确认、退款、合同变更、法律通知 | 邮件（必发）+ App 推送 / 微信服务通知 + 短信（备选） | ❌ 不可关闭 |
| **P1 服务类** | 预约确认、权益变动、到期提醒 | App 推送 / 微信服务通知（主）→ 短信（辅）→ 邮件（保底） | ❌ 不可关闭 |
| **P2 权益类** | 月度账单、升级通知、积分变动 | App 推送 / 微信服务通知（主）→ 邮件（保底） | ✅ 可关闭 |
| **P3 营销类** | 活动促销、新品推荐、品牌传播 | App 推送 / 微信服务通知（主）→ 社媒/公众号（辅） | ✅ 可关闭 |

### 2.2 分级落地约束

1. 推送记录表 `push_record` 必须包含 `push_level` 字段（enum: P0 | P1 | P2 | P3）。
2. 渠道路由引擎根据 push_level + 渠道健康度 + 成本评分 + 客户偏好综合决策。
3. P0 / P1 不可被客户关闭；P2 / P3 可关闭。

### 2.3 各渠道推送矩阵

| 渠道 | P0 交易 | P1 服务 | P2 权益 | P3 营销 |
|------|---------|---------|---------|---------|
| App 推送 | ✅ 必发 | ✅ 必发 | ✅ 可选 | ✅ 可选 |
| 微信服务通知 | ✅ 必发 | ✅ 必发 | ✅ 可选 | ✅ 可选 |
| 短信 | ✅ 备选 | ✅ 降级 | ❌ 不发 | ❌ 不发 |
| 邮件 | ✅ 必发 | ✅ 必发 | ✅ 必发 | ✅ 可选 |
| 社媒 / SNS | ❌ 不发 | ❌ 不发 | ✅ 可选 | ✅ 可选 |
| 小程序订阅消息 | ✅ 可选 | ✅ 可选 | ✅ 可选 | ❌ 不发 |

## 三、频率与疲劳度管控

### 3.1 频率上限

| 维度 | 上限 |
|------|------|
| 单客户每日 P2 + P3 | ≤ 1 条 |
| 单客户每周 P3 | ≤ 3 条 |

**落地约束：**
1. Redis + Lua 脚本实现原子计数。
2. `daily_push_count:{tenant}:{member}:{date}` —— 每日计数器。
3. `weekly_push_count:{tenant}:{member}:{week_number}` —— 每周计数器（仅 P3）。
4. 超过上限的推送请求直接拒绝并记录到 `push_denied_log`。

### 3.2 免打扰时段

| 规则 | 说明 |
|------|------|
| 23:00 - 08:00 | 所有 P2 / P3 静默 |
| P0 / P1 | 不受免打扰限制 |
| 时段配置 | 可系统级调整（通过配置中心下发） |

### 3.3 会员差异化推送频率

| 会员等级 | P2 / P3 频率 | 推送侧重 |
|----------|--------------|----------|
| 青铜 - 白银 | 每日 ≤ 1 条 | 升级引导、新人福利、基础活动 |
| 黄金 - 铂金 | 每日 ≤ 1 条 | SVIP 引导、专属活动、高阶特权 |
| 钻石 - 王者 | 每 3 天 ≤ 1 条 | 专属活动、新品内测、生日关怀 |
| SVIP 会员 | 每周 ≤ 2 条 | SVIP 专属活动、新品优先购 |

### 3.4 营销疲劳度保护

| 疲劳度区间 | 措施 |
|------------|------|
| 0 - 70 | 正常推送，按等级频率执行 |
| 70 - 90 | 降为每 3 天 1 条（P2 + P3） |
| 90 - 100 | 暂停 P2 / P3 推送 7 天 |

**落地约束：**
1. 疲劳度评分：AI 根据近 7 天接收数、打开率、退订率计算（0 - 100 分）。
2. 疲劳度评分写入 `member_fatigue_score` 表，每小时 / 事件触发更新。
3. 过度营销抑制：同一客户 7 天内 P2 + P3 推送超 3 条，自动暂停 7 天。

### 3.5 自动效果追踪与自愈

| 规则 | 阈值 | 自动处置 |
|------|------|----------|
| 打开率过低 | < 10% | 该推送模板自动下架 |
| 退订率过高 | > 1% | 该活动自动暂停 |

**落地约束：**
1. 每次推送追踪：打开率、转化率、退订率。
2. 全量推送记录写入 `push_event_log`，支持多维分析看板。

| 通道 | 服务商 | 角色 |
|------|--------|------|
| 主通道 | 阿里云短信 | 日常发送 |
| 备通道 | 腾讯云短信 | 故障切换 |

**落地约束：**
1. 主通道发送失败率 > 5% 时自动切换备通道。
2. 健康度探针每 5 分钟检测双通道可用性。
3. 短信成本控制：每客户每月 ≤ 5 条（验证码除外）。

### 4.4 AI 渠道决策引擎

**输入维度：**
- push_level（P0 - P3）
- 渠道历史打开率
- 渠道成本
- 用户偏好
- 实时健康度

**V5.1 优化：**
1. 渠道健康度评分：根据历史送达率和打开率动态调整渠道权重。
2. 渠道性价比评分：综合成本与转化率进行决策。

**渠道组合策略：**
- 主通道 + 辅通道 + 保底通道
- 效果追踪：每次推送追踪送达率、打开率、转化率

### 4.5 客户全渠道偏好设置

| 渠道 | 可关闭范围 | 备注 |

| 邮件 | 营销类（P3）可关闭 | 交易/服务/权益类不可关闭 |
| 短信 | P0 / P1 可关闭 | 验证码除外 |

## 五、AI 精准营销引擎

### 5.1 客户画像持续学习（9 维）

| 编号 | 画像维度 | 说明 |
|------|----------|------|
| 1 | 消费偏好 | 品类、价位、频次 |
| 2 | 价格敏感度 | 对折扣/优惠的响应程度 |
| 3 | 活跃时段 | 到店/线上活跃时间分布 |
| 4 | 渠道偏好 | 各渠道打开/转化数据 |
| 5 | 营销疲劳度 | 近 7 天接收数、打开率、退订率 |
| 6 | 生命周期阶段 | 新客 / 活跃 / 沉睡 / 流失 |
| 7 | 会员等级 | 青铜 / 白银 / 黄金 / 铂金 / 钻石 / 王者 / SVIP |
| 8 | 推送偏好 | 偏好的推送类型、时段 |
| 9 | 社交偏好 | 团建 / 生日趴 / 分享行为 |

**落地约束：**
1. 实时更新：每次客户行为事件触发画像更新。
2. 所有营销决策基于最新画像快照（禁止使用过期画像）。
3. 画像数据存储：`member_profile` 表，JSONB 格式可扩展。

### 5.2 场景触发器（8 种）

| 编号 | 场景 | 触发条件 | 推送内容 |
|------|------|----------|----------|
| 1 | 到店触发 | 门闸入场 / 签到 | 到店有礼、新品推荐 |
| 2 | 消费后触发 | 交易完成 | 下次优惠、会员升级引导 |
| 3 | 排队中触发 | 排队取号 | 闲时优惠推荐 |
| 4 | 离店后触发 | 离店事件 | 评价邀约、复购唤醒 |
| 5 | 沉睡唤醒 | 休眠 > 30 天 | 沉睡召回 |
| 6 | 生日前 30 天 | 距生日 30 天 | 生日趴引导、一键邀请好友 |
| 7 | 团建后 7 天 | 团建完成 7 天 | 团建报告、下次推荐 |
| 8 | 联盟券推荐 | 异业联盟券可用 | 联盟券精准推送 |

### 5.3 营销内容 AI 生成

| 模块 | 说明 |
|------|------|
| 内容库 | 基于营销知识库（RAG）的模板库 |
| 个性化匹配 | 根据画像从内容库选取匹配度 > 80% 的内容 |
| 动态生成 | AI 生成个性化文案（标题 + 正文 + CTA） |
| A/B 测试 | 每次推送可选 A/B 变体，自动优选出转化率高的变体 |

### 5.4 协同过滤算法（V5.1 优化）

| 算法 | 说明 |
|------|------|
| User-based CF | 根据相似客户的偏好推荐内容 |
| Item-based CF | 根据相似内容的历史表现推荐 |
| 混合模型 | 融合 User CF + Item CF 权重 |
| 冷启动推荐 | 新客户使用基于标签的推荐 |

### 5.5 营销效果 AI 评估

**实时监控指标：**
- 打开率
- 转化率（推送后 7 天内）
- 退订率

**AI 每周输出《营销效果周报》：**
1. 各活动 ROI 排名
2. 各渠道效果对比
3. 客户疲劳度趋势
4. 优化建议
