## 二、每日流水线 — 以专家为轴心

### 核心节奏

```
                        🏛️专家驱动
                    ┌─────────────────────┐
  09:00 用户/策划   │ 专家说需要什么       │ 需求提出
                    └──────────┬──────────┘
                               ↓
                    ┌─────────────────────┐
  10:00 策划/需求   │ 专家说怎么设计       │ 设计确认
                    └──────────┬──────────┘
                               ↓
                    ┌─────────────────────┐
  11-17 开发        │ 树哥按专家要求写     │ 代码开发
                    └──────────┬──────────┘
                               ↓
                    ┌─────────────────────┐
  15:00 测试        │ 专家试能否用         │ 场景验证
                    └──────────┬──────────┘

                    ┌─────────────────────┐
  20:00 验收        │ 专家说好用了         │ ✅ 闭环
                    └─────────────────────┘
```

## 三、每模块6道专家门

### Gate1 🏗️ 架构安全 (E1+E44签署)
- [ ] 模块依赖无循环
- [ ] 接口符合RESTful
- [ ] 安全基线通过
- [ ] E44周技术签署

### Gate2 🏪 业务流程 (对口业务专家签署)
- [ ] 覆盖该模块的对口业务专家使用场景
- [ ] 核心操作≤30秒
- [ ] 异常处理完整
- [ ] 例如：收银类→E13李收银签署；营销类→E4张营销签署

### Gate3 📊 数据AI (E5赵数据或E9吴AI签署)
- [ ] 数据模型规范
- [ ] 查询有索引/缓存
- [ ] AI I/O契约完整

### Gate4 👤 用户体验 (E7孙体验或E40杨客户签署)
- [ ] 加载/空/错误状态覆盖
- [ ] 操作不超过3步
- [ ] 不会用错

### Gate5 💰 合规财务 (E36卫审计或E2李安全签署)
- [ ] 交易审计日志完整
- [ ] PII脱敏
- [ ] 退款/支付有状态机

### Gate6 🏛️ 审计治理 (E38沈监管或E6刘合规签署)
- [ ] 操作日志可追溯
- [ ] QA报告完整
- [ ] 知识库更新

## 七、专家卡片如何用

每位专家在 `experts/E*.md` 已有卡片。日常操作：

```
需求提出 → 龙虾哥读相关专家的需求章节 → 整理成任务卡
开发完成 → 龙虾哥通知对口专家测试 → 专家说"好用了" = 闭环
签署验收 → 龙虾哥在晚会上汇总6道门状态
```

第一轮操作（从今天开始）：
1. 读 E4张营销/E8周运营/E11钱店长 的 `experts/*.md` 需求章节
2. 提取P0需求作为明日开发任务
3. 专家晨会签发后执行
