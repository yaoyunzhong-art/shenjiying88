## 1. 架构 & 云原生 (E1+E3+E39+E52+E45)

### E1 陈架构 — 系统架构 · ⭐⭐⭐Approver

**核心建议**:
1. **P-50 必须打通侦察兵数据库**：运营参谋系统的数据源必须是真实的 349 场馆×9维数据，当前全 mock 设计导致可行性报告评分、设备建议、定价建议**全部不可信**。建议提供 IntelligenceDataProvider 接口层，支持 mock→DB 的无缝切换。
2. **P-49 Sitemap 生成 cron 应该放在后端而非前端**：当前 sitemap 生成脚本放到 `scripts/` 目录，实际生产应走 NestJS ScheduleModule + 写入对象存储/CDN，而不是写入前端的 public 目录。
3. **多模块的 tenant_id 穿透检查**：P-38 对账、P-49 SEO、P-50 运营参谋三个新模块的 tenant_id 字段缺失——多租户隔离是 SaaS 的命门。
4. **API 限流**：P-50 的 POST /feasibility 接口可能被竞对调用分析我们的算法，必须加 rate limit。

### E3 王硬件 — 硬件架构 · Observer

**核心建议**:
1. **运营参谋中的设备建议应该关联到具体的硬件型号**：当前只写了"建议 XX 台街机射击区"，没有具体品牌、型号、功耗、售后周期。店长无法据此做采购决策。
2. **竞品监控应覆盖硬件维度**：竞品 A 升级了 VR 设备、引进了新模拟器——这些硬件异动监控对我们的门店竞争力判断很重要。

### E39 韩开发 (ISV/开发者) · Observer

**核心建议**:
1. **运营参谋系统应该提供开放 API**：让加盟商/合作伙伴能通过 API 调用可行性分析，嵌入到他们的选址工具中。这是 SaaS 变现的入口点之一。
2. **Seo 模块的元数据和 sitemap 应该开放 API 让 ISV 调用**：多品牌多门店系统，每个加盟品牌可能需要独立的 sitemap 策略。

### E52 季原生 — 云原生 · Observer

**核心建议**:
1. **竞争监控的数据采集应该走边缘节点**：爬取美团/抖音/大众点评等外部数据源，应当使用边缘计算节点(门店 NUC 空闲算力)做分布式数据采集，上传区域中心汇总。
2. **Sitemap 和网站静态应该走 CDN 边缘缓存**：tob-web 的 SEO 页面如果走 K8s + CDN，首屏性能和 SEO 评分都会提升。

### E45 廖前端 — 前端架构 · Observer

**核心建议**:
1. **运营参谋的三个页面缺乏统一的设计模式**：/feasibility、/operations、/monitor 各自用了不同的交互模式（滑块/卡片/列表），应该统一到同一个组件体系下。
2. **SEO 仪表盘和健康报告的可视化需要 echarts 增强**：当前基于 CSS 条形的可视化过于朴素，缺少趋势线、雷达图、饼图。店长需要一眼看出优化进展。

## 2. 安全 & 攻防 (E2+E6+E38+E51)

### E2 李安全 — 安全攻防 · ⭐⭐⭐Approver

**核心建议**:
1. **运营参谋系统的竞品数据采集必须有风控**: 美团/抖音/大众点评的页面爬取，如果频率过高会被封 IP。必须使用代理池 + 随机 User-Agent + 行为模拟。
2. **P-50 feasible API 缺乏输入校验**: 如果有人传 `city=''` 会静默返回默认值而不是 404——这是安全审计中的「静默失败」反模式。
3. **P-49 sitemap 生成脚本可能成为 SSRF 攻击入口**: 如果 sitemap 脚本支持外部输入的 URL，则可能被用于 SSRF 攻击。
4. **所有新模块都需要过 OWASP Top 10 自动化扫描**。

### E6 刘合规 — 法务合规 · ⭐⭐⭐Approver

**核心建议**:
1. **运营参谋必须增加法律声明**："本报告仅供内部决策参考，不构成对外投资建议"。尤其涉及"预估月收入"和"回收期"的页面，可能被认为是诱导性财务预测。
2. **竞争监控的合规边界**: 从公开平台(美团/大众点评)采集价格和活动数据属于合法公开信息采集，但**不得通过爬虫突破反爬机制**（这可能构成计算机信息系统侵入）。
3. **可行性报告的免责提醒**: 报告顶部必须醒目显示"基于公开数据估算，实际业绩因经营能力、市场变化等因素而异"。

### E38 沈监管 — 行业监管 · ⭐⭐⭐Approver

**核心建议**:
1. **运营参谋中的"活动建议"必须考虑行业合规**: 盲盒促销、抽奖活动需要符合《反不正当竞争法》和《盲盒经营行为规范指引》。活动模板中不能包含违规的"100%中奖"描述。
2. **P-49 SEO meta data 不能包含虚假宣传**: 元数据中的 title/description 不得含有"最佳""第一""最大"等绝对化用语。
3. **会员定价建议要考虑合规**: 分时段定价策略不能构成对特定人群的歧视（例如对青少年的限时段加价）。

### E51 章安全 — 渗透测试 · Observer

**核心建议**:
1. **为运营参谋系统做一次渗透测试**: 这个系统涉及竞品数据采集和内部商业分析，是**高价值目标**。需要做一次全面的白盒测试。
2. **API 认证和授权检查**: P-49/P-50 的所有新接口，必须加上 `@UseGuards` 和角色检查，防止未授权访问内部商业分析数据。

## 3. 业务 & 营销运营

### E4 张营销 — 营销策略 · ⭐⭐⭐Approver

**核心建议**:
1. ⭐⭐⭐ **活动参谋必须覆盖 6 大活动类型**: 当前只有抖音团购/主题赛/会员日 3 种。必须覆盖：盲盒促销、联名活动、周末快闪、亲子工作坊、夜间专场、暑假/寒假限定活动。（优先级 P1）
2. **运营参谋应该支持"历史案例库"**: 同一个城市做过哪些活动、效果如何——形成一个活动效果数据库，让客户通过"效果最好"排序而不是被动看推荐。
3. **定价模块应该包含"竞品价格跟踪"**: 不只是给出定价建议，而是实时显示 3 家最近竞品的价格变化曲线。给客户看的不只是选项，还有市场行情。
4. **活动方案建议应该关联同城节假日**：根据城市和日期自动匹配当地节庆、展会、学校假期等。

### E8 周运营 — 实体商业运营 · ⭐⭐⭐Approver

**核心建议**:
1. ⭐⭐⭐ **竞争监控必须接入真实数据管道（M5）**: 当前所有告警是 mock 数据。必须建立：美团团购价格采集 → 抖音活动爬取 → 大众点评评价抓取 自动流水线。建议与侦察兵周日 04:00 采集联动，同时增加每日增量扫描。
2. **运营参谋应该包含"季节性建议"**: 7-8 月暑假策略（学生客群）、9-10 月开学季策略（亲子回归）、11-12 月冬季策略（室内活动高峰）、1-2 月春节策略（返乡客群）。
3. **可行性报告应该加入"同店对比"**: 显示同城同档次门店的经营数据和 KPI 对比，让投资决策者有行业维度。
4. **监控告警的时效性至关重要**: 竞品价格调整监测应该做到 2 小时内检测到，而不是每周一次。

### E11 钱店长 — 门店店长 · ⭐⭐⭐Approver

**核心建议**:
1. ⭐⭐⭐ **可行性报告必须增加装修成本和周期（M4）**：当前只评估设备成本，但装修通常占总预算的 30-40%。需要按场地面积(300/500/800/1000㎡)、装修档次(经济/标准/精装/豪华)给出装修估算。
2. **设备建议应该关联具体供应商和型号**: 比如"推荐 XX 品牌的 8 台射击街机，单价 ¥40,000，来自深圳的设备商 YY，保修 3 年，月维护成本约 ¥200/台"。店长要的是可直接采购的建议。
3. **竞争监控应该增加"设备异动"维度**: 竞品新进了什么设备？设备换新频率？这个对我们采购设备更新决策帮助很大。
4. **可行性报告的评分应该显示"置信区间"**: 不仅仅是 72/100 这样的具体数字，应该显示"根据 5 家同城竞品分析，可行性评分 65-80 分"。

### E12 孙导玩员 · Observer

**核心建议**:
1. **运营参谋应该包含"人力配置建议"**: 根据店面面积和预估客流，建议需要多少导玩员、收银员、清洁工。这是开店决策的重要部分。

### E13 李收银 · Observer

**核心建议**:
1. **运营参谋中的收银系统建议**: 不同规模门店应该推荐不同的收银硬件方案(小型用 iPad POS，大型用双屏收银机)。

### E14 赵场地 — 场地运维 · Observer

**核心建议**:
1. **可行性报告的场地选址应该更精细**: 除了城市和区域，应该支持"商场名称"输入（如 XX万达/XX大融城），系统对比该商场已有的娱乐业态。

### E15 吴内容 · Observer

**核心建议**:
1. **SEO 模块应该和内容营销打通**: 生成最热门店内容的 CTAs，驱动 tob-web 品牌站的内容 SEO 排名。

### E16 郑社群 · Observer

**核心建议**:
1. **运营参谋应该考虑社群运营维度**: 给出"开业前 30 天社群预热方案"建议。

### E17 周活动 · Observer

**核心建议**:
1. **活动参谋的建议应该更懂"节日"**: 根据城市 + 日期自动匹配当地节庆假日。

### E24 马招商 — 渠道招商 · Observer

**核心建议**:
1. **运营参谋应该成为招商工具**: 给潜在加盟商演示时，输入选址城市一键生成可行性报告——这是**强大的销售工具**。

### E37 江媒体 · Observer

**核心建议**:
1. **SEO 模块应该和舆情监测联动**: 当某城市有关于电玩城的负面新闻时，自动触发 SEO 内容调整（发布正面内容对冲）。

## 4. 数据 & AI (E5+E9+E48)

### E5 赵数据 — 数据科学 · 👑👑👑 Champion • 单票否决权

> **注意**: E5 拥有单票否决权，以下建议优先级 P0。

**核心建议**:
1. ⭐⭐⭐ **P-50 V2 必须接入侦察兵数据库（M1/M3）**: 可行性评分必须基于真实竞品数据。算法：选择城市 → 查询该城市所有竞品场馆的 9 维数据 → 统计营收/客流/价格分布 → 加权计算可行性得分。这不是选择题，是底线。
2. ⭐⭐⭐ **预估月收入和回收期必须有数据支撑（M3）**: 不能靠 `65 - count×3 + budget×0.04` 这种硬编码公式。需要：同城同类型竞品的真实抖音团购销量 × 客单价 = 营收代理变量；同面积场馆的租金/人工/水电 = 成本代理变量。
3. ⭐⭐ **建立"场馆绩效对比系统"**: 不只是可行性报告，经营中的门店应该能随时看到：本店 vs 同城竞品在 9 个维度上的对比。包括：客单价对比、设备丰富度对比、会员转化率对比、好评率对比。这是一个**持续运营工具**。
4. ⭐ **使用历史累积极大地提升可行性评分准确率**: 每开一家新店后回填实际经营数据，用 ML 模型迭代评分算法。

### E9 吴AI — AI算法 · ⭐⭐⭐Approver

**核心建议**:
1. ⭐⭐⭐ **运营参谋的 AI 建议必须基于真实推理（M2）**: 当前选择题的"AI建议"字段是预置文案，没有 AI 参与。需要完成：客户选择城市 → 查询同城竞品数据分析 → LLM 根据数据分析结果生成个性化建议 → 显示在客户端。理想做法：接入 empower_card 知识库做 RAG。
2. ⭐⭐⭐ **建立竞争对手异动自动分析**: 当检测到竞品价格调整时，自动调用 AI 分析：此价格调整可能影响本店多少客流？最优应对策略是什么？分析结果 + 3 种应对方案推送给店长。
3. ⭐⭐ **运营参谋加入'AI对话模式'**: 除了选择题，给高阶管理员一个对话界面——"我在上海徐汇开 500 ㎡的店，预算 500 万，针对白领客群，给出最优方案"。AI 综合分析后给出完整答复。
4. ⭐ **活动效果预测模型**: 基于历史数据训练，输入活动类型 + 城市 + 季节 → 预测客流增量/营收增量。

### E48 霍数据 — 大数据平台 · Observer

**核心建议**:
1. **侦察兵数据不能只存在于 md 文件**: 349 场馆的 9 维数据应该建立统一的数据仓库（PostgreSQL 中建表），供 P-50 运营参谋系统直接查询。
2. **竞争监控需要建立数据湖**: 美团/抖音/大众点评的采集数据需要建立时序存储 —— 既能看当前值，也能看 90 天趋势。
3. **SEO 数据应该与经营数据关联**: 哪些页面/渠道带来了最多客户？建立从 SEO 曝光→点击→到店→消费的完整数据漏斗。

## 5. 财务 & 审计 (E10+E29+E36+E54)

### E10 郑财务 — 财务风控 · ⭐⭐⭐Approver

**核心建议**:
1. ⭐⭐⭐ **可行性报告必须建立"财务全景表"（M4）**:
   - **首期投入**: 设备 + 装修 + 系统软件 + 押金/保证金
   - **月固定成本**: 租金(营收 15-25%) + 人力(6-8 人) + 设备维护(设备 10%/月) + 系统订阅
   - **月变动成本**: 电费 + 耗材 + 营销推广
   - **利润预估**: 基于同城同类型竞品营收中位数 - 成本合计
   - **回收期**: 首期投入 / 月均利润 + 装修摊销(5 年) + 设备折旧(3 年直线)

2. ⭐ **运营参谋的定价策略应该内置盈亏平衡点计算**: 当客户选择定价方案时，自动显示该价格下需要多少客流量才能盈利。例如："选择 ¥88/人定价，需要日均 150 人以上才能达到盈亏平衡。根据同城数据，该区域日均客流 120-180 人，建议选择分时段定价方案提高高峰利用率。"

3. ⭐ **B2 发票管理需要批量开票和自动对账**: 当前 B2 发票管理已经实现基础 CRUD(12 tests)，但缺少批量开票、与支付流水自动对账、发票状态推送功能。

### E29 李股东 — 投资 · Observer

**核心建议**:
1. **运营参谋应该增加"投资回报分析"**: 不只是给店长看，还要给投资人/加盟商看——包含投资回报率(ROI)、内部收益率(IRR)、净现值(NPV)等财务指标。

### E36 卫审计 — 审计链 · ⭐⭐⭐Approver

**核心建议**:
1. **所有 AI 决策必须审计**: P-50 生成的可行性报告、参谋建议都需要记录 AI 决策日志——什么数据来源、什么算法、什么版本、什么权重。
2. **竞争监控的数据来源必须可追溯**: 每条告警的数据来源、采集时间、采集方法必须有审计日志。

### E54 胥审计 — 区块链审计 · Observer

**核心建议**:
1. **建议 P-50 的可行性报告生成过程使用哈希链记录**: 每个版本的报告生成后，记录其数据快照的哈希值。这可以防止事后篡改，同时也是合规审计的依据。

## 6. 体验 & 品牌

### E7 孙体验 — 用户体验 · ⭐⭐⭐Approver

**核心建议**:
1. **运营参谋中所有页面需要骨架屏加载**: 当前 feasibility 使用 spinner，operations 和 monitor 无加载态。数据完全 mock 所以感觉不到延迟，真实接入 DB 后会有 1-3 秒加载时间。
2. **可行性报告的交互流程需要优化**: 当前是"输入→点击→显示"三级。应该改成"选择城市→自动显示区域'下拉带竞品密度数据'→选择区域后自动显示预估评分预览→点击生成完整报告"——**渐进式展示**。
3. **移动端适配**: 店长最常使用设备是手机。运营参谋系统必须支持移动端——当前三个页面都是桌面优先，手机上看字太小了。建议 nextjs 直接响应式适配。
4. **操作日志可视化**: P-38 对账差异详情页的操作日志使用时间线垂直 UI 效果很好(E7之前的建议已经被采纳)，继续保持这种风格。

### E21 张设计 — 视觉设计 · Observer

**核心建议**:
1. **运营参谋的视觉应该统一到同一设计系统**: 三个页面的颜色、间距、字体大小不一致。建议和品牌设计规范对齐（主色系 #1E88E5）。
2. **数据可视化需要真正的图表**: 可行性报告的竞品密度、收入预估用 CSS 条形图不够专业。建议使用 recharts/echarts 绘制：饼图(成本结构)、柱状图(营收预测)、折线图(回收期预测)。
3. **SEO 健康报告页建议变化趋势增加 sparkline**: 在每个数值旁边加一个小型趋势线，显示"相比上周"的变化。

### E40 杨客户 — 客户成功 · ⭐⭐⭐Approver

**核心建议**:
1. **运营参谋应该区分"新客户"和"老客户"视图**: 
   - **新客户(尚在选址)**: 显示可行性报告 + 城市数据概览
   - **老客户(已开业)**: 显示竞争监控 + 运营参谋 + 本店经营对比
2. **增加"设为已读"按钮**: 竞争监控的告警应该允许标记为已处理，避免重复出现引起焦虑。
3. **SEO 模块提升建议应该按优先级排序**: 健康报告的问题列表应该按"修复后影响大小"排序，让客户先做产出最大的改动。

### E30 周品牌(高端) · Observer

**核心建议**:
1. **高端门店(SVIP级)的定价策略应该与普通门店不同**: 定价参谋要考虑品牌定位——高端店不能跟风降价,低端店可以打价格战。运营参谋应该根据店铺档次智能调节建议策略。

### E31 吴品牌(流量) · Observer

**核心建议**:
1. **活动参谋建议应该关联短视频平台趋势**: 当前抖音热门话题、小红书爆款内容——让门店活动蹭上热门。

### E32 郑品牌(亲子) · Observer

**核心建议**:
1. **亲子向门店的运营参谋应该强调家庭客群**: 周末的亲子活动、平安夜的亲子派对、寒暑假的托管营。亲子客群是电玩城最大的收入来源。

### E33 王品牌联名 · Observer

**核心建议**:
1. **增加"联名活动"选项到活动参谋中**: IP 联名、品牌跨界合作是低投入高曝光的营销方式。

## 8. 产品 & 质量 (E50+E47)

### E50 柯产品 — 产品策略 · Observer

**核心建议**:
1. **P-50 需求范围需要明确**: 当前大飞哥的需求横跨三个子模块(可行性/参谋/监控)，但核心用户画像不清晰。建议先确定"首位用户"——是店长？区域经理？加盟投资人？不同用户看到的信息和交互完全不同。
2. **产品路线图**: 建议 P-50 分三阶段推进：
   - V1 (已交付): 骨架 + 选择题模式验证
   - V2 (待开发): 接入侦察兵数据 + 真实 AI + 财务全景
   - V3 (规划中): 全自动运营分析报告 + 多门店对比 + 移动端
3. **P-49 SEO 模块的产品价值有待验证**: 当前做了 API + 管理端 + 脚本 + sitemap cron + tob-web 5页，但缺少一个关键指标——"SEO 自然流量到店转化率"。建议在 V2 加入归因分析。

### E47 谷测试 — 测试架构 · Observer

**核心建议**:
1. **P-50 前端测试严重不足**: 3 个页面只有仪表盘有 2 个测试，增强 22 tests 后达到 24，但缺少 E2E 测试和边界值测试。
2. **P-49 SEO 的 API 测试覆盖度不够**: 39 个 api tests 覆盖了基本 CRUD，但缺少：并发请求测试、大文本(optimization suggestion 超过 1000 字符)测试、城市不存在时性能退化测试。
3. **竞争监控应该加入场景测试**: 模拟竞品按顺序发布价格调整→活动上线→评分变化，测试整个告警流水线。
4. **建议建立 AI 驱动测试**: 运营参谋的 AI 建议部分，人工测试不可靠（每次生成不同），建议用 snapshot 测试 + 关键词命中率测试。

## 9. 基础设施 (E46+E49+E53)

### E46 施运维 — SRE · Observer

**核心建议**:
1. **Sitemap cron 必须有健康检查和失败告警**: 如果 sitemap 生成失败两次，必须立即通知运维，否则搜索引擎会标记网站 sitemap 异常。
2. **竞争监控的数据采集需要资源隔离**: 爬虫任务不能占用 API 服务器资源。建议用独立 job pod 运行。
3. **监控 @m5/api 的可用性**: 前 20 天的 hang/404 问题未解决——所有新模块(包括 P-49/P-50)在高并发下可能触发同样的 bug。

### E49 贺部署 — DevOps · Observer

**核心建议**:
1. **所有新模块必须纳入 CI/CD 管道**: P-49/P-50 新增的模块和测试需要加入 turbo 缓存策略。
2. **22 天倒计时部署计划**: P-49 sitemap 需要 CDN 预热、P-50 需要 load test 确保 API 能扛住首批 10 家门店的并发查询。

### E53 简多语 — 全球化 · Observer

**核心建议**:
1. **运营参谋应该支持 i18n**: 不同语言的店长/投资人看到本地化的报告（英文/日文/韩文）。
2. **SEO 元数据需要多语言版本**: 国际化规划下，tob-web 需要支持多语言 sitemap 和多语言 SEO metadata。

### E42 李总经理 · Observer

**核心建议**:

### E43 张区域总监 · Observer

**核心建议**:

3. **P-50 API 接口设计需要补充**: 缺少分页、排序、搜索——三个端点目前都是全量返回。如果接入数据库后数据量大了，必须加分页。
4. **知识体系 V2 已入库 233 条很好**: 但 empower_card API 尚未集成到 CI/CD——树哥派单时不能每次都手工调用。
