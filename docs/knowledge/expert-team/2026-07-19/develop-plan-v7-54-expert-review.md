## 评审方法

### 量化模型

每组按 4 个维度评分，总分 100:

- `覆盖度` 30 分: 是否覆盖当前真实阻塞与核心收益
- `可执行性` 25 分: 是否能直接派工、按周落地
- `风险控制` 25 分: 是否覆盖生产、安全、回滚、质量风险
- `闭环性` 20 分: 是否有验收标准、退出条件、证据链

### 评级阈值

- `85-100`: 🟢 通过
- `70-84`: 🟡 有条件通过
- `<70`: 🔴 退回

## 关键发现

1. **计划方向正确，但少一条“Pad/POS 一线经营链”**
   - 当前计划更偏平台、后台、发布和治理
   - 一线收银/开单/核账/经营操作没有被提升为独立主线

2. **P-38 已写“补税务”，但不够显式**
   - 税务/发票不应只藏在“尾项”描述里
   - 需要单列交付物、验收页和 API 范围

3. **真实体验验收还不够硬**
   - 当前计划有 Playwright 增强方向
   - 但没有明确要求 `checkout 金额链 L3` 和 `VRT` 原型必须落地

4. **外部阻塞需要管理机制**
   - DNS / TLS / 域名 是战略 P0
   - 但目前计划还缺 `责任人 / 证据物 / 超时升级 / 日报可视化`

5. **工程收口方向正确，但还需要更强的“主信号”**
   - 覆盖率门禁、假阳治理、CI/GitLab 统一方向对
   - 但建议指定先试点哪些模块，避免阶段目标太宽
