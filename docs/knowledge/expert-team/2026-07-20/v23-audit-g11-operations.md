🟡 **有条件通过**

## 评审意见

### 1️⃣ 知识日采 cron 已设计并列入时间预算，但执行可靠性与回灌机制待验证

V23 每日时间预算表在 **11:00-12:00** 安排「📚知识日采」，并在时间预算说明中强调"不可跳过"。知识赋能体系的三层架构（API 优先 → DB 直连 → Manual 降级）在 `manual-knowledge-injection.md` 中有完整文档。

**审计发现的关键问题**：

**① 日采 cron 的"执行率"没有可观测手段**
- 当前时间预算表只是"规划"，没有实际的 cron 任务状态监控
- 没有 `LastExecutionTime` / `ExecutionDuration` / `CardCountInjected` 等指标采集
- V20 经验教训中明确"知识日采有时停跑"——V23 的"不可跳过"原则需要从**制度约束**落地为**代码可观测**：至少加一个 `knowledge-daily-ingestion-status` 的 health check 端点

**② 知识卡片被"消费"的使用率指标为零**
- 知识赋能体系有 100 张 Expert Insights 卡片（KB-001→KB-100）+ `empower_card` 表（SQL seed）
- `POST /api/empower-cards/match` API 可以匹配任务相关的知识卡片
- **但没有任何用量指标**：每周被匹配了多少次？哪些卡片从未被匹配？AI 开发过程中卡片的引用率？
- V20 圈梁升格为 P1 的"知识赋能"箍，如果没有使用率指标，P1 就是空头评级

**③ 知识日采增量缺乏种子倍增计划的具体执行方案**
- V23 要求"知识种子数据倍增，确保 S1-S6 数据导入+周更新"
- 但当前种子数据是固定的 100 张 + `seed-empower-cards.sql` 的静态内容
- "周更新"目前没有自动化脚本：没有 `scripts/daily-knowledge-ingest.sh`，没有 cron job 读取 RSS/API/日报来源
- V23 时间预算表的 11:00-12:00 知识日采块，需要配套实现的知识源采集脚本，否则这个时间块就是空转

### 2️⃣ Knowledge API 稳定性有架构设计但无 SLA 承诺

知识赋能系统的 API 调用链：

```
开发过程 → dispatch-knowledge.ts → POST /api/empower-cards/match → empower_cards 表查询
                                                                  ↓ 失败
                                                              Manual 降级
```

**审计发现的可靠性缺陷**：
- **API 没有速率限制**：如果有大量树哥并发调用知识匹配 API，没有任何限流/RateLimit 保护。dispatch-knowledge.ts 可能被 Redis 或高并发调用打满
- **API 没有缓存层**：知识卡片匹配查询命中相同的 tag/模块时，每次都走 DB 查询。100 张卡片时不是问题，1000 张时间长了就是问题
- **Manual 降级路径没有自动化**：当 API/DB 都不可用时，运营人员需要"手动操作"。但 V23 的自动化原则（V21 教训：机制配套）要求任何降级路径都应有自动化兜底——Manual 降级应当封装为一个 CLI 命令 `bash scripts/knowledge-manual-inject.sh`，而不是靠人力逐条写 SQL
- **无知识 API 的存活/健康状态监控**：当前 L3 22:00 评分系统不监控知识 API 的健康状态。如果知识 API 挂了，LL3 评分不会感知

### 3️⃣ 知识卡片全量审计机制缺失——圈梁审计箍只扫模块不扫知识

V23 的🟢审计箍要求`phase-to-module-mapping.md` 每周更新，这个扫的是**后端模块状态**，而不是**知识库健康度**。

但 V20 宪法级的**知识体系 V2** 定义了一套"赋能→引用→升级"的知识生命周期：

```
知识卡片注入 → 开发时被 match → 被引用次数增加 → 高引用卡片自动升级优先级
```

V23 缺失的是：
- **知识卡片低引用率告警**：至今未被 match 过的知识卡片（超过 14 天）应当自动标记为"低价值，需审核"
- **知识卡片陈旧度检查**：KB 卡片标注了 freshness_score 和 confidence，但没有定期衰减机制
- **知识库与模块的覆盖度对照**：KB-066（门店开业清单）→ 对接 P-30 后勤模块了吗？KB-027（设备维保）→ P-30 MaintenanceOrder 实现了吗？这种知识覆盖度扫描不存在

### 4️⃣ 日采 cron 在 V23 双轨制中的定位需澄清：属于轨道A还是轨道B？

时间预算表将 11:00-12:00 列为非轨道A也非轨道B的独立「📚知识日采」段。但双轨制其他时段明确分配了 A（基建 60%）和 B（功能 40%）。

**问题**：知识日采如果不属于轨道A也不属于轨道B，那么它的工时权重应如何计算？如果知识日采占用了本应属于轨道A或轨道B的时间，会不会导致时间预算超支？

**建议**：
- 将知识日采归属于**轨道B（持续交付）**的子任务——因为知识赋能服务于功能开发的效率提升
- 在时间预算表中明确 11:00-12:00 是轨道B的一个固定阶段段，不算额外时间

## 关注点

### 🔴 知识日采"执行率"不可观测——如果不先解决监控问题，V23 将重演 V20 "知识日采有时停跑"的教训

V20 的教训是"知识日采有时停跑，没人知道"。V23 的"不可跳过"原则很好，但如果**没有检测"是否跳过"的机制**，这个原则就是纸面上的。

**建议立即实施**的监控项：
- `scripts/check-knowledge-ingestion-status.sh` — 检测当日是否有知识日采记录
- 在 L3 22:00 评分系统中增加"知识日采执行率"子维度（权重 5/120）
- 日采 cron 的 stdout 应写入日志文件 `logs/knowledge-ingestion-$(date +%Y-%m-%d).log`

### 🟡 知识卡片使用率低（V20 教训）——需要引入"卡片冷启动"机制

V20 最深刻的教训之一：PRD 和知识赋能使用率低，因为开发者（AI）不知道或不习惯在开发过程中调用知识卡片匹配。

V23 将知识赋能升格为 P1 级，但**开发者侧没有变化**——仍然是"建议使用"而非"自动注入"。建议：
- 在每次树哥批量交付的 `agent-collaboration-rfc.md` 或 `README.md` 的模板中嵌入强制步骤：`* 步骤 3: 对每项开发任务，调用 dispatch-knowledge 获取相关卡片 *`
- 在 L2 树哥回写模板中增加一行：`* 本轮开发中引用了知识卡片：[KB-XXX/KB-YYY] *`

### 🟡 知识种子数据 100 张卡片对 50+ 模块的覆盖度不足

当前 Expert Insights 卡片数为 100，覆盖了 50+ 模块。平均 2 张卡片/模块。对业务深度模块（后勤、品牌、财务）的理论覆盖可能足够，但对横向能力模块（监控、部署、安全）的覆盖可能不够。

建议 G11 在 Phase 2 的目标中加入知识覆盖度指标：每个活跃模块至少有 2 张关联知识卡片。

## 建议

### 1. 建立知识赋能"三率"监控体系——注入率、匹配率、引用率

**建议内容**：

| 指标 | 定义 | 目标值 | 监控方式 |
|:-----|:-----|:------:|:---------|
| 注入率 | 当日知识日采是否执行 | 100% | `scripts/check-knowledge-ingestion-status.sh` |
| 匹配率 | 知识 API 每日被调用次数 | ≥20 次/天 | `GET /api/empower-cards/stats` 端点 |
| 引用率 | 开发输出中引用知识卡片的占比 | ≥30% | L2 回写日志解析 + `rg "KB-" docs/knowledge/evolution-log.md` |

**优先级**：P1（Phase 2 前实现监控，Phase 1 暂不阻塞关闸）

### 2. 将日采 cron 从"制度约束"落地为"代码+监控+告警"

**操作方案**：
- 在 `docs/knowledge/manual-knowledge-injection.md` 补充自动化降级脚本 `scripts/knowledge-auto-inject.sh`
- 脚本内容：读取 `scripts/seed-empower-cards.sql` 的最新变更 → 尝试通过 API 注入 → DB 直连 → SQL 文件输出（人工介入）
- 在 L3 评分脚本中增加知识日采执行率的子指标

**优先级**：P0（V23 机制配套原则——不应只有文档没有脚本）

### 3. 知识卡片使用率提升——从"被动匹配"到"主动推送"

**建议方案**：
- 在树哥每次批量交付前的标准协议中增加一步：`dispatch-knowledge` 主动推送当前任务相关卡片
- 将 `POST /api/empower-cards/match` 的返回结果自动写入树哥的临时内存，减少开发者的调用摩擦
- 对于高 confidence（≥80）的知识卡片，在树哥开发过程中自动引用到代码注释或 commit 消息中

**优先级**：P2（持续优化，非阻塞项）
