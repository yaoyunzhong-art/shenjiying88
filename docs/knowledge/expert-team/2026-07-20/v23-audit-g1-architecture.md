## 评审意见

### 1️⃣ 双轨制设计合理，但存在"基建侵蚀功能"的陷进条件

V23 双轨制（轨道A 60%基建 + 轨道B 40%功能）明确解决了 V22 "基建0%执行" 的核心矛盾。轨道A 锁定 P0 权重，配合每日时间预算表，理论上能够阻止基建再次被功能挤压。

**但存在三个结构性风险：**

- **风险1 —— 时间预算没有熔断阀**：08:00-10:00 的 CI/Docker 块如果遇到配置问题卡住，没有任何"超时机制"保证剩余时间切回轨道B。V21 教训（四层引擎执行率22%）表明，时间安排不等于时间执行。
- **风险2 —— 基建任务无 TSC 门禁**：圈梁🟢代码箍要求 TSC 零错误，但 Dockerfile/nginx/certbot 脚本根本不经过 TSC 检查。Phase 0 的基建交付实际上在圈梁之外——一个不受🟢代码箍约束的灰色地带。
- **风险3 —— "永续最终版"缺乏变更管理**：V23 宣告"不迭代版本号"，但 V20 圈梁5道箍→V22 变为6道→V23 变为7道，版本号不迭代不代表制度不演化。固化版本号可能掩盖未来必要的架构升级（如微服务拆分、事件溯源）。

### 2️⃣ 7道箍覆盖了架构质量的四个关键维度，但缺失"演进性"箍

| 箍 | 覆盖维度 | 评价 |
|:-:|:---------|:----:|
| 🟢代码 | 代码质量/类型安全 | ✅ TSC零错误是硬标准 |
| 🟢测试 | 测试健壮性 | ✅ 0 fail无skip |
| 🟢审计 | 架构透明度 | ✅ 圈梁表周更新 |
| 🟡PRD | 需求可追溯 | ⚠️ P1建议级，非必需 |
| 🟠知识赋能 | 知识管理 | ⚠️ P1建议级 |
| 🔴基建 | 部署就绪 | ✅ V23新增 P0 |
| 🧪E2E | 跨模块集成 | ✅ V23新增 P1 |

**缺失维度**：
- **可演进性箍**：没有箍检查"代码腐烂"——模块耦合度、循环依赖、LCOM（缺乏内聚）、包依赖图。50条E2E链覆盖的是业务层，不是架构层。建议补充「🔵演进箍」每周检查依赖健康。
- **性能箍**：7道箍中没有一道检查 API 响应时间、数据库查询延迟、页面加载速度。V18 G4退回的加载速度问题至今没有一个"必须通过"的门禁。

### 3️⃣ TSC 熔断机制设计方向对，但触发粒度太粗

V23 规定"TSC+测试+E2E三件套通过才算 Phase 关"——这是正确的熔断思路。

**但不一致的是**：圈梁执行的"24h硬线原则"说 Phase 截止 = 三件套通过，而 V23 时间预算表显示 Phase 1（7/24→26）仅有3天窗口。3天内完成 P-47 从100%→100%（维护）和 P-30 从100%→100%（维护），看似可行；但若基建任务（CI/Docker）阻塞到7/23仍未完成，Phase 1 开始前的实际可用窗口已缩小。

**建议的方案**：
- 将 TSC 熔断细化为**两个级别的门禁**：Level 1（Phase 内门槛：模块级 TSC + 测试）和 Level 2（Phase 关门：全量 TSC + 测试 + E2E + 基建）。Level 1 每天跑，Level 2 仅在 Phase 切换时跑。
- 基建的🔴基建箍应当与🟢代码箍联动：Dockerfile/nginx 等非 TS 文件应增加 `docker compose build` + shellcheck 作为替代门禁。

### 4️⃣ 模块边界定义清晰，但基建与功能的"共享资源冲突"未处理

双轨制假设轨道A和轨道B可以并行推进。但现实中：

- **Dockerfile 修改 ↔ CI 配置变更**：如果轨道A在改 Dockerfile，而轨道B的功能有新的 native 依赖（如 puppeteer 镜像），两者可能在 docker-compose.yml 上产生冲突。
- **nginx 路由重写 ↔ 新增 Controller**: 如果轨道B新增了一个 API 端点而轨道A改了 nginx 代理规则，可能造成 404 或反向代理绕行。
- **测试数据库 Schema 变更**: 基建可能修改数据库迁移脚本（如添加索引），而轨道B的功能可能刚好依赖这个 Schema。

**V23 的"无版本切换"策略反而放大了这个风险**——因为没有中间版本隔离，基建和功能的并发变更在同一个代码库中直接碰撞。

**建议**：
- 基建变更和功能变更使用 **feature branch** 或 **基建专用分支**，待 CI 验证通过后再合并到主线。
- 在 Phase 0 阶段的最后 1 天（7/23），锁死基建分支，仅允许功能分支合入，避免周日深夜同时 push 冲突。

## 关注点

### 🔴 R-001: docker-compose build 不通过的阻断效应

Phase 0 已验证了 nginx build 通过，但 Phase 2 要求 `docker-compose up` 全服务启动。如果 native 依赖（如 Prisma Client 生成、sharp 原生模块、canvas 二进制）在 Docker 环境中 build 失败，整个 🔴基建箍 就会挂红，进而阻止 Phase 1 关闸。

**V22 教训**：docker compose build 对 native deps 敏感，已在风险清单 R-001 标记。建议在 Phase 0 (7/20-23) 内先做一次完整的 `docker compose build --parallel` 测试，提前暴露 build 问题，不要等到 Phase 2。

### 🟡 R-003: 基建占比60% → 功能交付延期

V23 时间预算明确轨道A占60%，但 Phase 1 (P-47/P-30) 的任务是"维持100%"——实际上没有新增功能需求。然而 Phase 2 需要 E2E链 50→60（+10条）以及真实 API 接入测试，这些工作不属于基建。如果轨道A持续吃时间，E2E链扩展可能会被压缩。

**历史模式**：V22 一天 111 commits 但基建 0，是"功能压基建"。V23 的反向问题可能是"基建压功能"——E2E 链扩展不够导致🧪E2E 箍挂红。

### 🟡 圈梁表维护的持续投入

圈梁表从 V20 的 224 行升级到 V23 的 276 行（V2 版本），详细记录了 10 个 Phase、~48 个模块的审计状态。每周人工更新是🟢审计箍的要求。

**但问题是**：V21 教训显示 L3 自进化从未 cron 化（后来补了），类似的"周更新"也可能因为工期紧而被跳过。建议将圈梁表更新脚本化：自动化扫描模块状态，生成 diff，树哥确认后合入。

## 建议

### 1. 增加「🔵演进箍」—— 每周依赖健康检查

**内容**：
- 模块间循环依赖检测（`depcruise` 或 `madge`）
- Prisma model 与 Controller 的覆盖度对比（无 Controller 的 Model）
- 各模块的 LCOM 值趋势（类内聚度）
- E2E 链覆盖到的模块数 vs 未覆盖到的模块数

**权重**: P1（建议级，同 PRD 箍），但 E2E 链未覆盖模块>10个时升级为 P0

**自动化**: 用 `npx madge` 每周扫描 + markdown 报告写入 `docs/knowledge/architecture-health-weekly.md`

### 2. TSC 熔断机制扩展为"三明治熔断"

```
功能层:   TSC 零错误          ← 已有
集成层:   E2E 50/50 通过      ← 已有
部署层:   docker compose build + CI green  ← 新增联动
```

三件套分三层熔断，任何一层不通过 → 无法合入主线分支。基建变更在部署层熔断，功能变更在功能层熔断，互不干扰。

### 3. 基建变更专用分支 + 快速验收

**操作方案**：
- 创建 `infra/v23-docker` 和 `infra/v23-ci` 分支，所有 Docker/nginx/CI 变更在此分支
- 相关 PR 必须通过：`shellcheck` + `docker compose build` + `CI dry-run`
- 基建分支每 2 天合并一次到 `main`（时间预算表可以固定 18:00-19:00 为合并窗口）
- 合入前验证基础功能不破坏（跑最小 E2E 集：cashier/POS/checkout）

**优势**：避免基建和功能在同一分支上产生冲突，减少"基建挂红阻止功能合入"的连锁反应。
