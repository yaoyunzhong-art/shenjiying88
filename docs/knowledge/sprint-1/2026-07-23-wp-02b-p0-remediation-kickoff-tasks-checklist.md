# Sprint-1 · WP-02B P0 安全整改执行包

> 工作包: `WP-02B-S1`
> 对应 BS: `BS-0050`, `BS-0057`, `BS-0059`, `BS-0064`
> 对应偏离单: `DEV-0001`, `DEV-0002`, `DEV-0003`, `DEV-0004`
> 优先级: `P0`
> 来源: `docs/knowledge/acceptance/2026-07-23-wp-02b-acceptance.md`

---

## Kickoff

### 目标

- 将 `WP-02B` 已识别的 4 个 P0 缺口从“已登记偏离”推进到“可执行整改”。
- 先建立最小可拦截门禁，再推进增强能力，避免继续口头承诺。

### 本包范围

- `DEV-0001` 审计日志持久化与留存策略
- `DEV-0002` 管理员 2FA 最小可用方案
- `DEV-0003` 超 500 条导出审批阀门
- `DEV-0004` 登录暴力破解 5 次锁定

### 不在本包范围

- 不做年度第三方渗透测试采购与执行
- 不做全域敏感字段一次性加密重构
- 不做所有安全条款的全集修复

### 入场条件

- `DEV-0001~0004` 已写入 `.trae/compliance/deviation-registry.json`
- `WP-02B` Sprint-0 基线验收卡已确认
- `WP-COMPLIANCE` 模板与编号规则已可复用

### 退出标准

- 4 个偏离单均有明确整改方案、代码落点、验收口径、回滚口径
- 至少形成一套可进入 Block Merge/pre-release 的最小门禁
- 每个偏离单都挂有 acceptance 证据落点

### 风险

- 如果只补文档不补拦截，仍然无法形成真正门禁
- 如果只补后端逻辑不补运维/审批链，会造成“代码上看似完成，流程上仍失守”

### 回滚口径

- 若任一整改涉及登录、审计、导出主链默认行为变更，必须单列 `RB-WP-02B-00x`
- 若 2FA 或锁定策略引发大面积误伤，允许由大飞哥拍板降级为“记录+告警优先”

## Tasks

### T1 审计留存（DEV-0001 / BS-0050）

- 确定审计日志持久化落点:
  - DB 表 / 对象存储 / 冷存储归档
- 明确保留策略:
  - 在线保留
  - 归档保留
  - 查询出口
- 补最小验收:
  - 新增记录
  - 查询
  - 留存时间口径

### T2 管理员 2FA（DEV-0002 / BS-0057）

- 确定最小方案:
  - TOTP
  - 备用恢复码
  - 高危操作二次验证
- 明确适用范围:
  - admin 登录
  - 导出
  - 权限变更
  - 配置变更

### T3 导出审批阀门（DEV-0003 / BS-0059）

- 定义阈值:
  - `<= 500` 直接导出
  - `> 500` 自动拦截并发起审批
- 对齐流程:
  - RBAC
  - workflow
  - 审计记录
  - 导出结果回写

### T4 暴力破解防护（DEV-0004 / BS-0064）

- 定义规则:
  - 连续失败 5 次锁定
  - 锁定时长
  - 解锁方式
  - 审计记录
- 明确适用入口:
  - admin 登录
  - API 认证

### T5 门禁接入

- 明确哪些属于:
  - Block Merge
  - pre-release
  - 仅记录 DEV
- 把检查口径回写到:
  - `scripts/pre-release-check.sh`
  - 相关 acceptance card
