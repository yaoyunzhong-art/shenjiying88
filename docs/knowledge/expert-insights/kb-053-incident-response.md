# 事件响应流程

> 分类: DevOps | 标签: 事件响应, 故障管理, MTTR, Runbook | 适用: SRE, 运维团队

## 概述

事件响应(Incident Response）是指在系统发生故障或异常时，组织、协调和执行恢复活动的结构化流程。一个良好的事件响应流程可以显著缩短MTTR(平均故障修复时间）、减少故障影响范围并降低类似故障的复发率。Shenjiying88的事件响应流程基于Google SRE的实践经验，结合团队规模和业务特点进行了调整。

根据Gartner的预测，拥有正式事件管理流程的组织在故障恢复速度上比没有流程的组织快约3倍。Shenjiying88定义了完整的事件生命周期：检测→分类→响应→恢复→复盘→改进。每个阶段有明确的角色分工和操作指南。事件响应不是临时组织的"救火队"，而是一个经过反复演练的标准操作流程。

## 核心原则

- **原则1: 事件分类明确优先级**: P0(系统完全不可用，影响所有租户）→ 立即响应，5分钟确认，30分钟恢复SLA。P1(主要功能受损，影响部分租户）→ 15分钟确认，2小时恢复。P2(次要功能受损，影响小部分租户）→ 1小时确认，24小时内修复。P3(非功能性影响，如UI显示问题）→ 正常工作时段处理。
- **原则2: 事件响应统一入口**: 所有告警和事件通过统一平台(PagerDuty/DutyPhp）管理，值班工程师通过App确认和更新事件状态。禁止通过私人消息处理事件——所有沟通在事件频道中进行(飞书/企业微信专有群组），事后自动归档作为复盘材料。
- **原则3: 时间线记录为第一响应动作**: 在事件响应开始时，立即在事件频道中建立时间线。记录每个关键动作的时间点、操作人和结果。这为事后的根因分析和故障报告提供了最准确的数据。Shenjiying88的事件文档模板包含"时间线"章节。
- **原则4: 每个事件需要有对应Runbook**: P0和P1级别的故障都应有一个对应的Runbook——详细描述故障的处理步骤、回滚方案、回滚验证方法和升级路径。Shenjiying88的Runbook存储在Git仓库(`ops/runbooks/` 目录）中持续维护。没有Runbook的服务不能上线。
- **原则5: 复盘会(Postmortem）无责备文化**: 事件恢复后进行复盘，根本目标是改进系统和流程而非追责。Shenjiying88的复盘报告包含五个核心问题：发生了什么？为什么？我们如何快速恢复？如何避免再次发生？如何更快检测和响应？

## 实践案例（基于shenjiying88项目）

- **案例1: 数据库故障事件响应流程**: 当数据库连接池耗尽告警触发时：(1) 值班工程师5分钟内确认并建立事件频道； (2) 检查数据库连接数和慢查询日志； (3) 如果是慢查询导致，kill慢查询进程并优化相关SQL； (4) 如果是连接泄漏，重启数据库连接池或紧急扩容； (5) 恢复后记录时间线、创建Postmortem issue。每个操作步骤都有对应的命令模板可直接复制使用。

- **案例2: 审计任务超时事件Runbook**: `ops/runbooks/audit-timeout.md` 包含：(1) 检查告警指标确认受影响范围； (2) 查看队列积压情况——`redis-cli LLEN audit:queue`； (3) 查看最近失败的审计任务——日志查询模板已提供； (4) 手动重试失败的任务——API调用命令； (5) 如果队列积压超过阈值，临时增加消费者数量——K8s Scale命令。

## 反模式警示

- **反模式1: 通过私人消息处理故障**: 故障发生时，几个人各自在私聊里说"发现了一个问题"。这导致信息孤岛、多人重复排查同一问题、找不到完整的时间线。所有事件沟通必须在公共频道进行，事后自动存档可追溯。

- **反模式2: 跳过复盘直接修复**: 故障恢复后就说"好了，继续吧"，不进行复盘。这导致同样的故障可能再次发生。Shenjiying88的规则是：产生用户影响的P0/P1事件必须进行复盘，复盘报告需要在事件恢复后72小时以内发布。
