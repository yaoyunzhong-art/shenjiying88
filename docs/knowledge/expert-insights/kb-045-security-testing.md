# 安全测试指南

> 分类: 测试策略 | 标签: 安全测试, 渗透测试, OWASP, 漏洞扫描 | 适用: 安全工程师, QA

## 概述

安全测试是保障应用免受恶意攻击和未授权访问的重要手段。与功能测试不同，安全测试需要从攻击者的视角思考——"如果我是黑客，我会从哪里突破？"。Shenjiying88系统承接企业审计数据，涉及大量敏感信息(财务报表、合规数据、员工记录），安全漏洞可能导致灾难性的数据泄露。系统在应对OWASP Top 10 2021中所列的各类安全风险时，需要建立分层防护和系统性的安全测试体系。

根据IBM 2025年数据泄露成本报告，数据泄露的平均成本为486万美元，其中识别和遏制泄露平均需要277天。Shenjiying88将安全测试分为三个阶段：开发期的SAST(静态安全测试）集成到CI，测试期的DAST(动态安全测试）扫描运行中的应用，以及发布前的第三方渗透测试。

## 核心原则

- **原则1: 安全测试左移(Shift Left）**: 在开发阶段就引入安全检测，而非等到发布前才进行安全审计。Shenjiying88在CI中集成SonarQube进行SAST扫描——每次PR提交自动检测SQL注入、XSS、CSRF等漏洞模式。发现的严重漏洞拦截PR合并。
- **原则2: 基于OWASP Top 10的测试体系**: 针对每类主要安全威胁编写专门的测试用例——SQL注入测试使用 `' OR 1=1 --` 等Payload验证输入过滤有效性；XSS测试使用 `<script>alert(1)</script>` 验证输出编码；CSRF测试验证Token机制。
- **原则3: 认证授权测试覆盖所有边界**: 测试所有API端点是否校验用户认证。Shenjiying88的自动化脚本遍历所有API路由(从OpenAPI文档提取），对未携带Token的请求验证是否返回401，对携带Token但权限不足的请求验证是否返回403。
- **原则4: 依赖漏洞持续扫描**: 使用 `npm audit` 和Snyk每周扫描项目依赖，检测CVE漏洞。Shenjiying88配置了自动化的依赖更新——当依赖中存在Critical级别漏洞且已有修复版本时，自动创建Dependabot PR。npm依赖的前3层(直接依赖+间接依赖）必须保持零Critical漏洞。
- **原则5: 加密传输和存储的验证**: 验证HTTPS强制跳转(301）、API请求是否监听在环回地址、密码/密钥是否使用了BCrypt/Argon2强哈希、数据库连接是否使用TLS。Shenjiying88的Security检查脚本自动验证这些配置是否正确。

## 实践案例（基于shenjiying88项目）

- **案例1: API安全测试自动化**: 编写了自动化安全测试套件，对122个API端点执行：(1) 认证绕过——发送不带Token的请求，验证返回401； (2) 水平越权——租户A的Token访问租户B的资源，验证返回403； (3) SQL注入——在查询参数和请求体中注入SQL Payload，验证不会被解析为SQL命令。每天定时执行一次，问题即时告警。

- **案例2: 依赖漏洞扫描流水线**: CI的'security-audit'阶段运行 `npm audit --audit-level=high`，当存在High及以上级别漏洞时流水线失败输出漏洞详情。Shenjiying88维护了 `audit-resolve.json` 文件记录已确认不会造成实际影响(如仅在开发依赖中）的漏洞。每季度人工review这个白名单。

## 反模式警示

- **反模式1: 只在发布前做一次安全测试**: 安全测试应当是持续的过程，而非"发布前大检查"。代码每天都在变——新的功能可能引入新的漏洞。每周的安全扫描 + 每次PR的安全检查是合理的节奏。

- **反模式2: 忽视第三方依赖安全**: 认为"只要我的代码安全就行"。现代应用中75%以上的代码来自第三方依赖。lodash的漏洞、axios的CVE、甚至TypeScript编译器的恶意版本都可能成为攻击入口。必须持续监控和更新所有依赖。
