# 租户隔离策略深度分析

> 分类: 技术架构 | 标签: 多租户, 数据隔离, 安全, SaaS | 适用: 架构师, 安全工程师

## 概述

租户隔离是多租户SaaS系统中最关键的安全设计决策。隔离粒度的选择直接影响数据安全性、运维复杂度和成本结构。根据CIS SaaS Security Benchmark 2025的分类，租户隔离呈现一个谱系：从最低成本的共享一切模式(Shared Everything），到最高安全性的每租户独立部署(Dedicated Instance）。Shenjiying88系统处理的审计数据涉及企业客户的合规报告和敏感数据，需要在成本和安全性之间找到最合适的平衡点。

OWASP多租户安全指南建议对审计合规类数据进行"SOC 2级别"的隔离保护。Shenjiying88采用混合隔离模型：共享应用层计算资源 + 数据库行级别隔离(Schema隔离用于高价值租户） + 文件存储桶隔离。这一模型在成本上接近共享模式，在安全级别上接近独立模式，实现了约83%的隔离有效性同时保持约40%的成本节省。

## 核心原则

- **原则1: 数据隔离三层防线**: 第一层: 应用层中间件强制添加租户ID过滤。第二层: 数据库层SQL添加WHERE tenant_id约束。第三层: 存储层按租户分区存储文件。三层中任意两层失效仍有兜底保护。Shenjiying88在数据访问审计中定期验证三层隔离的一致性。
- **原则2: 租户ID不可篡改**: 租户ID从认证Token中提取，而非请求体中获取。任何试图通过API参数传入不同租户ID的行为都会被网关拦截。后端服务仅信任从JWT中解析的tenant_id，拒绝请求体中的租户信息。
- **原则3: 高安全租户独立基础设施**: 对于金融、政务等强合规行业的租户，提供独立Schema(数据库内独立Schema）或独立数据库实例选项。Shenjiying88在租户配置中标记 `isolation_level` 字段，自动路由到对应的连接池。
- **原则4: 存储隔离不可绕过**: 文件(审计报告附件、截图）存储在按租户ID命名的目录/AWS S3 Prefix下。STS临时凭证的权限策略限定在特定Prefix，确保租户A无法访问租户B的文件。这是一道独立于应用层的数据保护墙。
- **原则5: 跨租户操作一律拒绝**: 所有跨租户的数据访问(如企业管理员查看所有租户的审计报告）必须通过专门的跨租户管理接口，且记录审计日志。此操作需要超级管理员权限并需要二次确认。

## 实践案例（基于shenjiying88项目）

- **案例1: Repository层强制注入租户过滤**: Shenjiying88的BaseRepository类封装了 `applyTenantFilter()` 方法，每个查询自动附加 `WHERE tenant_id = :currentTenantId`。所有聚合查询(COUNT、SUM等）也自动添加租户过滤，避免跨租户统计数据泄露。此机制在单元测试中通过mock tenantId上下文进行验证。

- **案例2: 租户级别的资源配额隔离**: 每个租户拥有独立的存储配额(免费版10GB/专业版100GB/企业版1TB）、API请求配额(并发100/500/5000）和数据库连接池配额。当租户达到资源上限时，仅对该租户限流，不影响其他租户的服务质量。实现上采用Redis多维度计数+租户ID Key前缀。

## 反模式警示

- **反模式1: 应用层隔离但数据库层不分隔**: 所有数据混在同一张表中仅靠应用层过滤是最危险的设计。一次SQL注入或代码Bug就可能导致整个数据库的数据泄露。Shenjiying88即使在共享数据库模式下，也使用单独的Schema作为第二层隔离。

- **反模式2: 共享缓存中混用租户数据**: 如果Redis缓存未按租户隔离，租户A可能通过缓存Key遍历获取到租户B的数据。Shenjiying88的缓存Key强制包含 `tenant_id` 前缀(如 `cache:tenant_123:audit:456`），并在缓存查询时校验访问租户身份。
