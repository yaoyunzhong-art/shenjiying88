# 容灾备份策略

> 分类: DevOps | 标签: 灾备, 数据备份, RPO, RTO, 高可用 | 适用: SRE, DevOps

## 概述

容灾备份策略定义了系统在面对灾难性故障(数据中心级故障、自然灾害、人为失误）时如何保护数据和恢复服务。容灾设计的两个核心指标是RPO(Recovery Point Objective）——允许丢失的数据量对应的时间窗口，和RTO(Recovery Time Objective）——服务完全恢复所需的时间。Shenjiying88系统作为企业级审计平台，数据完整性和服务可用性直接影响客户的合规工作。

根据Veritas 2025年数据保护报告，企业平均每次宕机的成本为16.8万美元。RPO为1小时的业务每小时数据丢失成本约为8500美元。Shenjiying88根据业务重要性和合规要求，为不同数据和服务定义了差异化的RPO/RTO目标: 审计报告数据RPO≤5分钟(准实时备份），非核心配置数据RPO≤1小时，核心审计服务RTO≤30分钟。

## 核心原则

- **原则1: 3-2-1备份法则**: 至少保留3份数据副本，存储在2种不同类型的介质上，其中1份在异地保存。Shenjiying88的审计数据备份策略：主数据库(PostgreSQL）+ 每日全量备份(本地存储）+ 近实时WAL归档到S3(异地存储）。三份独立副本确保任何单一故障不会导致数据丢失。
- **原则2: 异地多活/冷备**: 生产环境部署在香港阿里云(主），新加坡阿里云(灾备）。主站点运行所有服务，灾备站点保持数据库同步(通过流复制），但应用层不启动。当主站点不可用时，通过DNS切换将流量导向灾备站点，启动应用容器。RTO目标为30分钟。
- **原则3: 定期恢复演练**: 备份不经过恢复测试就等于没有备份。Shenjiying88每季度执行一次完整的灾备恢复演练：从备份重建数据库→启动应用→验证数据完整性→确认API可用→切换DNS。演练结果记录在演练报告中，发现的问题纳入改进清单。
- **原则4: 备份加密与合规**: 所有备份数据在传输过程中使用TLS加密，在存储端使用AES-256加密。备份Key与数据分离存储。Shenjiying88的备份策略符合SOC 2和GDPR对数据保护的要求——加密的备份文件如果不幸泄露也无法解密。
- **原则5: 数据冷热分离归档**: 活跃的审计数据(180天内）存储在PostgreSQL主数据库。归档数据(180天至5年）存储在S3/OSS的Parquet格式文件中，通过Presto/Trino查询引擎访问。归档数据在S3上开启版本控制，防止误删除。超过5年的数据按合规政策销毁。

## 实践案例（基于shenjiying88项目）

- **案例1: PostgreSQL WAL归档 + PITR**: 启用PostgreSQL的 `archive_mode = on`，WAL segment通过 `archive_command` 持续归档到阿里云OSS。当需要进行Point-in-Time Recovery时，使用 `pg_restore` + WAL replay恢复数据库到故障前的秒级精确状态。RPO实现为5分钟(每5分钟强制切换WAL segment）。备份文件保留30天。

- **案例2: 灾备切换演练流程**: (1) 演练启动，定时暂停主站点的API Gateway(模拟故障）； (2) DNS自动切换到灾备站点的健康IP(如果超过30秒无响应）； (3) 灾备站点同时接管服务：数据库Promote为主库(启动写操作）、应用容器自动扩容、连接池重建； (4) 验证所有核心API返回200，审计任务可以正常创建和执行； (5) 演练结束后切换回主站点并同步增量数据。

## 反模式警示

- **反模式1: 只备份不验证**: 定期备份但不做恢复测试，等到真的需要恢复时才发现备份文件损坏或无法使用。Shenjiying88的每周恢复演练是对备份有效性的最低验证。每月一次的完整恢复演练则覆盖更全面的场景。

- **反模式2: 备份与生产数据共用一个存储系统**: 将备份数据存储在与生产数据相同的存储系统或同一区域。一旦存储系统故障，主数据和备份同时丢失。备份必须存储在独立的存储系统上，最好是不同的云区域或不同的对象存储桶。
