# 告警策略设计

> 分类: DevOps | 标签: 告警, Alertmanager, PagerDuty, 值班 | 适用: SRE, DevOps

## 概述

告警是监控系统的"最后一英里"——如果告警没有在正确的时间以正确的方式通知到正确的人，那么整个可观测性基础设施的价值会大打折扣。"告警疲劳"(Alert Fatigue）是一个普遍问题：过多的、低信噪比的告警会让值班人员变得麻木，最终忽略真正重要的告警。Shenjiying88的告警策略设计遵循"减少噪音、加速响应、明确升级"的原则。

根据PagerDuty的2025年事件响应报告，高效团队每天的人均告警数控制在3个以下，而低效团队高达50+个。告警数量与事件响应质量呈反比——每增加10个告警/天，MTTA(平均确认时间）增加15%。Shenjiying88的告警策略明确区分了P0(系统不可用）、P1(功能受损）、P2(次要影响）和P3(警告性通知）四个优先级。

## 核心原则

- **原则1: 告警必须可操作(Actionable）**: 每个告警必须触发一个明确的动作——页面响应、进入作战室、自动拉起新实例等。如果一个告警不需要任何人做任何操作，它就是噪音，应当被抑制或删除。Shenjiying88对每个告警规则要求Review并确认"收到告警后做什么"。
- **原则2: 告警阈值合理且有缓冲**: 使用多阈值策略——WARNING阈值(提前预警告警）和CRITICAL阈值(需要立即响应）。Shenjiying88的CPU使用率告警: WARNING@>70%持续5分钟，CRITICAL@>90%持续2分钟。5分钟的缓冲避免了因流量毛刺导致的虚假告警。
- **原则3: 告警分组与关联(Alert Correlation）**: 来自同一故障源的多个告警应当被分组为一次事件，而非每个单独触发。当数据库宕机时，与之相关的数据库连接失败、查询超时、服务不可用等告警应当聚合为一个事件。Shenjiying88使用Alertmanager的`group_by: ['cluster', 'alertname']` 配置实现分组。
- **原则4: 告警升级机制**: P0告警在5分钟未被确认时自动升级到值班主管，10分钟升级到运维经理。P1告警在15分钟未确认升级。P2告警升级到团队频道。Shenjiying88的PagerDuty配置了三级升级策略：一级值班工程师→二级SRE Lead→三级CTO。
- **原则5: 静默期(Maintenance Window）**: 计划内的维护操作期间，相关告警自动静默。Shenjiying88在部署窗口前通过Prometheus API创建定时静默规则，部署结束后自动清理。静默规则包含静默理由和维护人，用于事后审计。

## 实践案例（基于shenjiying88项目）

- **案例1: 审计服务核心告警规则**: (1) P0: Audit Service响应错误率>5%持续1分钟 → 电话通知值班工程师； (2) P1: 审计任务队列积压>1000等待2分钟 → 企业微信通知+群@； (3) P2: 数据库连接池使用率>75% → 发送到Ops频道； (4) P3: 报告生成任务平均耗时增加>50%(同比上周同时段） → 发送到团队频道。每个告警规则包含Runbook链接。

- **案例2: 告警沉默窗口**: 每两周周三凌晨2:00-5:00为系统维护窗口。在维护开始前，通过Grafana的Alerting界面为本次窗口创建静默规则。维护期间任何来自被维护组件的告警都会被自动抑制。维护结束后自动清理静默规则并检查是否有被抑制但未解决的告警。

## 反模式警示

- **反模式1: 为每个指标都创建告警**: 把所有Prometheus指标都配上告警规则是"告警疲劳"的直接原因。Shenjiying88只对四个黄金信号和明确的SLO相关指标创建告警。每个告警规则必须回答"如果这个告警触发，我是否需要中断正在做的事情来处理它？"如果不需要，就不能设置为告警。

- **反模式2: 告警阈值一成不变**: 系统上线初期的50%CPU阈值可能在3个月后已经不合理(业务增长）。Shenjiying88每季度review告警阈值，结合实际数据(通过PromQL的 `quantile_over_time` 和历史趋势图）进行调整。阈值建议基于实际监控数据的百分位数而非拍脑袋决定。
