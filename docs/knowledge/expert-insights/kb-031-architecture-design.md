# 多租户SaaS架构设计原则

> 分类: 技术架构 | 标签: SaaS, 多租户, 架构设计, 隔离 | 适用: 全栈架构师

## 概述

多租户SaaS架构是企业级云服务的核心设计模式，其核心目标是在共享基础设施的前提下，为每个租户提供安全、隔离且可定制化的运行环境。Shenjiying88系统作为服务于多个企业的审计自动化平台，天然需要支持多租户场景——每个企业用户需要独立的工作空间、数据存储和权限体系，同时共享平台的计算资源与服务能力。

传统的单租户部署模式虽然隔离性最强，但在资源利用率和运维成本上存在明显劣势。多租户SaaS架构通过共享基础设施，将边际服务成本降低60-80%，同时通过精心设计的隔离层保障数据安全。根据Gartner 2025年的研究报告，85%的SaaS企业采用基于行的多租户模型(每租户数据共享同一数据库但通过租户ID隔离），仅有10%采用每租户独立数据库模式。

## 核心原则

- **原则1: 租户感知贯穿全栈**: 从API网关到数据层的每一层都必须具备租户上下文识别能力，避免"漏层"导致的数据交叉。Shenjiying88在中间件层注入租户ID，确保服务层无需手动处理租户隔离。
- **原则2: 最小共享最大隔离**: 共享计算层(CPU/内存）、隔离数据层(数据库）。对于审计合规类数据，采用数据层严格隔离(每租户独立Schema或独立表空间），应用层通过连接池路由实现。
- **原则3: 租户画像驱动资源分配**: 根据租户规模(SMB/企业/集团）分配不同的计算资源和配额(QPS限制、存储上限、并发连接数）。Shenjiying88在租户注册时采集企业规模信息，自动匹配资源模板。
- **原则4: 无感扩缩容**: 新增租户和租户升级应完全自动化，无需人工介入。采用注册即激活模式，租户创建后自动完成Schema初始化、默认角色创建和基础数据填充。
- **原则5: 可观测性维度包含租户标签**: 所有监控指标、日志和追踪都必须携带tenant_id标签，支持按租户粒度排查问题和生成账单。这是计费和SLA审计的基础。

## 实践案例（基于shenjiying88项目）

- **案例1: 中间件层租户注入**: Shenjiying88在NestJS全局中间件中解析JWT中的租户ID，通过RxJS Observable流在请求生命周期内传递tenant_id。所有数据库查询在Repository层自动添加WHERE tenant_id = :tenantId条件，开发者无需手动编写租户过滤逻辑。这一设计将租户隔离的开发者心智负担降至零，同时避免了99%的漏隔离Bug。

- **案例2: 租户限流与配额**: API网关层根据租户的订阅套餐动态调整限流阈值。免费租户每分钟100次请求，企业租户每分钟5000次。实现上采用Redis Sliding Window + 租户ID维度进行计数，超过配额时返回429状态码并附带Retry-After头。Shenjiying88的租户配置中心维护了这份配额映射表。

## 反模式警示

- **反模式1: 仅在应用层做租户隔离**: 如果数据库层没有租户过滤(所有租户数据混在同一个表里），一次应用层Bug就可能导致数据泄露。Shenjiying88强制在数据库层也存储tenant_id并建立索引，即使应用层隔离失效，数据库级过滤仍能兜底。

- **反模式2: 共享连接池不做隔离**: 一个租户的慢查询可能耗尽整个连接池，影响所有租户。应当为高价值租户配置独立连接池，或使用数据库代理层(如PgBouncer）基于租户ID进行连接路由。
