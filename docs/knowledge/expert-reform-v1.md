# 🦞 神机营SaaS 专家体系改革 V1
> 2026-07-12 制定 · 54位专家全流程赋能

---

## 一、问题诊断

### 当前问题
1. **专家报告没人看**：每天9份报告(晨会/午会/晚会/自学/验收/对齐等)，累计35KB+/天，但龙虾哥没时间读，树哥根本不读
2. **专家不碰代码**：54个专家从未实际review过一行代码，所有Gate自动签署
3. **专家不测功能**：测试全交给树哥写角色测试，专家从未手动跑过

### 根因
- 报告是"写给cron看的"→为了完成任务而写，不是为了指导开发
- Gate自动签署→变成流程装饰，没有实际把关
- 专家以"写报告"代替"干活"→形式主义

## 二、改革核心：报告变任务卡

### 原则
- **报告长度砍掉80%**，只留"一句话结论"+"三个问题"
- **每个报告必须产出可执行任务**（格式：行动卡片）
- **行动卡片必须入任务池**，要不被关闭要不被验证

## 三、三阶段专家赋能

### 阶段A：写前（需求大门）

每张功能开发卡必须经历：
```
E3张营销(业务必要性) → 签"需求合理"或"需求不合理"
E1陈架构(技术可行性) → 签"架构OK"或"需修改"
E13收银专家(P-35时) → 签"收银流程正确"
```

**产出一张需求卡片**，存为 docs/knowledge/requirement-cards/YYYY-MM-DD-PHASE.md

### 阶段B：写中（代码质量门）

每2天，E1陈架构从最新commit随机抽3个TS文件：
```
E1评分: A=优秀(没问题) B=合格(小问题) C=需修改(退修) D=重写(整模块重写)
C和D的模块标记为"质量风险"，下个脉冲重点回查
```

### 阶段C：写后（功能验收门）

每个Phase发版前，由对口专家手动跑1个核心角色测试：
```
例: P-35收银 → E13收银专家跑1次"店长→开单→加商品→结账→打印"
E13签: "功能可用" 或 "功能不可用-原因XXX"
不可用的Phase不能close
```

## 四、报告模板改革

所有专家报告改为三部分：

```
# [报告名] · 日期

## 一句话结论
[不超过2行]

## 发现的问题
1. [问题1] → 任务卡ID: TSK-XXX
2. [问题2] → 任务卡ID: TSK-XXX

## 产出的行动卡片
| 卡片ID | 动作 | 负责人 | 截止 |
|--------|------|--------|------|
| TSK-XXX | 具体做什么 | 谁做 | 什么时间 |
```

**报告谁写的：54人按月轮值，每天只有2位专家值班写报告**
**报告谁看的：龙虾哥每天早上读"一句话结论"部分 + 所有任务卡入任务池**

## 五、测试专家体系

每个功能指定2位测试专家：
- **业务测试专家**：从角色测试中选择1个最核心的，手动跑
- **技术测试专家**：验证自动测试覆盖率和边界条件

8角色对应关系：
| 角色 | 测试专家 | 签名要求 |
|------|---------|---------|
| 👔店长 | E13(收银)+E40(客户) | P-35收银签收银测试 |
| 🛒前台 | E23(前台)+E7(体验) | P-36会员签注册充值 |
| 👥HR | E18(HR)+E25(后勤) | 排班测试未安排等 |
| 🔧安监 | E2(安全)+E35(运维) | 安全审计测试 |
| 🎮导玩员 | E16(导玩)+E28(设备) | 设备操作测试 |
| 🎯运行专员 | E24(运行)+E31(巡检) | 运营验收测试 |
| 🤝团建 | E27(团建)+E33(联名) | 团建活动测试 |
| 📢营销 | E3(营销)+E21(增长) | 营销活动测试 |
