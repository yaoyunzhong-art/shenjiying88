# ADR-045: 科学知识体系 V2 — 五环闭环 + 12种科学玩法

## 状态
**提议** → 接受 → 废弃

## 背景

### 问题诊断

现有知识体系有 5 个产生源、6 个核心知识库、3 种处理机制，但它们**各自为政**：

| 环节 | 现状 | 痛点 |
|:-----|:-----|:------|
| 知识产生 | 搜索/专家/验收各自独立 | 不交叉验证，虚假/过期知识常驻 |
| 知识处理 | 无标准化处理流程 | 日采→入库之间缺"结构化 extraction" |
| 知识入库 | 6 个核心库各玩各的 | 无关联，知识之间缺映射 |
| 知识应用 | 派单不引用知识 | 树哥不知道最新行业动态/竞品数据 |
| 知识闭环 | 从不检查知识使用率 | 80% 知识产生后从未被引用 |

### 核心缺失

1. **知识→任务映射**：新知识没有自动变成开发指令
2. **知识审计缺失**：没人检查"6 个知识库是否被引用"
3. **知识间无关联**：日采、专家简报、反模式库互相独立
4. **知识生命周期未规划**：新知识→成熟→应用→反馈→归档/废除
5. **知识使用率不可测量**：无 KPI

## 方案：五环知识闭环 + 12种玩法

### 总体架构

```
┌──────────────────────────────────────────────────┐
│              五环知识闭环 (5 Rings)               │
│                                                   │
│  【采集】→【处理】→【入库】→【赋能】→【反馈】   │
│    R层       E层       S层       A层       F层    │
│                                                   │
│  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐  ┌──────┐│
│  │ R1-R5│→│E1-E5 │→│S1-S6 │→│A1-A5 │→│F1-F5 ││
│  │ 5种  │  │ 5种  │  │ 6个  │  │ 5种  │  │ 5种  ││
│  │采集源│  │处理法│  │知识库│  │赋能法│  │反馈法││
│  └──────┘  └──────┘  └──────┘  └──────┘  └──────┘│
│                                                   │
│       ↑ - - - - - - - - - - - - - - - - - ↓       │
│       12种科学玩法贯穿所有环节                     │
└──────────────────────────────────────────────────┘
```

### 第一环：知识采集 (R1-R5)

| 编号 | 采集源 | 触发时机 | 产出物 | 验证方式 |
|:----:|:-------|:--------:|:-------|:---------|
| R1 | 网络搜索 | 11:00 日采 | 结构化行业洞察条目 | ≥2 独立源交叉验证 |
| R2 | 专家学习 | 08:00/14:00/20:45 | 专家建议 + 知识卡片 | 学习组长签署 |
| R3 | 验收反馈 | 每个脉冲 | 反模式/正向模式条目 | PRD 对齐验证 |
| R4 | 竞品数据 | 周日 04:00 | 竞品动向摘要 + 战术建议 | 至少 5 城数据 |
| R5 | 安全扫描 | 07:30 | 合规要求检查项 | 基线合规验证 |

### 第二环：知识处理 (E1-E5)

每个采集源产出必须经过标准化处理：

| 编号 | 处理动作 | 输入 | 输出 |
|:----:|:---------|:-----|:-----|
| E1 | **提取结构化** | 原始搜索文本 | 行业洞察条目(≤140字摘要+标签+来源) |
| E2 | **归类标记** | E1 输出 | 映射到 6 大知识库某一条 |
| E3 | **权重计算** | 知识条目 | 评分 = 源头可信度×30% + 专家验证×40% + 时效性×30% |
| E4 | **赋能卡片生成** | 经过 E1-E3 | 可注入派单的卡片(<=200字) |
| E5 | **关联图谱** | 全部知识条目 | "知识→模块"映射表（见 alignmnet docs） |

### 第三环：知识存储 (S1-S6)

| 编号 | 知识库 | 用途 | 更新频率 | 关联文件 |
|:----:|:-------|:-----|:--------:|:---------|
| S1 | 业务洞察 | 竞品·行业·市场·用户 | 每日(日采日更) | `competitive-intelligence.md` |
| S2 | 技术决策 | ADR·架构·模式·反模式 | 触发式 | `adr/*.md`, `architecture-decisions.md` |
| S3 | 合规要求 | 安全·数据·法规 | 每周(扫描+日采) | `business-rules.md`, `cx-constraints.md` |
| S4 | 反模式库 | AM-001 ~ AM-005+ | 脉冲触发 | `code-ringbeam-alignment.md` |
| S5 | 正向模式库 | PP-001 ~ PP-006+ | 脉冲触发 | 同上 |
| S6 | 每日精华 | 当日最值得记住的 3-5 条知识 | 每日 | `daily-brief.md` 知识 KPI 栏 |

### 第四环：知识赋能 (A1-A5)

**这是最关键的一环——知识必须变成实际行动：**

| 编号 | 赋能方式 | 触发 | 执行方式 |
|:----:|:---------|:-----|:---------|
| A1 | **派单知识注入** | 每次派单 | 自动检索最相关 3 条知识追加到 task 末尾 |
| A2 | **代码审查阻断** | 验收脉冲 | 匹配反模式库 → 阻断 → 指向库条目 |
| A3 | **验收知识检查** | 每个 commit | 圈梁第三道箍：检查最新知识是否被引用 |
| A4 | **能力基线** | 每周日 | 更新"已知知识基线"——团队知道什么 |
| A5 | **竞品响应任务** | 竞品周更新 | 竞品动向→自动生成战术开发任务 |

### 第五环：知识闭环 (F1-F5)

| 编号 | 反馈方式 | 频率 | 说明 |
|:----:|:---------|:----:|:------|
| F1 | **使用率审计** | 每日 22:00 | 每条知识被引用次数、模块、时间戳 |
| F2 | **效果验证** | 每周日 | 使用了知识的代码 vs 未使用的 Bug 率差异 |
| F3 | **知识退化曲线** | 每日 | 未使用知识每 7 天降 10 分, <50 提醒, <20 归档 |
| F4 | **赋能评分** | 每日 | 开发效率提升 × 知识使用率 |
| F5 | **知识老化清理** | 每周日 | 过期知识→archive 目录标记 |

## 12 种科学玩法

### 第一组：采集侧科学 (3 种)

**① 多源交叉验证** — 同一知识点必须 ≥ 2 独立源确认才入库
> 例：日采搜索到的"订阅制会员 22% 流水贡献"，需要至少 1 篇独立文章+1 条竞品数据佐证
> 执行：日采 E1 阶段标记 verified_by 字段

**② 新鲜度评分** — 每条知识标注源时间和新鲜度(0-100)
> 定义：今天产生的知识=100，每天减(100/90)≈1.11，90天后=0
> 触发：日采 E3 权重计算时 + 查询时自动计算
> 应用：知识查询结果按新鲜度排序，新鲜度<30 自动降级提示"该信息已过时"

**③ 反虚假机制** — 搜索+专家+验收三方交叉验真
> 规则：知识被写入 S1-S3 前必须通过至少 2 个源确认
> 例外：搜索结果自带来源标签（如 ZVZO 报告），无需额外验证
> 假知识标记：被标记为虚假的知识，在反模式库记录

### 第二组：处理侧科学 (3 种)

**④ 自动归类** — 搜索产生的新知识按领域自动归类
> 领域标签: `竞品` `技术` `市场` `用户` `合规` `设备` `会员` `运营` `选址`
> 执行：E2 归类基于关键词匹配+专家复核

**⑤ 权重评分公式** — 每条知识评分的量化模型
```
score = source_trust × 0.30 + expert_vetted × 0.40 + freshness × 0.30
```
> source_trust: 官方数据=100, 行业报告=80, 自媒体=50, 用户评论=30
> expert_vetted: 专家校审过=100, 未校审=50
> freshness: 新鲜度计算(见②)

**⑥ 知识摘要** — 每条知识浓缩为 ≤140 字的"赋能卡片"
> 格式: `[标签] 摘要 (来源, 新鲜度分)`
> 例: `[竞品] 订阅制会员收入占22%全国平均 (ZVZO2026, 90分)`

### 第三组：应用侧科学 (3 种)

**⑦ 知识-任务匹配** — 每次派单自动检索最相关的 3 条知识
> 搜索范围：S1-S6 所有知识库
> 匹配算法：标签匹配(70%) + 关键词语义(30%)
> 输出：任务 prompt 末尾追加"📚 相关知识点: xxx"

**⑧ 知识阻断** — 代码审查检测反模式 → 阻断 → 指向库条目
> 触发：验收脉冲发现 as any / ts-nocheck / 未测试
> 动作：阻断该 commit，指向对应 AM 条目
> 升级：同一模块连续 3 次阻断 → 升级为知识欠费(知识密度不足)

**⑨ 知识密度** — 衡量每个模块的知识负载
> 公式：知识密度 = 关联知识条目数 / 该模块千行代码
> 目标：每个活跃模块 > 3 条知识/千行
> 检查：每日 14:00 午间知识审计

### 第四组：反馈侧科学 (3 种)

**⑩ 退化曲线** — 未使用知识每 7 天降 10 分
> 应用：22:00 每日闭环节读取所有知识使用计数, 未被当前日任务引用的知识评分 -10
> 退化阈值: <50 → 加入"知识提醒"推送至 daily-brief
> 归档阈值: <20 → 自动归档到 archive 目录
> 活化：被重新引用后评分回到 80(快速恢复)

**⑪ 知识使用统计** — 每条知识被引用的次数、模块、时间戳
> 数据源：派单注入记录、代码审查记录、专家简报引用
> 存储：`docs/knowledge/usage-stats.md`
> 展示：daily-brief 知识 KPI 栏

**⑫ 知识赋能 ROI** — 量化知识对开发效率的影响
```
ROI = (应用知识后的 bug 率下降百分比) / (知识获取成本)
```
> bug 率下降 = 使用了知识的模块 vs 未使用的模块
> 知识获取成本 = 日采 30min × 模型成本 + 处理 10min
> 输出：每周日 L4 批处理时加入
