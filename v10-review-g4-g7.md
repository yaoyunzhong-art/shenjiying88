## 一、B层情报库：营销/运营决策够吗？

### 评分: 65/100

**积极面:**
- T层在V10中分配了12%时间做知识库，有日采机制(11:00 isolated)，有T→O简报推送
- 四条断链修复保证了知识库与专家/AI的闭环
- 20遍审计全部通过，基础闭环链路完整

### ❌ 关键缺失——B层情报库缺少的6个关键维度

**1. 缺少客户行为实时看板**
- 现状：V10无任何"客户行为分析"模块开发规划
- 需要：到店频次、消费偏好、游戏路径、停留时长、放弃原因的实时聚合
- 影响：营销推送、活动设计、排班管理全部依赖行为数据，现在B层没有

**2. 缺少竞品情报维度**
- E17周活动需要竞品活动分析来策划对策，B层目前全无
- V10只有"周日04:00侦察兵2"一条竞品定时任务，但竞品数据入B层的链路未定义
- 建议：至少建立"竞品活动库"模板字段（活动类型/折扣/周期/参与人数）

**3. 缺少营销ROI追踪库**
- E4张营销的P3营销投产比模型需要历史ROI数据支撑
- V10无campaign_tracking表或营销活动效果归因模块
- 每个推送/活动的转化漏斗(触达→打开→到店→消费)链路数据未纳入B层

**4. 缺少内容素材库**
- E15吴内容需要内容模板、AI生成素材、A/B测试结果归档
- V10无content_asset库，AI文案生成素材无复用机制
- E16郑社群的社群活动模板需要"模板配置→一键发布→效果追踪"数据链

**5. 缺少门店运营SOP知识库**
- E8周运营的3套SOP(高峰/平峰/应急)需要结构化存储+自动切换触发条件
- 设备稼动率、排队等待时间、前台转化率等运营KPI数据需实时入B层
- V10无运营SOP在知识库中的数据模型

**6. 缺少客户体验基线数据库**
- E7孙体验的NPS、疲劳度、免打扰偏好等体验数据无存储位置
- E40杨会员的会员生命周期触点数据（注册→首单→复购→休眠→挽回→VIP）无数据模型
- 这些数据分散在各模块，B层应作为体验数据的中枢

### 优化建议

```
B层建议新增模块:
  ┌─ 客户行为数据湖 (实时聚合)
  ├─ 竞品情报库 (周度采集)
  ├─ 营销活动ROI库 (按campaign_id归档)
  ├─ 内容素材库 (模板+AI素材+A/B结果)
  ├─ 门店运营SOP库 (3套SOP结构化)
  └─ 体验基线库 (NPS/疲劳度/偏好)
```

## 二、AI赋能(G环)：数据分析/AI模块够吗？

### 评分: 55/100

**积极面:**
- TOGA四环中G环占据30%Phase开发，有完整的"需求卡→开发→验收→回写"闭环
- 20遍审计中有13-17条覆盖G→A、A→G断链修复，自进化机制有设计
- E9吴AI在R21赋能后完成了push模块缺陷审计，为V10提供了问题清单

### ❌ 严重不足——核心AI能力为零

根据E9吴AI在R21的代码审查结果，**`ai-marketing-service.ts`的核心AI能力全部是Mock**：

| AI能力 | 现状 | 严重程度 |
|--------|------|:--------:|
| 客户画像学习(9维) | ❌ 无member_profile表，无事件驱动更新 | P0致命 |
| 协同过滤推荐 | ❌ 无User-based/Item-based CF | P0致命 |
| 冷启动推荐 | ❌ 无标签推荐兜底 | P0致命 |
| 场景触发引擎(8种) | ❌ 无到店/消费后/排队中/离店后等触发器 | P0致命 |
| 营销时机AI决策 | ❌ 无最佳时机模型 | P1 |
| AI决策日志 | ❌ 无marketing_push_decision_log | P0致命 |
| 疲劳度评分 | ❌ 无member_fatigue_score表 | P1 |
| A/B测试自动优选 | ⚠️ 骨架+MOCK | P1 |

**最严重的风险**: 所有API调用失败fallback到Mock数据——线上环境接口不通时，系统静默返回假数据，无人知晓。

### 研发预算足够吗？——分析

V10时间分配中，**G环(Phase开发)=30%**，但其中：
- 树哥开发时间还需要覆盖P0-P3推送分级、盲盒引擎修复、财务对账等业务功能
- 真正留给AI模块(画像+协同过滤+场景触发+决策日志)的时间微乎其微
- E9吴AI列出的8个核心AI缺陷，保守估算需要 **40-60人天** 开发，V10可行的时间窗口(30%×8h ≈ 2.4h纯开发/天×7天≈17h ≈ 3人天)完全不够

**结论**: 研发预算严重不足。AI赋能停留在"框架"层面，实质内容为零。

### 优化建议

```
AI模块分三期交付:
  Phase 1 (V10优先): AI决策日志 + member_profile表骨架
    → 约8h开发，先建立数据追溯能力
  Phase 2 (V11): 协同过滤离线计算 + 冷启动标签推荐
  Phase 3 (V12): 场景触发引擎 + 疲劳度实时评分 + 营销时机模型
```

## 四、用户体验/客户视角：前端被忽视

### 评分: 40/100

### ❌ 严重问题：V10用户视角缺失

**1. 前端开发时间为零**
- V10时间分配表中，**没有一个板块专门分配给前端/UI/UX开发**
- "Phase开发30%"需要覆盖后端+AI+数据+前端全部，前端大概率被压缩
- E7孙体验的P0-P3推送分级改造（BS-0168~0176）是P0级需求，但V10时间线中无对应锚点

**2. 专家会议15% = 讨论≠落地**
- E7/E40/E21的反馈已经有了（R21赋能产出），但缺少转化为开发任务的时间
- 会议15%中真正留给用户体验讨论的时间≈每天30分钟晨会+30分钟午会=1小时/天
- 这1小时内G4+G5+G6+G7+其他组轮流发言，G7实际发言时间≈15分钟/天

**3. 用户视角在20遍审计中缺失**
- 20遍审计全是"T→O→G→A"的内部闭环审计
- **没有任何一遍验证"最终用户体验是否达标"**——如：
  - 用户是否能在3步内完成预约/下单？
  - SVIP是否感受到黑卡体验？
  - 营销推送是否让用户觉得"帮我"而非"烦我"？
  - 免打扰时段是否生效？

**4. 前端开发无硬保障机制**
- cron清单28个全部是后台/系统任务
- 没有任何前端开发相关的cron（如"15:00 前端组件验收"）
- E21张设计的"加载态/空态/错误态"要求需要UI规范落地，V10无此计划

### 如何保证前端不被忽视？

**强烈建议:**
```
V10-修正案: 将12%知识库中的5%重新分配给前端开发保障

1. 增加cron: "10:30 前端体验检查" (systemEvent)
   → 检查当天开发的C端组件是否有加载态/空态/错误态
   
2. 晨会+午会固定10分钟: 用户体验状态通报
   → G7专家分享当天用户反馈/体验数据

3. 建立"用户验收"第21遍审计:
   → 每2天由E40杨会员对当天上线的功能做用户视角验收
   → 结果写入C层开发库
```

### B层当前只解决了一个问题

TOGA四环中T层的设计目标是"自进化的知识库"，提供了三层(T1/T2/T3)结构、日采机制、简报推送。这**解决了知识库的"存在性"问题**。

但B层(情报库)的定位应该是**支持实时业务决策的数据中枢**，当前版本缺少：
- **实时性**: 11:00日采isolated，业务指标延迟1天
- **业务化**: 数据结构设计偏向"知识管理"而非"情报分析"
- **可视化**: 无看板/报表/Alert机制
- **外部数据聚合**: 竞品、市场趋势、客户舆情未纳入

## 六、对V11的3条核心建议

### 建议1: 🔴 重新分配V11时间预算——业务功能占比不低于50%

**V10的硬伤**: 技术闭环43% vs 业务开发30%，严重头重脚轻

**V11建议分配**:
```
  🐜 业务功能开发(含前端): 50%  ← 从30%提升
  👥 专家会议(侧重业务验收): 10%  ← 从15%降低
  📚 知识库: 5%  ← 12%→5%，B层跑起来后维护为主
  🧪 测试+验收: 10%
  🔄 自进化+监控+审计: 25%  ← 8+6+4=18%→25%，但增加质检
```

**核心逻辑**: V10已建立了TOGA闭环框架，V11应该将重心从"让闭环跑通"转向"让闭环产出价值"。当业务功能不落地时，再完美的闭环也是空转。

### 建议2: 🟡 设立"用户验收"第21遍闭环——G7专家行使否决权

V10的20遍审计全部是系统内部闭环(知识库→专家→AI→自进化)，缺少最关键的**外部闭环：用户反馈→产品迭代**。

**V11新增第21遍审计**:
```
用户验收闭环 (E40杨会员·Champion, 行使否决权)

#21 用户验收  |  每2天  |  检查项:
  a. 新上线功能是否有空态/加载态/错误态?
  b. 核心操作路径是否≤3步?
  c. P0-P3推送分级是否按规范实现?
  d. 免打扰时段是否硬拦截P2/P3?
  e. 用户是否可一键关闭P3营销?

⚠️ 任一项不通过 → E40杨会员行使单票否决 → 功能不可上线
```

**为什么要给G7否决权？**
- E40是Champion级别(权重2.0/单票否决)
- 所有开发最终服务于用户体验
- V10完全缺用户视角，V11必须建立硬保障

### 建议3: 🟢 拆分"大功能组"为"小预算包"——每个预算包独立验收

V10的预算法则("预算≤剩余时间→开跑")过于粗糙：
1. 没有区分功能优先级（P0整改和P3优化一样排队）
2. 没有独立核算预算包（一个Phase占了时间，其他Phase全部停摆）
3. 验收标准没有业务价值验证

**V11建议方案:**
```
预算包粒度 = 0.5~2h开发需求卡 (RQ-XXX)

每个功能包的"三段签名":
  [1] 需求卡签名（E10郑财务校验ROI）
  [2] 开发完成签名（G环自动回写）
  [3] 用户验收签名（E40杨会员检验体验）

预算包优先级策略:
  P0级(交易/合规/体验基础) → 不参与排队，即时插队
  P1级(运营/营销/数据分析) → 等15%时间预算
  P2级(优化/探索/非必需) → 等剩余时间
```

**预期效果**:
- 杜绝"一个大Phase吞掉所有预算，剩下几天无事可做"
- 每个功能包在交付前都有业务+体验双重验收
- 研发预算可观测、可追踪、可追溯
