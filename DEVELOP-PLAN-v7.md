## 1️⃣ 科学决策模型

### 1.1 优先级量化公式

```
Score = Impact × Urgency × Feasibility

Impact (1-5)
  5 = 影响全平台所有租户
  4 = 影响大部分租户（>50%）
  3 = 影响中部分租户（20-50%）
  2 = 影响少数租户（<20%）
  1 = 影响单租户或个例

Urgency (1-5)
  5 = 无此功能 → 业务不可运行
  4 = 无此功能 → 重大业务损失
  3 = 无此功能 → 业务效率下降
  2 = 无此功能 → 体验不佳
  1 = 有替代方案

Feasibility (0.5-2)
  2 = 现有架构直接支持, <2天
  1.5 = 需小幅改造, 2-5天
  1 = 需中幅改造, 5-10天
  0.5 = 需大幅重构, >10天

Score ≥50 → P0 (立即执行)
Score 30-49 → P1 (本周)
Score 15-29 → P2 (本月)
Score <15 → P3 (日后)
```

### 1.3 KPI体系（每个需求必须回答）

每个开发任务必须附带的科学度量:

```
[需求] 来源: E36卫审计
[度量]
  影响面: 全平台盲盒用户（5/5）
  紧迫度: 高并发超卖=安全事故（5/5）
  可行性: 现有模块可改造（2/5）
  评分: 50 → P0
[验证指标]
  上线前: 盲盒超卖率=0%
  上线后: 盲盒秒杀成功率>99.9%
```

## 2️⃣ 科学流程: 专家需求 → 量化 → 开发 → 验证

```
🏛️ 专家提需求
   ↓
🦞 填写《专家需求表》: 需求描述 + 使用场景 + Impact/Urgency/Feasibility
   ↓
🏛️ 各组长复核评分
   ↓
🦞 按Score排优先级 → 更新TASKS_STATUS.md
   ↓
🐜 开发 → 每个commit必须有BS编号+Score引用
   ↓
🏛️ 专家测试: 填《功能验证表》（通过/不通过/改进建议）
   ↓
🦞 验证KPI达标 → 功能闭环 → 知识沉淀
```
