## 2. 🧪 测试设计模式

### 2.1 P(正例) 模式
| 模式 | 说明 | 示例 |
|:-----|:-----|:------|
| P1 全链路 | 数据正向完整流转 | 创建→处理→展示→消费 |
| P2 变体 | 同一流程不同参数 | 多套餐/多支付方式 |
| P3 复合 | 多实体组合场景 | 多合同/多优惠券 |

### 2.2 N(反例) 模式
| 模式 | 说明 | 优先级 |
|:-----|:-----|:------:|
| N1 权限拒绝 | RBAC/鉴权拦截 | P0 |
| N2 资源不足 | 余额/名额/额度 | P1 |
| N3 状态冲突 | 重复操作/过期 | P1 |
| N4 输入校验 | 格式/长度/范围 | P2 |

### 2.3 B(边界) 模式
| 模式 | 说明 | 示例 |
|:-----|:-----|:------|
| B1 零值 | 0元/0数量 | 免费套餐/0元活动 |
| B2 大容量 | 批量操作 | 100员工导入/批量采购 |
| B3 时间边界 | 跨天/并发 | 凌晨打卡/多活动重叠 |
| B4 配额边界 | 达到限制 | 名额满/100人上限 |

## 3. ⚠️ 常见陷阱

### 3.1 数据污染
```typescript
// ❌ 错误: 浅拷贝导致引用共享
const mockData = [...originalData];
// ✅ 正确: 深拷贝
const mockData = originalData.map(d => ({ ...d }));
```

### 3.2 检查顺序
```typescript
// ❌ 错误: 状态检查在前, 防重检查在后(防重应优先)
if (status !== 'pending_payment') throw '状态不对';
if (paymentStatus === 'success') throw '已支付';

// ✅ 正确: 防重优先
if (paymentStatus === 'success') throw '已支付';
if (status !== 'pending_payment') throw '状态不对';
```

### 3.3 多条件错误优先级
```typescript
// 多条件校验建议优先级:
// 1. 安全检查 (权限/鉴权)
// 2. 时间检查 (过期/未开始)
// 3. 状态检查 (发布/激活)
// 4. 防重检查 (重复提交)
// 5. 资源检查 (余额/名额)
```

### 3.4 权限矩阵
```typescript
// 每个 role 的权限应与业务场景匹配:
// store_admin → attendance/approve + leave/approve (门店管理)
// hr → employee/read/write/delete (HR管理)
// staff → attendance/read + leave/read (仅查看)
```
