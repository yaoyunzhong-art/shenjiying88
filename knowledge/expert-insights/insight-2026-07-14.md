### T2-A: 数据管道同步与一致性模式 (链22)

**测试链路**: Admin(数据配置) → API(数据同步) → Domain(领域转换) → TOB(企业展示) → Storefront(前端消费)

**核心机制**:
1. **三种同步模式**: 全量(`full`)、增量(`incremental`)、定时(`scheduled`)
2. **版本冲突策略**: `latest_wins` / `admin_wins` / `timestamp_based` — 同一版本号不同payload产生冲突记录
3. **数据变换引擎**: 币种转换(CNY↔USD/JPY/EUR/HKD)、合规检查(名称/价格/分类)
4. **数据质量评分**: TOB视角的字段完整性评估

**覆盖场景**:
| 子测试 | 类型 | 关键验证 |
|:------:|:----:|:---------|
| P1 全量同步+展示 | 正例 | 配置→3条全量同步→Domain转换→TOB展示→Storefront消费, 全链路一致性 |
| P2 增量+冲突解决 | 正例 | v1→v2增量同步成功, 相同版本冲突检测 |
| N1 合规失败+版本落后 | 反例 | 空名称/零价格/非法分类→合规检查失败; 版本落后→同步失败 |
| N2 数据质量低+价格异常 | 反例 | 不完整数据→TOB质量<100%; 有效记录→正常价格范围 |
| B1 空同步+批量100条 | 边界 | 零记录同步; 100条批量同步性能(<50ms) |
| B2 促销时间边界 | 边界 | 已过期/进行中/即将开始分类; 刚刚结束/刚刚开始微边界; 多目标模块一致性 |
| B3 配置生命周期 | 边界 | 创建→停用→更新模式→重新启用→列表过滤 |

**模式价值**: 数据同步作为微服务架构核心模式, 该测试模板可覆盖所有同步场景(全量/增量/冲突/合规)

### T2-B: 订单全生命周期模式 (链23)

**测试链路**: Mobile(移动端下单) → Storefront(门店确认) → API(订单处理) → Domain(履约/库存) → Admin(管理看板)

**订单状态机**:
```
cart → pending_payment → paid → confirmed → preparing → shipping → delivered → completed
                                             ↘ cancelled ↗                       ↘ refunding → refunded
```

**核心机制**:
1. **库存预占+扣减+释放**: 下单预占 → 确认备货 → 实际扣减 → 取消释放
2. **多渠道并发库存互斥**: mobile/miniapp/storefront/tob 四渠道同时抢购限量商品, 仅前三单成功
3. **订单看板统计**: 总订单/完成/取消/收入/峰值时段/TOP商品

**覆盖场景**:
| 子测试 | 类型 | 关键验证 |
|:------:|:----:|:---------|
| P1 正常下单履约 | 正例 | 创建→支付→确认→备货→配送→送达→完成+看板统计 |
| P2 多渠道并发库存 | 正例 | 10件库存, 4渠道各3件, 仅3成功→释放→重试成功 |
| N1 非法转换+库存不足 | 反例 | 跳过状态→拒绝; 库存不足→预占失败; 重复取消→拒绝 |
| N2 版本冲突+回退 | 反例 | 并发状态更新版本递增; 已完成→配送中→拒绝 |
| B1 空/超大/零金额 | 边界 | 空商品订单(0元); 99999件超大订单; 退款 |
| B2 超时异常+跨店汇总 | 边界 | 未支付滞留→异常检测; 3门店日收入汇总 |
| B3 退款逆向流程 | 边界 | 完成→退款→退款完成→库存回退 |

**模式价值**: 订单状态机是电商系统最核心的业务模型, 该全链路测试模板可覆盖状态转移、并发、逆向和看板

### T2-C: 企业多租户全流程模式 (链24)

**测试链路**: Tob(企业注册) → API(租户路由/认证) → Domain(数据隔离/配额) → Admin(审批配置) → Audit(合规审计)

**核心机制**:
1. **分层租户模型**: free(5用户/1门店) → starter(20/5) → professional(100/20) → enterprise(1000/100) → custom(99999/99999)
2. **功能特性矩阵**: 按企业级别自动分配功能开关(SSO/审计/API/自定义品牌仅enterprise+)
3. **资源配额**: API调用/存储/并发/导出 四项独立配额限流
4. **审计合规**: 操作日志不可篡改、合规报告(导出限制/非工作时间访问检测)

**覆盖场景**:
| 子测试 | 类型 | 关键验证 |
|:------:|:----:|:---------|
| P1 企业注册→审批→订阅→审计 | 正例 | 专业版注册→审批→年付订阅→数据写入→全链路审计 |
| P2 多租户数据严格隔离 | 正例 | 租户A(2条)和租户B(4条)数据完全不交叉; 功能特性隔离 |
| N1 配额超限+非法转换 | 反例 | API调用超限10/10→拒绝; 月度导出超2→拒绝; 重复激活/暂停 |
| N2 订阅过期+功能限制 | 反例 | 取消订阅→拒绝再次取消; 免费版无审计/API/SDK |
| B1 搜索过滤+自动续费 | 边界 | 状态筛选active(2)/pending(2); 级别筛选free(1); 自动续费→更新→取消 |
| B2 合规安全事件检测 | 边界 | 8次导出→DATA_EXPORT_LIMIT违规(medium); 正常操作→无违规 |
| B3 暂停/恢复循环+元数据 | 边界 | 3次暂停/恢复循环; 邮箱/region/industry元数据更新 |

**模式价值**: 多租户SaaS架构的标准化验收模板, 覆盖从企业注册、计费、隔离到合规的全链路
