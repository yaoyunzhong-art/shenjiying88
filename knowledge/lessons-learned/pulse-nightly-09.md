## 经验教训

### 1. 测试链设计: 模块间数据依赖的陷阱

链16 初始版本有 4 个测试失败(23→19 pass)，主要原因:
- **同 SKU 被多个 Phase 共享**: 同一 SKU 先被 Admin 修改(不影响库存), 又被 Mobile 下单(扣库存), 再到 Storefront 展示, 导致后续"库存不足"场景的预期值错位
- **quantity limit(10) > stock check 优先级**: `mobilePlaceOrder` 中限购 10 件的 check 早于库存检 check, 导致 qty=30 时被判定为"超额"而非"库存不足"
- **幂等性 requestId 冲突**: 使用 `Date.now()` 生成 requestId, 同一毫秒内多次调用返回相同值, 导致幂等性拦截

**教训**: 
- 跨模块链中, 若一个模块的操作影响仓储状态, 后续测试的后置条件(Postcondition)应计算实际剩余而非硬编码预期值
- 测试链中每个 Phase 尽量使用独立的测试数据(新 SKU/新订单), 避免前序 test 的副作用
- 请求幂等性 ID 必须用计数器+时间戳组合, 确保毫秒级唯一

### 2. 通知治理: 优先级链式过滤的正确验证

链17 先禁用一个规则→验证新事件被拒→恢复规则, 过程中需注意:
- 禁用规则后, 已生成的通知不会被删除, 只是后续不再创建
- 恢复规则后, 之前的通知状态不受影响
- 各角色的"未读"独立计数, 互不干扰

### 3. 退款流验证: 状态机的严格转换约束

链18 覆盖了完整的 4 态机: `requested→approved→processing→completed`
- `rejected` 是一条终止路径, 不能再翻转为 `approved`
- `completed` 是全终态 (terminal state), 不能再 `processRefund`
- 全额退款时订单标注 `cancelled`, 部分退款订单不变

验证状态机约束的最佳做法: 在状态转换函数入口加 `invalid_status:${current}` guard, 测试用 assert.rejects 覆盖。

### 6. SKU 生命周期 + 缓存一致性模式的关键验证点

链16 的 6 阶段验证设计:
- Phase 1: Admin 创建/上架/编辑(含反例: 重复/负价/停售) — 触发 SDK 事件
- Phase 2: Storefront 展示(已上架列表/详情缓存) — 缓存命中 + 一致性
- Phase 3: Mobile 下单(充足/不足/草稿/超限/无效) — 库存扣减
- Phase 4: Admin 编辑 → 缓存失效 → 重建 — 一致性复检
- Phase 5: SDK 事件追溯(按 SKU/按类型) — 完整事件链
- Phase 6: 边界(连续下单递减/改价即时生效/批量操作事件/库存更新缓存)
