## 1. 反向链路的设计要点

### 问题
之前 6 条跨模块 E2E 链全部以 admin-web 为起点，缺少「非管理端 → 后端」的反向数据流验证。

### 解决方案
链07 Miniapp→SDK→API→Domain 实现了首条反向链路，验证了小程序端发起请求经过各层的数据完整性。

### 经验
- 反向链路适合验证「消费端 → 服务端」的认证/权限/数据隔离
- 典型场景: 小程序/App/第三方集成 → SDK → API → Domain
- 反向链路能发现正向链路不易发现的安全问题（如从 admin-web 视角看没问题，但从 miniapp 视角看缺少认证保护）

## 2. 多状态机验证的方法

### 问题
链05 已验证了 campaign 的 4 状态状态机，但验证模式是固定的状态转换表。

### 解决方案
链08 扩展为 8 状态的订单状态机，升级验证模式:
- 合法路径验证: 正向链路 + 异常分支（取消/退款）
- 非法跳转验证: Domain 层拒绝跳过中间状态
- 全路径覆盖: 8 种状态的进度展示/操作按钮/履约状态一致性

### 经验
- 状态机测试应覆盖: 所有合法路径 + 至少一条非法路径 + 回退路径
- 跨模块状态下，每个端（Admin/Mobile/Storefront）对同状态的展示必须一致

## 3. RBAC 权限矩阵的测试分层

### 问题
链06 只覆盖了 4 种角色(consumer/merchant/admin/finance)的认证 token 校验。

### 解决方案
链09 升级为 6 角色 × 9 模块的完整矩阵验证:
- Admin 层: 权限配置校验（空角色拒绝、重复用户拒绝）
- API 层: 租户隔离存储、角色分离
- Domain 层: 模块级角色校验
- Tob-Web 层: 根据角色渲染模块可见性

### 经验
- RBAC 测试应分层设计: 权限分配层 → 存储层 → 校验层 → 展示层
- 多角色合并权限需要验证「并集」而非「交集」
- 跨租户隔离在每一层都要验证

## 4. 命名规范统一

### 问题
Pulse-Nightly-05 编写过程中发现 admin-web 测试用混乱的命名格式:
- `describe('<feature>: <scenario>')`
- `describe('[L3-E2E] 中文描述')`
- `describe('链0X: 中文描述')`

### 标准（后续统一）
统一使用 `describe('[L3-E2E] 模块: 场景描述')` 格式:
- 前缀: `[L3-E2E]` 标识跨模块
- 模块: 链编号 + 核心路径
- 描述: 中文场景描述
