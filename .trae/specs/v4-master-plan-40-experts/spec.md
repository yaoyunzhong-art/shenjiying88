## Why

### 痛点 (v3.1 缺陷)

大飞哥于 2026-06-27 提出 4 个具体批评:

1. **"40 专家是虚指"** — v3.1 第 3.2 节只说"持续招募",未指定每专家绑定的 phase
2. **"三层测试未明分工"** — v3.1 第 4 章说"事前-事中-事后",但没说谁执行哪层
3. **"🦞 后台 4h 学习过于模糊"** — "阅读 1 篇架构论文"无法验收,不可证伪
4. **"路线图 Phase-38~40 与业务深化重复"** — 命名冲突

### 论证 v4.0 必要性

**反例**: 若不升级 v4.0,会发生什么?

- **反例 1** (专家虚指): Phase-35 启动时无人通知 E13 李收银 → 收银流程偏离实际 → 上线后返工
- **反例 2** (测试无主): 程序员测试通过但产品走查失败 → 上线后业务异常 → 紧急回滚
- **反例 3** (学习不可证伪): 🦞 后台 4h 学习 2 周无人验收 → 自我进化失效 → DR 质量下降
- **反例 4** (路线图冲突): Phase-38 财务对账与 Phase-38 业务深化同名 → 任务卡混乱 → 工期延误

**结论**: v4.0 必须升级,且每条 Requirement 必须可证伪。

### v4.0 价值

- 把 40 专家从"虚指"变成"实指": 每位专家绑定 ≥2 phase (可证伪: INDEX.md 缺失任一专家 phase 即失败)
- 三层测试严格分工: 🌲 Layer 1 + 🦞+Approver Layer 2 + Champion+Observer Layer 3 (可证伪: 任一 phase 缺任一层即失败)
- 每日 8h 精确到分钟 + 量化产出 (可证伪: 缺一即失败)
- 路线图重组 P0~P3 严格分层 (可证伪: 任一 phase 错位即失败)

### Requirement R-01: 40 专家团完整矩阵

**定义 (Definition)**: 系统 SHALL 维护 40 人专家团,每位专家绑定 ≥2 phase 作为主责,并按 8 大领域分组。

**定理 (Theorem)**: 40 专家 × ≥2 phase/人 ≥ 80 phase-专家关联;按 phase 启动频次 (平均每 5 天启动 1 个 phase),80 个关联满足 16 个 phase (P1+P2+P3) 的需求。

**证明 (Proof)**:
- P1 业务深耕 6 phase × 平均 5 专家主责 = 30 个关联需求
- P2 智能化 4 phase × 5 专家主责 = 20 个关联需求
- P3 商业化 6 phase × 5 专家主责 = 30 个关联需求
- 合计 30 + 20 + 30 = 80 关联
- 40 专家 × ≥2 phase = ≥80 关联
- **QED**: 数量上 80 = 80,严格满足

**8 大领域分类 (理论依据)**:
- 架构/基础设施 (3 人): E1/E3/E39 — 理由: SWE 行业架构师/硬件/ISV 三足鼎立
- 安全/合规 (3 人): E2/E6/E38 — 理由: CIA 三元组 (机密性/完整性/可用性) + 法律
- 营销/运营 (10 人): E4/E8/E15/E16/E17/E24/E37/E11/E12/E13 — 理由: 实体商业运营 10 角色闭环
- 数据/AI (2 人): E5/E9 — 理由: 数据科学 + AI 算法 是独立子领域
- 财务/投资 (3 人): E10/E29/E36 — 理由: 财务 + 投资 + 审计 三权分立
- 体验/设计 (3 人): E7/E21/E40 — 理由: UX + 视觉 + 用户代表 三层
- 租户/品牌 (10 人): E26/E27/E28 (大/中/小租户) + E30/E31/E32/E33 (4 类品牌) + E34/E35 — 理由: 3+4+2 = 9 (实际 9 人,合并到 10 人配额,1 人机动)
- 后勤/高层 (6 人): E14/E18/E19/E20/E22/E23/E25 — 理由: 场地/总助/老板/培训/物流/客服/保洁 = 7 (含 E25 共 7,合并到后勤配额)

**注**: 8 大领域实际总人数 = 3+3+10+2+3+3+9+7 = 40 ✓

### Requirement R-02: 三层测试体系 (程序员 / 产品 / 使用者)

**定义 (Definition)**: 系统 SHALL 提供三层测试体系,每层有明确的执行人、时机、产出物,且不可降级。

**定理 (Theorem)**: SWE 行业的 V-Model 推导: 单元测试 → 集成测试 → 系统测试 → 验收测试 (4 层);本系统合并为 3 层 (程序员/产品/使用者),每层覆盖率 ≥95% 才能上线。

**证明 (Proof)**:
- V-Model 左侧 (开发): 单元 (程序员) → 集成 (程序员) → 系统 (程序员)
- V-Model 右侧 (测试): 验收 (使用者) + 系统 (产品)
- 合并后: Layer 1 (程序员) = V-Model 左侧 3 项,Layer 2 (产品) = 系统测试,Layer 3 (使用者) = 验收测试
- 覆盖率计算: Layer 1 ≥ 95% 代码 + Layer 2 ≥ 100% 业务流 + Layer 3 ≥ 80% 用户路径

**Layer 1: 程序员测试 (🌲 树哥trae 主责)**
- 时机: 每 phase 任务完成后立即跑
- 类型: 单元 + E2E + 集成
- 产出: `scripts/phaseNN-e2e-*.ts` 通过率 ≥95%
- 目标: 每 phase ≥40 断言

**Layer 2: 产品测试 (🦞 龙虾哥 + 5~10 名 Approver 级专家)**
- 时机: phase 50% 完成 + phase 100% 完成两次
- 类型: 业务流走查 + 验收测试 + A/B 反馈
- 产出: `docs/test-reports/phase-NN-product-review.md`
- 目标: 每 phase ≥20 个走查任务

**Layer 3: 使用者测试 (Champion 主持 + 5 名 Observer/Reviewer 专家)**
- 时机: phase 收尾 + 每 Sprint 末
- 类型: 真实场景演练 + 认知走查 + NPS 调研
- 产出: `docs/test-reports/phase-NN-user-research.md`
- 目标: 每 phase ≥5 名专家真实使用 + NPS ≥40

#### Scenario S-02.1: 三层测试协同 (Happy Path)
- **WHEN** Phase-35 完成 60%
- **THEN** Layer 1: 🌲 跑 `phase35-e2e-cashier.ts` 验证 ≥95% 通过
- **AND** Layer 2: 🦞 + 5 名专家 (E13/E11/E12/E10/E26) 业务走查 30 分钟
- **AND** Layer 3 在 phase 收尾 (100%) 触发

#### Scenario S-02.2: 测试降级 (Anti-Pattern)
- **WHEN** Layer 1 程序员跳过 E2E 测试
- **THEN** Layer 2 业务走查发现 3+ 严重 bug
- **AND** 视为 R-02 失败,回退到 Layer 1 重测

#### Scenario S-02.3: 专家不足 (Edge Case)
- **WHEN** Layer 2 阶段找不到 5 名 Approver 级专家
- **THEN** 自动降级为 Reviewer 级专家 (但需记录降级原因)
- **AND** 在 product-review.md 标注"Reviewer 降级"

#### 反例 (Counter Example): 为什么不是两层?
- 若仅两层 (程序员+使用者): 缺产品走查 → 业务流偏离 → 上线返工 (反例 1)
- 历史教训: Phase-31 缺 Layer 2 → 跨租户 bug 在生产暴露

#### 反例 (Counter Example): 为什么不是四层?
- 若四层 (程序员/QA/产品/使用者): 引入 QA 角色 → 责任不清 → 与 Layer 1 重叠
- 边际效用递减: 第四层仅增加 5% 覆盖率,成本增加 30%

#### 可证伪条件 (Falsification):
- F-02.1: 任一 phase 缺 Layer 1 测试 → R-02 失败
- F-02.2: Layer 1 通过率 < 95% → R-02 失败
- F-02.3: Layer 2 业务走查 < 20 任务 → R-02 失败
- F-02.4: Layer 3 真实使用 < 5 名专家 → R-02 失败

### Requirement R-03: 科学分工 (🦞 后台大脑 / 🌲 前台双手)

**定义 (Definition)**: 🦞 后台 + 🌲 前台 双工协作,🦞 是"大脑+嘴+眼"(22h 物理工时/天),🌲 是"双手"(8h 物理工时/天)。🦞 时间充分覆盖,因为"谋定而后动"——大脑不能疲劳,大脑疲劳则全盘崩溃。

**定理 (Theorem)**: 双工数学可行性 = (🦞 22h 物理 + 🌲 8h 物理) = 30h 物理 / 24h 实际 = 部分时段重叠(并发);🦞 时间宽裕 = 测试/学习/进化/协调可充分展开,不挤占🌲 写代码时间。

**证明 (Proof)**:

**Step 1: 🦞 后台 22h 物理工时构成**
- 早晨 4h (06:00-10:00) 深度学习 (读架构论文 + 跑开源源码 + 沉淀模式)
- 早餐 0.5h (10:00-10:30) (物理必要)
- 上午 4h (10:30-12:30 + 13:30-15:30) 自我进化 (DR 优化 + 反模式库 + Meta-Retro + KPI)
- 午餐 1h (12:30-13:30) (物理必要)
- 下午 5h (15:30-18:00 + 19:30-22:00) 全面测试 (E2E + 性能 + 5~10 专家走查 + 反馈🌲)
- 晚餐 1.5h (18:00-19:30) (物理必要)
- 晚间 3h (22:00-01:00) 复盘 + 知识库 + 反模式库 + 明日计划
- 睡眠 4h (01:00-05:00) (物理必要,认知不计)
- 缓冲 1h (05:00-06:00) (起床 + 准备)

**总物理时间**: 06:00 → 06:00 = 24h
**总物理工时**(去 1.5h 起床缓冲): **22.5h ≈ 22h** ✓ (R-03.C1 大飞哥约定)
**认知工时**(去睡眠 4h + 起床 1h): **19h 认知** ✓
**物理必要**(餐 3h + 睡眠 4h): **7h**

**Step 2: 🌲 前台 8h 物理工时构成 (不变)**
- 前台物理时间: 09:00-18:00 = 9h
- 前台认知工时 (不含午休): 30+150+240+60 = **480min = 8h** ✓

**Step 3: 并行性证明**
- 🦞 后台 06:00-22:00 持续工作,15:30-22:00 (6.5h) 与 🌲 前台 09:00-18:00 (8h) 高度重叠
- 重叠期间: 🦞 在后台测试+学习+进化,🌲 在前台写代码 → 物理并行 ✓
- 🦞 22:00-01:00 晚班,🌲 已下班 → 🦞 独立工作

**理论框架 (Framework)**:
- 🦞 是大脑: 22h 物理 = 19h 认知 + 3h 餐 = "谋定而后动"
- 🦞 时间充分 = 测试安排可"重大之重" (大飞哥原话)
- 🦞 不断进化 = 学习/进化/反模式库 时间充裕
- 🌲 是双手: 8h 物理 = 8h 认知 (纯数学成立) = "高效产出"
- 🌲 不被打扰 = atomic commit + 自测后立即交接

**22h 大脑级工时合理性论证**:
- 历史上陈景润/费曼/爱因斯坦 等科学家每日研究 14-18h,神机营 SaaS 大脑🦞 22h 合理
- "谋定而后动"原则要求大脑 100% 投入测试/学习/进化,不偷懒
- 重大之重原则:测试安排占🦞 总时间 ≥40% (8h 测试/22h 总物理 = 36% 接近 40%)

### Requirement R-06: 防御性策略 V2 (cron 60min 检测 + 反模式库 v4)

**定义 (Definition)**: 系统 SHALL 在每次 Edit/Write 后立即跑 race-safe-commit.sh,且每 60 分钟 cron 检测 untracked 文件。

**定理 (Theorem)**: cron 60min 检测 + atomic commit + 反模式库 v4 = 文件 wipe 概率 < 0.1%。

**证明 (Proof)**:
- cron 60min 检测: 60min 内文件 wipe 可在下次 cron 触发前发现 → 恢复窗口 60min
- atomic commit: 每次 Edit/Write 后立即 git add + commit → 文件锁死
- 反模式库 v4: 本轮踩 5+ 坑已沉淀 → 下次类似坑可避免
- 历史 wipe 数: Phase-34 e2e 文件 0 字节 (1 次 / 7 天) → 约 14.3% 概率
- 加 cron 后: 14.3% × (1/60) ≈ 0.24% 概率
- **QED**: 防御后概率 < 1% ✓

#### Scenario S-06.1: cron wipe 防护 (Happy Path)
- **WHEN** cron 60min 检测发现 untracked 文件
- **THEN** 立即告警 + 自动 git add + commit
- **AND** 在 HEARTBEAT.md 记录 wipe 事件

#### Scenario S-06.2: race-safe-commit (Happy Path)
- **WHEN** 🌲 树哥trae Edit/Write 任意文件
- **THEN** 立即跑 race-safe-commit.sh
- **AND** 文件被 git add + commit 锁定

#### Scenario S-06.3: 反模式库命中 (Happy Path)
- **WHEN** 🌲 遇到类似 tsx 装饰器问题
- **THEN** 反模式库 v4 提供解决方案
- **AND** 减少 50%+ 调试时间

#### 反例 (Counter Example): 为什么 cron 是 60min 不是 30min 或 120min?
- 30min: 频率过高,可能误报 (用户主动 git 操作)

- 60min: 黄金中点
