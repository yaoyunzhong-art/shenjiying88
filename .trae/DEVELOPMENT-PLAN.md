## 1. 🎯 项目愿景 & 战略目标

### 1.1 愿景
打造**神机营 SaaS** —— 一个面向游戏厅 / 娱乐场所的智能化运营平台,基于 **AI Agent** 提供 7×24 智能客服、订单查询、库存管理、会员服务、数据洞察等全场景能力。

### 1.2 12 个月战略目标
| 阶段 | 时间窗 | 核心目标 |
|------|--------|----------|
| **Q1 (Month 1-3)** | 2026-06 ~ 2026-08 | 基础平台 (Phase-25~44) - 多租户 + 流式 + 重连 + 持久化 |
| **Q2 (Month 4-6)** | 2026-09 ~ 2026-11 | 业务模块 - 收银 / 会员 / 库存 / 报表 |
| **Q3 (Month 7-9)** | 2026-12 ~ 2027-02 | 智能化升级 - 主动推荐 / 异常检测 / 自愈 |
| **Q4 (Month 10-12)** | 2027-03 ~ 2027-05 | 商业化 - 多区域 / 增值服务 / 开放平台 |

### 1.3 核心价值
- **AI Agent First**: 所有业务能力由 Agent 提供,UI 是 Agent 的视图
- **多租户 SaaS**: 一次部署,服务 N 个租户,数据 100% 隔离
- **实时流式**: 毫秒级响应,SSE 推送 + 重连 + 持久化
- **可观测**: 全链路事件流,可回放、可审计、可调试

### 3.2 40 名系统使用者专家团队
**专家构成** (来自神机营真实运营场景):
- **角色**: 收银员 / 店长 / 会员经理 / 库管 / 财务 / 客服 / 巡店督导 / 系统管理员 等
- **每个角色 5 名代表**: 不同年限、不同门店规模、不同区域

**专家参与开发的三阶段 (事前-事中-事后)**:

#### 事前 (Before)
- **DR 评审**: 任何 DR (架构决策) 发布前,必须由对应角色的 3-5 名专家评审
- **Spec 评审**: PRD / 接口规范 / UI 草图,接受专家反馈
- **用例补充**: 专家提供真实业务场景的边缘 case (如: 网络断线 / 打印机缺纸 / 顾客投诉)

#### 事中 (During)
- **可用性测试**: 每周一个 phase 完成后,组织 5-10 名专家做 30 分钟任务走查
- **认知走查**: 专家用 thinking-aloud 方式走关键路径 (收银流程 / 会员注册 / 退款)
- **A/B 反馈**: 两个 UI 方案让专家投票

#### 事后 (After)
- **回归验收**: Phase 完成后,专家做回归测试 (新功能不影响老流程)
- **满意度调研**: NPS + 文字反馈
- **事故复盘**: 线上问题专家参与根因分析 (5 Whys)

## 4. 🔄 完整工作流 (事前-事中-事后)

### 4.1 Phase 级工作流
```
[事前]
  1. 🦞 出 Spec (业务背景 + AC + 接口)
  2. 🦞 出 DR (架构决策 + 备选 + 后果)
  3. 🦞 调度 5 名相关专家评审 (书面 + 1h 视频)
  4. 🦞 出 Tasks 卡片 (T-NNN, 估时, 依赖)
  5. 🦞 与 🌲 对齐实施计划

[事中]
  6. 🌲 按 Tasks 实施 (每完成 1 个立即 atomic commit)
  7. 🦞 实时提供架构答疑
  8. 🌲 自测 (单元 + E2E)
  9. 🦞 组织 5 名专家走查 (认知走查 + 任务走查)
  10. 🦞 出 retro 文档 (做得好的 / 问题 / 改进)

[事后]
  11. 🦞 组织 10 名专家回归测试
  12. 🦞 出满意度调研 (NPS)
  13. 🦞 更新 DR / 反模式库 / 知识库
  14. 🦞 总结学到的模式,纳入 v(n+1) 模板
```

## 7. 🛡️ 防御性策略

### 7.1 cron auto-stash 防护 (最高优先级)
- **race-safe-commit.sh**: 60 行 bash, 自动 add + commit + 过滤 `.DS_Store/dist/node_modules`
- **使用规则**: 每次 Edit / Write 后立即跑, 不依赖 IDE git
- **绝对禁止**: `git reset --hard` / `git commit --amend` (已发生 1 次 catastrophic wipe)
- **恢复方案**: 用 `git reflog` + `git reset --hard <HEAD@{N}>` 找回

### 7.2 质量防护
- **E2E 优先**: 每个 Phase 至少 40 断言
- **DR 强制**: 架构变更必须有 DR 编号
- **类型严格**: TS strict + discriminated union
- **静态扫描**: grep 验证关键 token (防伪字符串匹配)

### 7.3 沟通防护
- **任务卡清晰**: T-NNN + 估时 + 依赖
- **专家反馈必收**: 任何反馈必须归档到 `.trae/specs/`
- **双向确认**: 🌲 完成 → 🦞 验收 (有 checklist)
