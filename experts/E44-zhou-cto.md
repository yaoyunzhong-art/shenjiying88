## 2. 💡 核心洞察

### 洞察 1: 多租户 = 纵深防御三层

**观点**: 多租户隔离不是「一个tenant_id字段」，而是纵深防御三层。

**三层纵深防御**:

| 层 | 实现 | 防御对象 |
|----|------|----------|
| L3 业务层 | 所有Service入口强制tenant_id校验 | 业务逻辑越权 |
| L2 数据层 | RLS（行级安全）+ 表级tenant_id | SQL注入/直连越权 |
| L1 网络层 | API Gateway + OAuth 2.0 scopes | 网络层越权 |

**TenantGuard设计原则**:
1. **默认拒绝**: 任何未携带有效tenant_id的请求，直接403
2. **双重检验**: Gateway层校验 + Service层二次校验
3. **最小可见**: 一个租户绝对看不到另一个租户的数据
4. **不可绕过**: 直接连数据库也看不到其他租户（RLS）

**实践案例-Phase-31多租户**:
- 数据库: 单实例+RLS（非独立库，降低运维成本）
- Redis: 自动在key前缀注入tenant_id（如 `t:123:coupon:list`）
- 消息队列: 每个租户独立队列（防止一个租户消息影响其他）
- 定时任务: 按租户范围执行（非全量）

---

### 洞察 2: 开放API = 「安全第一，性能第二」

**观点**: 开放给第三方的API，安全是底线，第一版宁可慢也要安全。

**开放API安全策略**:

| 领域 | 策略 | 实施 |
|------|------|------|
| 认证 | OAuth 2.0 + JWT | access_token 2h，refresh_token 30天 |
| 授权 | Scope细粒度 | read:order / write:order / admin |
| 限流 | 租户级+应用级 | 默认100/分钟，超限429 |
| 审计 | 100%请求日志 | 来源IP+应用ID+操作+时间 |
| 数据 | 脱敏+过滤 | 响应自动过滤字段（按scope） |
| 防篡改 | 请求签名 | HMAC-SHA256签名，timestamp防重放 |

**API版本管理**:
```
/v1 → 稳定版（2年支持期）
/v2 → 当前最新
/v3 → Preview版（30天试用，不保证向后兼容）
```

- 废弃API在废弃前180天通知开发者
- 版本迁移指南文档化
- 向后兼容承诺（必须，不破坏现有集成）

### 洞察 3: 架构决策 = 「不急着拆」

**观点**: 90%的「微服务冲动」其实是单体就能解决的问题。

**架构拆分的冷静决策树**:
```
团队规模 > 50人?   → No → 坚持模块化单体
  │
  Yes → 代码冲突 > 30%?  → No → 模块化单体+接口化
  │                             (不需要微服务)
  Yes → 部署频率 < 1次/天? → No → 提升CI/CD+自动化测试
  │
  Yes → 应用启动时间 > 5min?  → No → 优化构建+增量编译
  │
  Yes → 考虑微服务拆分
```

**神机营当前决策**:
- 团队: < 20人 → 模块化单体（apps/api 内部模块化）
- CI/CD: 已自动化 → 部署频率满足
- 构建时间: 2min → 不需要拆
- ✅ 结论: Phase 25-50 保持模块化单体

### 洞察 4: 技术团队管理的「三块基石」

**观点**: 技术管理不是「管代码」，是「管人+管事+管增长」。

**三块基石**:

**A. 招聘: 找「T型」人才**
- 一专: 某个领域深入（如数据库/前端/安全）
- 多能: 能看上下游代码（后端能写简单前端）
- 文化: 重视代码质量+测试+文档

**B. 培养: 双线晋升**
```
技术线: 初级 → 高级 → 架构师 → 首席
           │
管理线: 技术Lead → 技术经理 → 技术总监
```
- 每周技术分享: 一人讲，全员听
- 双月Code Review: 随机互审+点评
- 参与spec评审: 提升系统思维

**C. 考核: 量化+定性**
- 量化（60%）：代码质量（TSC 0/测试覆盖/缺陷率）
- 定性（40%）：团队贡献、知识分享、文档质量
- 负向指标: P0事故追责（系统性地，非惩罚个人）

---

### 洞察 5: 技术债务治理 = 「定期还贷」

**观点**: 技术债务不可怕，可怕的是「只借不还」。

**技术债务四象限**:

|  | 影响大 | 影响小 |
|--|--------|--------|
| **修复成本低** | 🔴 修复大+成本低 → 立即处理 | 🟡 影响小+成本低 → 顺手修 |
| **修复成本高** | 🟠 影响大+成本高 → 规划处理 | 🟢 影响小+成本高 → 技术评审后决定 |

**神机营当前技术债务清单(示例)**:
- 🔴 `@m5/api hang 6+ pulses` — 影响大+修复成本低 → 已修复
- 🟠 跨phase的TypeScript类型不一致 — 影响大+修复成本高 → 规划处理
- 🟡 某模块缺少单元测试 — 影响小+成本低 → 随手补充
- 🟢 老旧依赖升级 — 影响小+成本高 → 技术评审

**治理节奏**:
- 每个Sprint分配20%时间处理技术债务
- 新Phase提交前必须检查反模式规则（R-07）
- 月度技术债务会议（盘点+优先级+责任人）
