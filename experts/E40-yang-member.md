## 2. 💡 核心洞察

### 洞察 1: 会员生命周期 = 六个「关键触点」

**观点**: 会员不是一条横线，而是六个「触点」——每个触点决定存留。

**六个关键触点**:

```
注册 → 首单 → 复购 → 休眠 → 挽回 → VIP
  │      │      │      │      │      │
  │      │      │      │      │      └─ 升级SVIP（精细化服务）
  │      │      │      │      └──────── 沉睡提醒（优惠券+电话）
  │      │      │      └─────────────── 7天未到店（消息推送）
  │      │      └────────────────────── 次日优惠券（提升留存）
  │      └───────────────────────────── 新客礼包（转化首单）
  └──────────────────────────────────── 5秒注册（手机号+验证码）
```

**每个触点的问题与优化**:

| 触点 | 当前问题 | 优化方案 |
|------|----------|----------|
| 注册 | 填写信息太多（注册率低） | 手机号+验证码=5秒注册 |
| 首单 | 新客不知道玩什么 | 新客任务引导+首单折扣 |
| 复购 | 缺少二次触达 | 离店推送次日券 |
| 休眠 | D7~D30无操作 | 消费预测模型提前干预 |
| 挽回 | 30天未到店 | 智能优惠券+主动电话 |
| VIP | SVIP与普通会员区别不大 | SVIP专属权益+黑卡体验 |

### 洞察 2: 推荐系统的「三步走」原则

**观点**: 推荐系统不要一上来就做深度学习，三步走渐进式更务实。

**Step 1: 规则推荐（Phase-36）**
```
基础规则引擎:
  热销榜: 全站/本店/本品类 × 7天/30天
  凑单推荐: 游戏币+零食（购买组合热频）
  老带新: 推荐有礼（双方得积分）
```

**Step 2: 协同过滤（Phase-40）**

**Item-Based协同过滤**:
- 核心公式: 相似度(游戏A, 游戏B) = 同时玩过A和B的用户数 / 玩过A的用户数
- 推荐: 「玩过XX的人还玩了XX」
- 实现: 离线计算（每日凌晨更新）

**User-Based协同过滤**:
- 核心公式: 相似度(用户A, 用户B) = 共同游戏数 / 总分叉
- 推荐: 「和你相似的人都在玩XX」
- 实现: 近邻缓存（Redis, TTL=1h）

**Step 3: 混合推荐（Phase-41+）**
```
热销(50%) + 协同过滤(30%) + 新鲜度(20%) = 最终推荐列表
```

### 洞察 3: SVIP全周期 = 「黑卡体验」不可复制

**观点**: SVIP不是因为折扣多而付费，而是因为「黑卡体验」值。

**SVIP差异化体验**:

| 维度 | 普通会员 | SVIP |
|------|----------|------|
| 折扣 | 95折 | 85折+生日双倍积分 |
| 排队 | 正常排队 | 优先入场（专属通道） |
| 客服 | 机器人+工单 | 专属客服经理（1h响应） |
| 活动 | 普通活动通知 | 新品试玩+闭店活动邀请 |
| 积分 | 1元=1分 | 1元=2分+积分提现 |
| 存储 | 个人记录30天 | 永久存储+数据导出 |

**SVIP流失预警**:
- 消费下降30%持续30天 → SVIP专属优惠
- 7天未到店 → 专属客服电话
- 14天未到店 → 邀请闭店试玩
- 30天未到店 → 最高级别挽回（总经理级邀约）

---

### 洞察 4: NPS ≥ 50 的「三道防线」

**观点**: NPS 50不是天花板，是底线。

**三道防线**:
1. **主动防护**: 消费异常检测（如券核销失败自动补偿）
2. **快速响应**: 投诉工单<15min首回，<2h解决
3. **深度修复**: 投诉解决后72h回访 + 补偿积分

**网络研究主题**:
- 🔍 付费会员SVIP设计: SVIP的核心不是折扣深度，而是"差异化体验"。好市多/山姆会员制成功的关键是：会员觉得"值回票价"的增值服务。室内游乐场SVIP的差异化在于——优先入场（减少排队痛感）、专属活动（闭店试玩/新品首测）、专属客服经理。SVIP推送频率应降低（每周≤2条），但推送的内容质量必须高（专属活动/新品优先购）。
- 🔍 会员全生命周期管理: 会员从注册到流失的每个触点需精细化管理。推送在各触点扮演不同角色——注册时P2权益引导、首单后P3复购唤醒、休眠期P1权益提醒、流失前P1挽回优惠。全生命周期管理要求推送频率随会员阶段变化，而不是一刀切。

### 讨论: 导玩+会员排队系统的用户痛点

**讨论成果**:（与E12孙导玩联合分析）

1. **SVIP与普通会员排队体验落差**: 当前排队系统未区分会员等级，SVIP和普通会员共用排队队列。SVIP付费的核心权益之一是"优先入场"，但当前未实现。建议：
   - 排队系统增加SVIP优先通道（cx-constraints明确SVIP有优先入场权益）
   - SVIP排队时推送"您享有优先入场特权，可查看VIP通道位置"

2. **推送频率与会员价值匹配**: 普通会员（青铜-白银）每日≤1条P2/P3推荐，SVIP每周≤2条。但当前push模块完全无视会员等级——所有人都收到相同频率的推送。这是一个"逆向公平"问题：高价值会员反而被过度打扰。

3. **会员差异化推送时机**: 不同等级会员的推送时机也应不同。SVIP适合在午休时段（11:00-13:00）推送，普通会员适合在晚间（19:00-21:00）推送。AI最佳时机模型需按等级分组训练。

4. **离线排队与会员兜底**: 网络中断时（边缘服务器离线模式），排队数据应保留会员等级标记，确保离线场景下SVIP优先级仍然生效。

### 事中: push模块会员差异化推送缺陷

**检查结果**:
- ❌ push.service.ts 无member_id维度的频率计数——所有推送每客户每日≤1条这个约束无法实现

- ❌ push.controller.ts 无会员等级注入——API端点没有从请求上下文获取会员等级
- ❌ 会员差异化推送频率表（青铜-银/黄金-铂金/钻石-王者/SVIP的不同频率）未实现
- ❌ 疲劳度保护（member_fatigue_score表）未实现——无法做分层干预
- ⚠️ NotificationSettingsScreen有会员升级推送（member_upgrade），但无SVIP专属权益推送
